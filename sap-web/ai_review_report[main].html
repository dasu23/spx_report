    packages/chatbot/src/hooks/use-chat-session.ts3 Major3 个问题
    MAJORL21安全风险
    `useChatSession` 对整个 store 启用了 `persist`，而 store 中包含 `chatHistories`。这会把用户提问和机器人回复长期写入 `localStorage`，聊天内容中的手机号、工单信息等敏感数据会在浏览器本地明文留存，且可被同机其他使用者或后续 XSS 直接读取。
    修复建议与测试场景
    修复建议

    不要持久化完整聊天内容；使用 `partialize` 仅保留最小必要的非敏感状态，或改为仅保存在内存/`sessionStorage` 并配合到期清理。若必须落盘，应增加明确过期和清除策略。

    测试场景

    发送一条包含敏感信息的消息后刷新页面，检查 `localStorage['chat-session']` 是否仍能直接读到完整会话内容；修复后不应保留消息正文。

    MAJORL74-85状态管理
    新会话启动时先调用 `refreshSessionExpireTimer()`，再写入新的 `sessionExpireTime`。首次启动或已过期后重开时，定时器会基于旧值/0 计算，导致新会话不会注册新的过期定时器，30 分钟自动过期逻辑失效。
    修复建议与测试场景
    修复建议

    先计算并写入新的 `expireAt`，再调用 `refreshSessionExpireTimer()`；或在 `set({ sessionExpireTime: expireAt })` 之后补一次 `refreshSessionExpireTimer()`。例如：`const expireAt = Date.now() + SESSION_EXPIRE_TIME; set({ sessionExpireTime: expireAt }); refreshSessionExpireTimer();`。

    测试场景

    清空本地存储后打开聊天窗口并启动会话，等待或模拟推进到 30 分钟后，确认会话会自动关闭并写入过期记录。

    MAJORL98-106业务逻辑
    `closeSession()` 在调用 `pushSessionExpireHistory()` 之前先把 `sessionExpireTime` 重置为 `0`，导致新增的过期记录携带的 `expireTimestamp` 始终为 0，界面展示的结束时间会错误。
    修复建议与测试场景
    修复建议

    在重置状态前先保存当前过期时间，并将该值传给记录函数。比如先读取 `const { sessionExpireTime } = get();`，再 `set(...)`，最后用保存的时间写入 expire history。

    测试场景

    启动会话后手动关闭或触发超时关闭，检查聊天记录中的 `Session ended` 时间是否与实际结束时间一致，而不是 Unix epoch 对应时间。
