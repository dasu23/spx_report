    src/access.ts1 Major1 个问题
    MAJORL22-25权限与访问控制
    **原因：** 新增菜单权限代理对 `menu:` 前缀采用默认放行策略：当 `menuVisibility` 未加载或接口未返回对应菜单项时，直接 `return true`。这意味着菜单可见性从“显式授权”退化为“缺省可见”，任何依赖该前端权限位隐藏的管理入口都会在状态缺失、接口异常或字段遗漏时暴露给当前登录用户。 ```ts if (typeof p === 'string' && p.startsWith(MENU_PREFIX)) { // 接口未下发 menus 或未列出该菜单 → 默认放行 if (!menuVisibility) return true; const name = p.slice(MENU_PREFIX.length); return menuVisibility[name] ?? true; } ``` **影响：** 若路由、菜单或页面按钮仅依赖 `access['menu:xxx']` 控制展示，低权限用户可能看到本不应暴露的后台入口并尝试访问；一旦后端接口鉴权不严或存在遗漏，将进一步演变为越权访问。即使后端最终拦截，也会造成管理功能枚举与攻击面扩大。
    修复建议与测试场景
    修复建议

    将菜单可见性改为默认拒绝：未加载完成前返回 `false` 或结合显式 `shipperMenuLoaded` 状态阻止渲染；仅当后端明确下发 `true` 时才展示菜单。示例： ```ts if (typeof p === 'string' && p.startsWith(MENU_PREFIX)) { if (!initialState?.shipperMenuLoaded) return false; const name = p.slice(MENU_PREFIX.length); return menuVisibility?.[name] === true; } ```

    测试场景

    补充用例：1）`menuVisibility` 为 `undefined` 时管理菜单不显示；2）返回对象缺少某菜单键时该菜单仍不显示；3）仅显式为 `true` 的菜单可见。

    src/hooks/useDownloadFile.ts1 Major1 个问题

    这里把 `content-disposition` 从仅支持 `attachment` 扩展为同时匹配 `inline`，但文件名提取仍依赖 `filename=` 的简单替换。很多 `inline` 响应只返回 `inline` 或使用 `filename*=`（RFC 5987），此时 `replace` 会保留原始头值，最终生成错误文件名，导致下载文件名异常。

    测试场景

    分别覆盖 `attachment; filename="a.pdf"`、`inline; filename="a.pdf"`、`inline`、`inline; filename*=UTF-8''a%20b.pdf` 四种响应头，验证最终下载文件名正确。
