    mock/deliveryDashboard.ts1 Major1 个问题
    MAJORL9-109API契约
    新 Dashboard mock 的返回体统一使用了 `{ code, success, message, payload }`，与仓库现有 mock / service 约定的 `{ code, msg, data }` 不一致。该页面又明确是先用 mock 数据开发，这种 envelope 差异很容易让复用现有 `request(...).then(res => res.data)` 模式的页面在 mock 环境下拿到 `undefined`，导致整页无数据。
    修复建议与测试场景
    修复建议

    把 mock 返回结构统一改成项目已有约定，例如 `res.json({ code: 200, msg: 'Success', data: ... })`；如果业务侧必须保留 `payload`，那就需要在对应 service 层显式做一次字段映射，并补测试锁定这个契约。

    测试场景

    分别请求 `origin-hubs`、`overview/summary`、`overview/cycle-time`、`overview/state-distribution`、`overview/zones`，验证 service 层最终都能拿到非 `undefined` 的业务数据，并能直接驱动页面渲染。

    src/pages/shippers/delivery-performance/index.tsx2 Major 1 Minor3 个问题
    MAJORL65-99API契约

    修复建议

    这里应直接返回数组结果：`useRequest(() => fetchOriginHubs())` 和 `useRequest(() => fetchPickupAccounts())`。如果必须保留包装结构，则需要在传参前解包为 `origins.data` / `pickupAccounts.data`。

    主数据加载使用 `Promise.all` 并发请求，但没有任何错误兜底；一旦任一接口失败，`useRequest` 会结束 loading 且 `data` 为空，随后组件走到 `if (!data || !kpiSummary || !geoStats) return null;`，用户只会看到空白页，没有错误提示也无法判断是否可重试。
    修复建议与测试场景
    修复建议

    为 `useRequest` 增加 `onError`/`error` 渲染分支，至少展示可感知的失败态和重试入口；如果允许部分降级，可改为 `Promise.allSettled` 并对失败模块做局部占位。

    测试场景

    构造 `topState='CA'` 且 `stateVolumes[0]` 不是 `CA` 的返回数据，验证页面展示的 top state 名称、数量和占比保持一致。
