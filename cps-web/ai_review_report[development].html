    订单详情 mock 路由声明的参数名是 `:id11`，但处理函数读取的是 `req.params.id`。这里会得到 `undefined`，`Number(undefined)` 变成 `NaN`，导致 `find` 永远匹配不到订单，详情接口始终返回 `order not found`。
    修复建议与测试场景
    修复建议

    统一路由参数名与读取逻辑，例如保留当前路由时改为 `const id = Number(req.params.id11);`；或将路由改成 `GET /cps-api/orders11/:id` 并继续读取 `req.params.id`。示例：`'GET /cps-api/orders11/:id': async (...) => { const id = Number(req.params.id); ... }`。

    测试场景

    调用 `GET /cps-api/orders11/10086`，应返回 `code: 200` 且 `data.id === 10086`；再调用不存在的 ID，才应返回 `order not found`。

    `fetchData` 只在 `finally` 中关闭 loading，没有 `catch` 或错误提示。`listOrders` 请求失败时页面会静默保留旧列表数据，用户无法感知查询失败，搜索/翻页场景下会表现为结果不刷新。
    修复建议与测试场景
    修复建议

    为 `fetchData` 增加 `catch` 分支并提示失败，例如：`try { ... } catch (e) { message.error('Failed to load shipments'); setRowData([]); setTotal(0); } finally { setLoading(false); }`。至少应给出错误反馈，避免静默失败。

    测试场景

    模拟 `listOrders` 返回 rejected promise，验证列表页会提示错误，loading 结束，且不会继续展示误导性的旧查询结果。

    MINORL101-105性能隐患
    `handleSearch` 在先调用 `setPagination(...page: 1)` 后又立即手动调用 `fetchData`。当当前页不是 1 时，`useEffect` 会因分页状态变化再次触发 `fetchData`，导致一次搜索发起两次相同请求。`handleReset` 中的同类写法也有同样问题。
    修复建议与测试场景
    修复建议

    将搜索/重置逻辑统一为仅更新分页和筛选状态，由 `useEffect` 负责触发请求；或在 `setPagination` 后判断当前页是否已为 1，避免重复调用。例如：如果当前页不是 1，仅 `setPagination({ ...prev, page: 1 })`；如果已经是 1，再手动 `fetchData(...)`。

    测试场景

    将分页切到第 2 页后点击 Search/Reset，验证网络层只发送 1 次 `listOrders` 请求。

    src/pages/shippers/delivery-performance/index.tsx2 Major2 个问题
    MAJORL68-116异常处理
    这里把 4 个接口放进 `Promise.all`，任一请求失败都会让整组请求 reject；而组件下方在 `!data || !kpiSummary || !geoStats` 时直接 `return null`。这会让页面在单个接口异常时整体白屏，用户看不到任何错误提示或降级内容。
    修复建议与测试场景
    修复建议

    为 `useRequest` 增加 `onError`/`error` 分支并渲染错误态，或改成分别请求后按模块降级展示。例如：`const { data, loading, error } = useRequest(fetchAll, { refreshDeps:[params] }); if (error) return <Result status="error" title="Failed to load delivery performance" />;`。如果希望部分成功，避免用单个 `Promise.all` 绑死整页。

    测试场景

    模拟 `fetchZones` 或任一接口返回 rejected Promise，验证页面不会返回空白，而是展示错误提示或保留其他已成功模块。

    MAJORL94-103业务逻辑
    `topState` 使用了后端返回的 `sd.topState`，但 `topStateCount` 和 `topStatePercentage` 却直接取 `sd.stateVolumes[0]`。只要 `stateVolumes` 没有按数量倒序返回，页面就会出现“州名”和“数量/占比”不匹配的错误统计。
    修复建议与测试场景
    修复建议

    不要假设 `stateVolumes[0]` 就是 top state，应该显式按 `sd.topState` 查找，或先计算最大值项。例如：`const topItem = sd.stateVolumes.find((item) => item.state === sd.topState) ?? sd.stateVolumes.reduce((max, item) => item.count > max.count ? item : max, { state: '', count: 0 });`。

    测试场景

    构造 `topState = 'CA'`，但 `stateVolumes = [{ state:'TX', count:10 }, { state:'CA', count:20 }]` 的数据，验证页面展示的 top state 名称、数量、占比保持一致。
