    .claude/settings.json1 Critical 1 Minor 1 Info3 个问题
    CRITICALL5-9安全风险
    允许直接执行 `npx @ai-infra/skills@latest setup` 和 `npx @ai-infra/rules@latest setup`。`@latest` 会在每次执行时拉取最新远程包并立即运行安装脚本，属于未锁版本的远程代码执行入口；一旦上游包被劫持或发布恶意版本，会直接在本地开发环境中执行恶意代码并读取凭据/源码。
    修复建议与测试场景
    修复建议

    将 `@latest` 改为固定版本号并配合锁文件/校验机制，仅允许执行经过审核的精确包版本；更稳妥的做法是将所需脚本落库到仓库内，通过受控本地脚本替代远程 `npx` 安装执行。

    测试场景

    验证权限配置中不再出现 `@latest`；模拟包版本漂移时，确认执行的仍是固定版本，且未发生额外网络拉取或脚本变更。

    MINORL4-16配置变更
    权限配置硬编码为 Windows 路径 `//c/Users/Lenovo/**` 和 `C:\Users\Lenovo`。这会让该配置在 Linux/macOS 或其他用户名环境下失效，直接影响命令读取附加目录与本地文件的能力，属于明显的环境耦合。
    修复建议与测试场景
    修复建议

    改为与宿主无关的可配置路径，例如使用用户主目录变量或相对工作区路径；至少应把用户名和平台路径分离到可覆盖配置中。示例：使用 `${HOME}`/`~` 风格路径，或在文档中要求用户首次安装时注入本机目录。

    测试场景

    在 Linux CI、非 `Lenovo` 用户的 Windows 机器、以及默认开发机上分别加载该配置，验证读取允许目录的规则都能生效。

    INFOL6-8向后兼容

    修复建议与测试场景

    MAJORL66-80异常处理
    提交查询时只用了 `try/finally`，`batchQueryByFile(uploadedFileUrl)` 或 `triggerFileDownload(downloadUrl)` 失败会直接抛出，页面不会给用户任何失败提示，按钮 loading 虽会结束，但交互上表现为“点了没反应”。这属于新增异步流程缺少错误处理。
    修复建议与测试场景
    修复建议

    在 `finally` 前增加 `catch`，对失败场景给出明确提示，并避免未处理的 Promise 异常。例如：`try { ... } catch (err) { message.error(intl.formatMessage({ id: 'trackTrace.download.failed' })); } finally { setSubmitting(false); }`。

    测试场景

    模拟 `batchQueryByFile` reject，断言 loading 结束且展示错误消息；模拟 `triggerFileDownload` 抛错，断言同样有失败提示。

    src/utils/shared.tsx1 Critical1 个问题
    CRITICALL39前端安全
    `triggerFileDownload` 直接将外部返回的 `url` 赋给 `a.href` 并触发点击，未校验协议或来源。如果接口返回值被污染为 `javascript:`、`data:` 或恶意站点链接，用户点击查询后会直接执行脚本或跳转到恶意页面，形成前端 XSS/恶意跳转入口。
    修复建议与测试场景
    修复建议

    在赋值前显式校验下载链接，只允许 `https:`（如确有需要再白名单 `http:`）且限定可信域名；对非白名单协议/域名直接拒绝并提示错误。更稳妥的方式是后端返回受控文件流或文件 ID，由前端拼接固定下载域名。

    测试场景

    补充测试：传入 `javascript:alert(1)`、`data:text/html,<script>alert(1)</script>`、`https://evil.example/x` 时应阻止下载；传入白名单内的合法 HTTPS 下载地址时应正常触发。
