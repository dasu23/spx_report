    CRITICALL27安全风险
    新增代码将 `stopMimic()` 返回的真实登录凭证直接写入 `localStorage`：`localStorage.setItem(LS_AUTH_TOKEN, realToken);`。这是敏感 Token 持久化到浏览器可被任意同源脚本读取的位置，一旦页面存在 XSS、第三方脚本污染或浏览器扩展窃取，即可直接接管管理员真实会话。该风险属于明确的敏感数据暴露。
    修复建议与测试场景
    修复建议

    不要将认证 Token 持久化到 `localStorage`。优先改为服务端下发 `HttpOnly` + `Secure` + `SameSite` Cookie 维护会话；若现阶段必须前端持有，也应至少改为仅内存态保存并在刷新后重新协商会话，避免长期落盘。示例：移除 `localStorage.setItem(...)`，改为由 `stopMimic` 接口通过 Set-Cookie 恢复真实会话。

    测试场景

    构造一个同源脚本执行 `localStorage.getItem(LS_AUTH_TOKEN)`，验证退出 mimic 后无法再从 Web Storage 读取到真实管理员 Token。随后验证页面刷新后仍能通过安全会话机制保持或恢复登录。[]

    修复建议

    禁止将 mimic/impersonation Token 写入 `localStorage`。应改为由后端通过 `HttpOnly` Cookie 切换会话，或仅在内存中短暂保存并配合后端一次性会话标识。至少在前端侧避免 `token || ''` 这种落盘写入。

    测试场景

    在开启 mimic 后，通过浏览器控制台或注入脚本尝试读取 `localStorage.getItem(LS_AUTH_TOKEN)`；期望读取不到 impersonation Token。再验证 mimic 会话仍可通过安全 Cookie 正常工作。[]
