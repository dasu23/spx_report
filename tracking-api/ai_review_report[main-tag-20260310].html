major313-315边界条件`form.getEndNumber() == 0` 会触发自动拆箱。当前 DTO 只给 `endNumber` 加了 `@Min/@Max`，没有 `@NotNull`，所以 Excel 或 JSON 里 `End Number` 为空时会直接在这里抛 `NullPointerException`，请求返回 500，而不是可理解的参数错误。在 DTO 上补充 `@NotNull(message = "endNumber is required")`，并在这里改成显式空值判断：`if (form.getEndNumber() == null || form.getEndNumber() <= 0) { throw new RRException("endNumber must be greater than 0"); }`。`startNumber` 也建议同样加 `@NotNull`。上传/提交一条 `End Number` 为空的记录，断言接口返回 4xx/业务异常信息，而不是 500 或 NPE。
major353-358并发安全这里先用 `genSerialNumberValueByRedis(prefix, null, 0)` 读取当前序号，再根据差值回写一次增量，整个“预占号段”过程不是原子的。两个并发的批量打印/文件打印请求命中同一个 `prefix` 时，会读到相同的 `currentSeq`，随后各自生成相同的容器号区间，导致重复 container number。把号段预留改成单次原子 `INCRBY`。示例：先计算 `count = endNumber - startNumber + 1`，再执行一次 `Long endSeq = serialNumberService.genSerialNumberValueByRedis(prefix, null, count); Long beginSeq = endSeq - count + 1;`，后续按 `[beginSeq, endSeq]` 生成容器号，不再先读后写。并发发起两次 `/web/container-label/batch-print` 或 `print-by-file`，使用相同 facility/sortId/date，断言两次返回的 PDF 中 containerId 不重号，且 Redis 最终序号连续递增。
