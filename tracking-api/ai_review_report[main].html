SeverityLineCategoryDescriptionSuggestionTest Scenario
major27-30数据一致性异步监听器在 `sapService.updateFields(...)` 失败时仅记录 warn 日志后直接吞掉异常，既没有落库失败状态，也没有重试路径。对 SAP 这类外部系统调用来说，网络抖动或对端临时故障会把这次同步永久丢失，导致跟踪事件与 SAP 侧数据长期不一致。不要只做日志告警。至少在 catch 中记录一条可重试的失败记录（例如写入 `SAPUpdateHistory` 并标记 `success=false`），再由定时任务/消息队列重试；或者接入现有重试机制后显式抛出可识别异常。示例：`catch (Exception e) { sapUpdateHistoryService.recordFailure(trkEvent.getTrackingNo(), e.getMessage()); throw e; }`。模拟 `sapService.updateFields` 抛出超时异常，验证失败事件会被持久化或进入重试队列，而不是只打印日志后丢失。
major28边界条件这里直接把 `trkEvent.getTrackingNo()` 传给下游，没有校验是否为 `null` 或空串。若系统中存在只创建事件骨架、延迟补全运单号的场景，异步线程会对无效 trackingNo 发起处理，最终在下游报错并被当前 catch 吞掉，形成静默失败。在调用前增加非空和非空白校验，避免无效事件进入 SAP 同步链路。示例：`if (!StringUtils.hasText(trkEvent.getTrackingNo())) { log.warn("skip SAP update for empty trackingNo, eventId={}", trkEvent.getId()); return; }`。发布一个 `TrackingUpdateEvent`，其 `Event.trackingNo` 为 `null` 和空串各一次，验证监听器会直接跳过且不会调用 `sapService.updateFields`。
