    tracking-service/src/main/java/com/miq/trackingservice/service/SAPService.java3 Major3 个问题
    MAJORL301-313数据一致性
    关闭 case 后立即写入幂等表；如果随后 `addNoteToSAP` 返回失败或抛异常，记录会被更新为 `FAILED`，但 `tryCreateCloseEvent` 只在首次插入时允许继续，后续相同 `caseId + eventCode` 的重试会被唯一键直接拦截并返回 `null`。结果是一次临时的 SAP 调用失败会被永久固化，后续重复事件也无法再补发 note，形成关闭状态与 SAP note 状态不一致。
    修复建议与测试场景
    修复建议

    把 `FAILED` 状态纳入可重试流程，而不是仅靠插入去重。可改为先按 `caseId + eventCode` 查询已有记录：不存在时插入 `PENDING`；存在且状态为 `FAILED` 时允许重新调用并更新；仅 `SUCCESS` 时跳过。或者把唯一约束设计为只拦截成功记录。

    测试场景

    补充用例：首次关闭时 `sapC4CApiClient.addNote(...)` 失败，幂等表状态写成 `FAILED`；随后再次收到相同关闭事件时，应允许再次调用 SAP 并在成功后把状态更新为 `SUCCESS`。

    MAJORL343安全风险
    `trackingNo` 被直接替换进 HTML 模板并发送到 SAP：`template().replace("{tracking_no}", ...)`。如果该值来自外部事件且未做 HTML 转义，攻击者可构造包含标签或脚本片段的 tracking number，在下游以富文本方式展示时触发 HTML/脚本注入，属于新增的内容注入风险。
    修复建议与测试场景
    修复建议

    在拼接 HTML 前对 `trackingNo` 做严格的 HTML 转义或白名单校验，例如仅允许预期的字母数字和分隔符；若 SAP 接口支持纯文本备注，优先改为纯文本而不是 HTML 模板。

    测试场景

    增加用例：`trackingNo` 为 `<img src=x onerror=alert(1)>` 或 `</div><script>alert(1)</script>` 时，发送到 SAP 的备注内容应为转义后的文本，不应保留可执行/可解析的 HTML。

    MAJORL74-81业务逻辑
    这里把原先“只要 case 已关闭就重新打开”的分支改成了“只有 trackingNo 变化才更新并 reopen，否则直接 return”。如果同一个 case 在相同 trackingNo 下再次收到需要继续同步/更新的事件，方法会提前返回，既不会 `setClosed(false)`，也不会执行后续 `caseDao.save(dbObj)` 和 Shipment Details 更新，可能导致 case 维持关闭状态、后续事件被静默丢掉。
    修复建议与测试场景
    修复建议

    保留对已关闭 case 的 reopen 逻辑，不要把 reopen 绑定到 trackingNo 变更上。例如先判断 `Boolean.TRUE.equals(dbObj.getClosed())` 时执行 `dbObj.setClosed(false)`，再按需比较 trackingNo 是否需要更新；仅在“既未关闭且 trackingNo 也未变化”时才 `return`。

    测试场景

    补充用例：已存在 `closed=true` 且 `trackingNo` 未变化的 case，再次收到同一 trackingNo 的 webhook/event 时，应重新打开 case 并继续执行保存与后续同步。

    tracking-service/src/main/java/com/miq/trackingservice/service/impl/NewEventServiceImpl.java1 Major1 个问题
    MAJORL109-114业务逻辑
    `UnexpectedRollbackException` 被捕获后仅记录日志并继续执行，`event` 仍为 `null`。这会把“事件创建失败”误当成可继续流程处理，后续 POD 相关逻辑要么基于空事件继续运行触发空指针，要么产生仅处理附件/回执但没有事件主记录的部分成功状态，导致业务数据不一致。
    修复建议与测试场景
    修复建议

    不要吞掉该异常。若需要特殊记录日志，也应在记录后立刻中断流程并重抛，例如：`catch (UnexpectedRollbackException ure) { log.warn("createEvent rollback, trackingNo={}", form.getEventForm().getTrackingNo(), ure); throw ure; }`。如果确实要降级处理，也必须在继续前显式判断 `event != null` 并阻止后续 POD 分支执行。

    测试场景

    模拟 `eventService.createEvent(...)` 抛出 `UnexpectedRollbackException`，验证方法不会继续进入 POD 处理分支，且不会产生空指针或“事件未落库但附件已处理”的部分成功结果。
