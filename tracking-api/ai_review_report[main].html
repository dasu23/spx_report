    MAJORL137-251安全风险
    新增的失败日志将 `resp.getBody()` 原样写入告警日志。该响应体来自外部 SAP 接口，可能包含 case 详情、客户标识或其他敏感业务数据；在异常/非 2xx 场景下直接落盘会扩大敏感数据暴露面。
    修复建议与测试场景
    修复建议

    避免记录完整响应体；仅记录状态码、请求标识和截断后的错误摘要，必要时对返回内容做脱敏后再输出。

    测试场景

    构造 SAP 返回 4xx/5xx 且 body 含客户标识/联系方式等字段，确认日志中不再出现原始响应体。

    MINORL202-204业务逻辑
    `doFindAccountsByNetSuiteCode` 在 SAP 返回空列表时直接返回 `Collections.emptyList()`，而上层使用 `LoadingCache` 且缓存 10 分钟，这会把临时性空响应一并缓存下来。若 SAP 侧短暂抖动或结果延迟可见，后续 10 分钟内都会持续读到空账号列表，导致关联账号信息漏查。
    修复建议与测试场景
    修复建议

    区分“真实无数据”和“请求异常/可疑空响应”。异常场景不要写入缓存，或缩短空结果 TTL。例如在 loader 中抛出运行时异常让 Caffeine 不缓存失败，只有确认业务上确实不存在账号时才返回空列表。

    测试场景

    模拟 SAP 首次返回空 `value`、第二次立即返回正确账号列表；验证第二次调用不会因为首次空结果被缓存而仍返回空列表。

    MINORL250异常处理
    异常日志把 `Throwable` 放进了占位符 `error={}` 中，SLF4J 会将其当作普通参数格式化，通常不会输出完整堆栈。这里一旦 PATCH 失败，日志只保留异常字符串，排障信息明显不足。
    修复建议与测试场景
    修复建议

    去掉 `error={}` 占位符，把异常对象作为最后一个独立参数传入，例如：`log.warn("SAP C4C patch case failed, caseId={}, body={}", caseUuid, patchBody, e);`。

    测试场景

    构造 `requestPatch` 抛异常，验证日志中包含完整 stack trace，而不只是异常类名和 message。

    MAJORL43-52业务逻辑
    新增的负缓存会在 `apiKey` 无效或上游查不到用户时，把一个非空的 `new RpsUserBo()` 写入缓存。若后续 `validateUser(...)` 仅以“是否为 null”作为鉴别条件，这个占位对象会把原本应拒绝的无效 `apiKey` 误判为有效，造成认证失效。

    修复建议

    负缓存不要写入可与真实用户对象同类型、且默认非空的占位值。改为单独的哨兵标记（例如专用 Redis key/value 或显式 `invalid` 标识），并在 `validateUser(...)` 中明确拒绝该标记；同时补充对空对象、空状态值的显式校验。

    测试场景

    构造一个不存在的 `apiKey`，首次调用触发 `cacheNullValue`，随后在缓存 TTL 内再次调用 `validateUser(...)`；断言该 `apiKey` 仍返回未认证/抛出 `UNAUTHORIZED`，而不是被当作合法用户通过。

    修复建议

    至少记录告警日志并带上 `apiKey`/redis key 信息，必要时限制异常类型，避免无声失败。例如：`catch (Exception e) { log.warn("cache null rps user failed, apiKey:{}", apiKey, e); }`。如果该缓存对防穿透很关键，建议让异常继续抛出或接入监控计数。

    测试场景

    模拟 `redisUtils.set(...)` 抛出运行时异常，验证方法会输出告警日志且不会无声吞没；同时确认后续查询路径符合预期。
