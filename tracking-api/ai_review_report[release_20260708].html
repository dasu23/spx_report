    修复建议与测试场景
    修复建议

    恢复对关键目标参数的前置校验，至少校验 `caseUuid`、`properties.getBaseUrl()`、`properties.getCasePath()` 非空且格式合法；对 `caseUuid` 增加白名单约束（如 UUID/业务 ID 格式）。示例：`if (StrUtil.isBlank(caseUuid) || StrUtil.isBlank(properties.getBaseUrl()) || StrUtil.isBlank(properties.getCasePath())) { throw new RRException("invalid SAP patch params"); }`。

    CRITICALL86-93权限与访问控制
    **原因：** 新增逻辑允许调用方直接通过 `pushLabelForm.getOriginTrackingNumber()` 指定退件关联的源运单号，并立即据此执行 `labelSummaryDao.findByTrackingNo(sourceTrackingNumber)`，但在本次变更代码中看不到任何对该源单号归属、租户范围或调用方权限的校验。相比原先仅由当前 `trackingNumber` 截断推导源单号，这一改动把“源单选择权”交给了外部输入，扩大了可操作对象范围。 ```java String originTrackingNumber = this.pushLabelForm.getOriginTrackingNumber(); String sourceTrackingNumber; if (StrUtil.isNotBlank(originTrackingNumber)) { sourceTrackingNumber = originTrackingNumber; } else { String trackingNumber = this.pushLabelForm.getTrackingNumber(); sourceTrackingNumber = trackingNumber.substring(0, trackingNumber.length() - 1); } LabelSummary sourceLabelSummary = labelSummaryDao.findByTrackingNo(sourceTrackingNumber); ``` **影响：** 若上游 HTTP 入口可被低权限用户或跨租户调用，攻击者可构造任意 `originTrackingNumber`，把退件标签绑定到不属于自己的源运单，造成跨租户数据关联、越权修改或后续事件/状态流转污染。当前调用链给出了多个入口控制器，但实现未在本分片 diff 内，无法证明已存在充分的角色与归属校验，因此该新增输入面属于高风险越权点。
    修复建议与测试场景
    修复建议

    在使用 `originTrackingNumber` 前增加强制归属校验：校验当前用户/组织是否对 `sourceTrackingNumber` 对应的 `LabelSummary` 具有操作权限；若是退件场景，还应校验该源单与当前 `trackingNumber`、orgId/customerId 等业务主键的一致性。建议在 Facade/Controller 入口增加 `@PreAuthorize` 或等效角色校验，并在此处补充资源级校验，例如： ```java LabelSummary sourceLabelSummary = labelSummaryDao.findByTrackingNo(sourceTrackingNumber); if (sourceLabelSummary == null || !Objects.equals(sourceLabelSummary.getOrgId(), currentUser.getOrgId())) { throw new RRException("No permission to access source label"); } ``` 同时对 `originTrackingNumber` 建立白名单规则，仅允许与当前单据存在明确业务关联时覆盖默认推导值。
