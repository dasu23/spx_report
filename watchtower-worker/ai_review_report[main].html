    修复建议与测试场景
    修复建议

    将目录保持为可配置项并提供兼容默认值，例如 `value="${LOG_BASE_DIR:-logs/app}"`，部署环境再通过环境变量覆盖到 `/www/logs/${appId}`。如果必须使用绝对路径，至少在部署脚本中显式创建并授权该目录，并补充启动验证。

    测试场景

    在未创建 `/www/logs`、且应用以普通用户运行的环境启动 API/Worker，验证应用不会因日志目录不可写而启动失败，且日志能落到默认目录或配置目录。

    修复建议与测试场景

    确认所有采集、归档、清理规则已同步更新；若无法一次性完成联动，建议继续保留 `.json.log` 后缀，或同时输出兼容命名。

    测试场景

    在预发环境检查日志采集器的 glob/regex 是否仍能匹配小时切分文件，并验证新产生的日志能被正常采集到下游。

    这里将 `Authorization: Bearer` 直接与配置中的长期静态 service token 做等值比较，通过后即放行，但没有任何过期时间、时间窗、nonce 或轮换机制。只要该 token 泄露，攻击者可无限期重放使用，直到人工改配为止，属于认证失效风险。

    测试场景

    补充安全测试：使用一个历史签发的过期 token 请求受保护接口，预期返回 401；使用同一 token 在超出允许时间窗后重放，预期返回 401。
