严重 backend/watchtower-collector/src/main/java/com/watchtower/collector/query/SqlAssembler.java:24
存在 SQL 注入风险：`fields`、`table`、`orderBy.field` 等请求字段被直接拼接进 SQL（如 `String.join(", ", request.getFields())`、`append(request.getTable())`、`orderBy.getField() + " " + direction`）。`PreparedStatement` 这里只对 value 做了参数化，无法保护标识符位置；一旦白名单配置失误、校验遗漏或后续复用 `SqlAssembler` 时绕过 `WhitelistValidator`，攻击者即可注入任意 SQL 片段。
💡 修复建议：不要直接使用请求中的表名/列名拼接 SQL。改为基于服务端固定映射生成 SQL：将可选表名、字段名、排序字段定义为后端枚举或配置 ID，再由后端把 ID 映射为受信任的真实标识符；同时在 `SqlAssembler` 内再次做防御性校验，拒绝任何未命中受信任映射的标识符。
严重 backend/watchtower-collector/src/main/resources/application.yml:5
`collector.service-token` 配置为 `${COLLECTOR_TOKEN:changeme}`，在未显式注入环境变量时会退回到固定已知值 `changeme`。这会使 Collector 的 Bearer Token 认证在默认部署下退化为公开凭证，属于硬编码默认密钥/Token 风险。
💡 修复建议：移除默认明文 Token，改为启动时强制要求 `COLLECTOR_TOKEN` 存在，例如仅使用 `${COLLECTOR_TOKEN}` 并在启动校验中拒绝空值/弱值；同时补充启动失败提示，避免以已知默认凭证运行。
主要 backend/watchtower-api/src/main/resources/logback-spring.xml:6
日志根目录从相对路径改为绝对路径 `/www/logs/${appId}`，会把运行环境前提从“当前进程可写目录”提升为“宿主机必须存在且可写 `/www/logs`”。在本地开发、CI、容器非 root 运行或 Kubernetes 挂载未准备该目录时，Logback 初始化可能直接因权限或路径不存在而失败，导致服务启动异常或日志完全落盘失败。

主要 backend/watchtower-collector/src/main/java/com/watchtower/collector/config/DataSourceConfig.java:27
这里将 `HikariDataSource` 实例直接作为 `Map` 的 value 返回，Spring 只会管理这个 `Map` Bean，本身不会自动逐个关闭其中的连接池。应用停机或上下文刷新时，这些连接池可能不会释放，导致 JDBC 连接泄漏。
💡 修复建议：将数据源注册为可被 Spring 生命周期管理的 Bean，或为当前 `Map` Bean 增加销毁逻辑，遍历并显式关闭 `HikariDataSource`。例如让配置类实现 `DisposableBean`，或单独提供 `@Bean(destroyMethod = "close")` 的数据源定义；如果继续返回 `Map`，则至少在销毁阶段执行 `((HikariDataSource) ds).close()`。
主要 backend/watchtower-collector/src/main/java/com/watchtower/collector/config/RestClientConfig.java:13-14
`platformRestClient` 直接使用可配置的 `platform.url` 作为基地址，并无任何协议校验，同时在默认请求头中附带 Bearer Token。若部署时将该地址误配为 `http://` 或被降级到明文传输，服务令牌会随每次请求以明文在网络上传输，形成敏感凭证泄露风险。
