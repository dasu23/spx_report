严重 oms-domain/src/main/java/io/speedx/oms/domain/service/batchtask/processor/ApLmBillRecalculationProcessor.java:103-114
在分页重算过程中只要有任意页面失败，代码仍然先执行 `calculateTotalApLmBillAmount(apLmBill)`，随后才按失败结果返回。这样会把“部分 settlement 已重算、部分仍是旧值”的混合状态汇总到账单总金额，导致账单金额被错误落库，即使任务最终标记为失败。该路径属于资金计算错误/数据不一致。

严重 oms-domain/src/main/java/io/speedx/oms/domain/service/lastmile/LmCostCalculator.java:221-228
重量单位转换失败时，这里仅记录日志后直接返回原始重量，后续仍会继续参与费率计算。只要成本卡重量单位与包裹重量单位不同，任何非法/未覆盖的 UOM 或运行时转换异常都会把错误重量带入计费，直接产生错误金额。这个分支属于资金计算错误，不应静默降级。
💡 修复建议：将转换失败改为显式终止计费并抛出带错误码的业务异常，避免用错误重量继续算钱。例如：`catch (RuntimeException e) { BizException ex = new BizException("Weight conversion failed for tracking " + rateEntity.getTrackingNo()); ex.setCode(LmCostErrorCode.WEIGHT_UOM_INVALID.getCode()); throw ex; }`。如果必须兼容历史数据，也至少在进入计算前校验 `fromUom/toUom` 是否可转换。
