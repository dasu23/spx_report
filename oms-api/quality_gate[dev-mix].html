严重 oms-domain/src/main/java/io/speedx/oms/domain/service/ArRecalculationService.java:204-207
这里把 `confirmedAmount == null` 的应收强制置为 `0`，随后用 `computeUnpaidFromConfirmed` 以 `confirmedAmount - pay - deduct` 重算未付金额。对历史上尚未确认、或旧数据中 `confirmed_amount` 为空的 receivable，一旦触发 recalculation，`unpaidAmount` 会从原本基于 `totalAmount` 的应收直接变成 0 或负数，属于资金口径被新逻辑静默改写，可能导致少收款/错误核销。

严重 oms-domain/src/main/java/io/speedx/oms/domain/service/ArReceivableAsyncService.java:759-763
异步生成/刷新 AR 时新增逻辑在 `confirmedAmount` 为空时先写入 `0`，再按 `confirmedAmount - totalPay - totalDeduct` 计算 `unpaidAmount`。这会把未确认应收的未付金额直接清零，与 `totalAmount` 脱钩；一旦后台异步任务运行，旧单据或未确认单据的资金数据会被批量污染。

严重 oms-infrastructure/src/main/java/io/speedx/oms/infrastructure/db/finance/repository/ApLmBillSettlementRepositoryCustomImpl.java:50-58
