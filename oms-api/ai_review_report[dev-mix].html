major331-339数据一致性`apLmSettlementEvent()` 在处理异常时只记录日志并返回 `BaseResponse.error(...)`，不再抛出异常。对 Kafka listener 来说，这会把失败消息当作“已消费完成”处理，数据库抖动、分表缺失等临时故障都会直接丢失这条结算事件，后续不会自动重试，最终造成 AP LM settlement 缺失。对真正的处理失败重新抛出异常，让 Kafka 按既有失败策略处理；如果业务上需要跳过脏消息，至少把校验失败与系统异常分开处理，并把不可处理消息投递到 DLT。示例：`catch (Exception e) { log.error(..., e); throw e; }`。模拟 `apLmSettlementService.processSettlementEvent()` 抛出运行时异常，验证 listener 不会返回成功/错误响应后直接提交 offset，而是进入重试或死信流程。
