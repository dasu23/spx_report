    MAJORL741-745数据一致性
    `getRateCardVersionInfoCache(...)` 直接把 `cacheResult` 中的 `versionZipCodes`/`versionZones` 引用塞进返回 DTO。缓存对象是进程内共享状态，如果调用方对返回 DTO 内的集合做增删改，会反向污染缓存，后续请求读到被篡改的数据。
    修复建议与测试场景
    修复建议

    返回前做防御性拷贝，避免暴露缓存内部可变集合，例如：`dto.setVersionZipCodes(new HashMap<>(cacheResult.getVersionZipCodes()))`；`versionZones` 中如果 value 也是集合，也要继续拷贝内部集合，或统一返回不可变视图。

    测试场景

    构造一次缓存命中后，拿到返回 DTO 并修改 `versionZipCodes`/`versionZones`；随后再次调用同一 key，断言第二次返回结果不受第一次修改影响。

    MAJORL759-779状态管理
    `getZoneRateCardVersionCache(...)` 和 `getZipCodeRateCardVersionCache(...)` 直接返回缓存中的 `List<RateCardDto>` 实例。该列表一旦被上层排序、清空或追加元素，会直接修改共享缓存内容，导致跨请求数据串扰，属于典型的共享可变状态暴露问题。
    修复建议与测试场景
    修复建议

    不要直接暴露缓存值，至少返回副本：`return new ArrayList<>(rateCardDtos);`；如果希望彻底防止调用方修改，可返回 `Collections.unmodifiableList(new ArrayList<>(rateCardDtos))`。

    测试场景

    第一次调用后对返回列表执行 `clear()` 或 `sort()`，再以相同参数调用一次；修复前第二次结果会被第一次操作影响，修复后应保持原始缓存数据不变。
