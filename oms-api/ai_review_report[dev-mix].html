    CRITICALL203-207业务逻辑
    新增逻辑在 `feesCarriedInline=true` 时直接信任并使用 MQ 请求体中的 `charges` 作为结算金额来源（`buildRateResultFromInlineCharges(request.getCharges(), billDate)`），未做来源完整性校验、签名校验、金额边界校验或与服务端可重算结果比对。该改动把“应收金额”从数据库/服务端计算链路切换为消息载荷驱动，若消息被伪造、重放或篡改（含负数金额、异常 surcharge code、伪造 rateId/rateVersion），会直接进入结算持久化流程，导致少收/多收，属于资金计算高风险。
    修复建议与测试场景
    修复建议

    对 inline charges 增加“强校验后再入账”策略：1) 仅接受受信生产者签名（HMAC/数字签名）且验签通过的消息；2) 校验金额和 chargeCode 白名单（禁止负数/超限值/未知 code）；3) 关键字段（rateId、rateVersion、baseAmount、totalAmount）与服务端重算或数据库快照做一致性校验，不一致则拒绝入账并告警；4) 对 `feesCarriedInline` 增加幂等与重放防护（message id + TTL + 去重）。示例：在进入 `buildRateResultFromInlineCharges` 前先 `verifySignature(request)`、`validateChargesSchema(request.getCharges())`，并执行 `compareWithServerSideRecalc(...)`。

    测试场景

    构造一条 `feesCarriedInline=true` 的 MQ 消息，将 `charges` 中 FREIGHT 改为负数或极小值、追加未知 surcharge code，验证系统当前是否仍生成结算并落库；修复后应被拒绝并记录安全告警。
