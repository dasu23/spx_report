    MAJORL231-237边界条件
    `logError(String context, Throwable throwable)` 和带 `timeSpent` 的重载直接调用 `throwable.getMessage()`，当调用方传入 `null` 时会在降级日志路径上触发 `NullPointerException`。禁用日志器本应是兜底实现，这里反而引入了崩溃风险。
    修复建议与测试场景
    修复建议

    在读取异常信息前做空值保护，并提供兜底文案，避免降级路径崩溃。例如：`String errorMessage = throwable != null ? String.valueOf(throwable.getMessage()) : "unknown error"; logError(context, errorMessage);`。如果需要保留更多诊断信息，可在非空时额外输出 `Log.getStackTraceString(throwable)`。

    测试场景

    补充单测或集成测试：在 `DisabledLogger` 上分别调用 `logError("ctx", (Throwable) null)` 和 `logError("ctx", (Throwable) null, 123L)`，断言不会抛异常且能输出兜底日志。

    MINORL211-227性能隐患
    `DisabledLogger` 从原来的静默 no-op 改为每次 `log`/`logError` 都写 `android.util.Log.w`。如果设备长期处于“无凭证”状态，这些调用会把业务埋点全部转成本地日志，容易在高频路径上造成日志刷屏，干扰问题定位并带来不必要的 I/O 开销。
    修复建议与测试场景
    修复建议

    将这类降级日志限制为调试环境或做采样/限频处理。例如仅在 `BuildConfig.DEBUG` 下打印，或记录首次告警后静默：`if (BuildConfig.DEBUG && warned.compareAndSet(false, true)) { Log.w(...); }`。

    测试场景

    在无凭证场景下压测高频日志调用，验证 Logcat 不会持续刷屏；同时验证调试环境仍能看到一次性告警。

    modules/printer/android/src/main/java/io/speedx/printer/common/PrinterSettings.java1 Major1 个问题
    MAJORL23-25边界条件
    新增的 `setString` 直接将 `value` 写入 `settings` 和 `cached`，没有对 `fieldName`/`value` 做空值校验。该类下方 `getInt()` 会继续对对应字段执行 `Integer.parseInt(getString(...))`；一旦调用方传入 `null`，后续读取数值型配置时会触发 `NumberFormatException`/空值异常，表现为打印配置读取失败甚至崩溃。
    修复建议与测试场景
    修复建议

    在写入前显式校验参数，至少禁止 `fieldName` 为空且对 `value == null` 做兜底处理；如果允许清空值，应改为 `remove` 而不是写入 `null`。示例：`public void setString(String fieldName, String value) { if (fieldName == null || fieldName.isEmpty()) throw new IllegalArgumentException("fieldName is required"); if (value == null) { this.settings.remove(fieldName); this.cached.remove(fieldName); return; } this.settings.put(fieldName, value); this.cached.put(fieldName, value); }`

    测试场景

    补充用例：1）调用 `setString("LABEL_LENGTH", null)` 后再读取 `getLabelLength()`，应返回受控错误或默认值，而不是运行时崩溃；2）调用 `setString(null, "10")` 时应被明确拒绝。

    modules/printer/android/src/main/java/io/speedx/printer/impl/UrovoPrinterInstance.java3 Major3 个问题
    MAJORL127-130资源管理
    `printPDF` 循环中每页都会创建新的 `Bitmap`，但打印完成后没有回收。这里单页位图尺寸接近 812x1218，连续打印多页 PDF 时会快速堆积 native/Java 内存，容易在低内存设备上触发 OOM 或打印进程被系统杀掉。
    修复建议与测试场景
    修复建议

    在每页打印后立即回收位图，并用 `finally` 保证异常路径也释放资源，例如：`Bitmap printBitmap = renderPage(page, landscape); try { printBitmapAsZpl(printBitmap, landscape); } finally { if (!printBitmap.isRecycled()) { printBitmap.recycle(); } }`。

    测试场景

    在内存较小的 Android 设备上连续打印 10 页以上 PDF，验证打印过程中内存不持续增长且不会崩溃。

    MAJORL247-249业务逻辑
    `checkPrinterStatus()` 被改成了空实现，调用方即使在打印机离线、不可达或端口不可用时也会得到“状态正常”的结果。原有前置状态检查流程会失效，用户只能在后续实际写入时才看到失败，属于明显的功能回退。
    修复建议与测试场景
    修复建议

    恢复最基本的状态校验逻辑，至少在该方法内调用 `connect()`/`getOutputStream()` 并将连接异常转换为 `BusinessException`；如果设备支持状态指令，应继续返回缺纸、离线等明确状态。

    测试场景

    关闭或断开 Urovo 打印机后触发应用里的“检查打印机状态”流程，预期应直接返回连接失败，而不是静默成功。

    MAJORL66-99并发安全
    新增的网络打印实现把 `socket` 作为实例级共享状态在 `connect()`、`disconnect()`、`getOutputStream()` 中读写，但这些生命周期方法没有使用统一锁保护；只有 `print*` 的发送片段做了 `synchronized (this)`。并发调用时，一个线程可能在另一个线程获取/写入输出流期间关闭或替换 `socket`，导致打印内容丢失、连接抖动或持续失败，形成可被触发的服务不可用风险。
    修复建议与测试场景
    修复建议

    用同一把锁串行化 `socket` 的创建、关闭、获取和写入全过程，例如把 `connect()`、`disconnect()`、`getOutputStream()` 及 `print*` 全部纳入同一同步块，或改为每次打印使用独立连接对象，避免共享可变 `socket` 状态。

    测试场景

    补充并发测试：两个线程同时调用 `print()`/`printPDF()`，另一个线程交错调用 `disconnect()`；验证不会出现 `SocketException`、部分输出、错页或后续请求永久失败。

    src/stores/printerStore.ts1 Major1 个问题
    MAJORL34-63数据一致性
    新增的持久化逻辑在 `setPrinterIp`、`setPrinterModel`、`setPrintLandscape`、`setConnected` 中都采用 `AsyncStorage.getItem(...).then(...)` 的读改写模式，但这些写入彼此没有串行化，也没有基于最新内存态统一落盘。多个 setter 连续触发时，后一次回调可能基于更旧的 `saved` 快照覆盖前一次结果，导致 `printerModel`、`printLandscape` 或 `printerAddress` 在重启后丢失。
    修复建议与测试场景
    修复建议

    将打印设置持久化收敛为单个同步于当前 store 状态的写入函数，例如在每个 setter 里先 `set(...)`，再通过 `get()` 读取最新状态一次性 `AsyncStorage.setItem(STORAGE_KEY, JSON.stringify(...))`；或者引入串行写队列，避免并发读改写覆盖。

    测试场景

    连续快速执行“切换打印机型号 -> 切换横向打印 -> 连接打印机”，然后重启应用，验证 `printerModel`、`printLandscape`、`printerAddress` 都能稳定恢复。
