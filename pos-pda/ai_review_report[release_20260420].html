    app/encoding/relabel-scan.tsx1 Major 1 Minor2 个问题
    MAJORL70-72异常处理
    这里把打印改成了 `await printPDF/printPDFFromUrl`，但仍复用外层 `try/catch`。一旦打印失败，会直接进入 `catch`，导致本次已成功的 relabel 扫描结果被当成整体失败处理，页面无法进入后续 Validation 步骤。这和之前 fire-and-forget 的行为相比是明显回归。
    修复建议与测试场景
    修复建议

    将打印失败与 relabel 接口成功解耦：先保存 `relabelResult`、推进到下一步，再单独 `try/catch` 打印异常并提示用户。例如：`try { await printPDF(...) } catch (err) { showInfo('Print failed, please reprint manually') }`，不要让打印异常中断已成功的 relabel 流程。

    测试场景

    模拟 `relabelApi.scan` 成功但 `printPDF` 抛错，确认页面仍进入 Validation 步骤，且显示可重打提示。

    MINORL103-107API契约
    `validate` 请求新增了 `printDateTime`，但该值只在首次 relabel 成功后设置一次；后续点击 Reprint 时并没有更新这个时间。这样接口拿到的并不一定是“实际打印时间”，而是第一次生成标签时的时间，容易造成校验依据与真实操作不一致。
    修复建议与测试场景
    修复建议

    如果后端需要真实打印时间，应在每次成功打印/重打后更新 `printDateTime`，并在 `handleReprint` 的打印成功回调后重新 `setPrintDateTime(new Date().toISOString())`；如果只是记录标签生成时间，则应改名为更准确的字段，避免契约歧义。

    测试场景

    先完成 relabel，再等待一段时间执行 Reprint，随后做 validate，确认请求中的时间字段与最后一次实际打印时间一致。

    src/services/http.ts2 Major2 个问题
    MAJORL90-95安全风险
    响应拦截器新增把 `cfg.params`、`cfg.data` 以及 `data.data` 全量 `JSON.stringify` 后写入 SLS。对扫描/补打等接口，这些字段很可能包含运单号、customsRefId、打印内容，甚至可能包含 base64 标签数据。把完整请求体和响应体落日志会显著扩大敏感数据暴露面，并增加日志系统泄露时的影响范围。
    修复建议与测试场景
    修复建议

    日志中禁止记录完整请求/响应载荷。仅保留接口名、结果状态、错误码、耗时等必要元数据；若必须保留业务字段，需建立明确白名单并对 tracking number、refId、base64 等字段做脱敏/截断。

    测试场景

    调用带 `playSound: true` 的扫描或 relabel 接口，验证 SLS 日志不包含完整请求体、响应体、tracking number 或 base64 内容。

    MAJORL91-105性能隐患
    响应拦截器现在会对每个 `playSound` 请求同步执行 `JSON.stringify(data.data)`，并把完整 `resp` 拼进日志；失败分支也会把完整响应体写入错误日志。这里运行在扫码请求的主路径上，序列化大对象会直接阻塞 JS 线程。当前分片里 relabel 相关接口已开始扩展打印场景参数，后续一旦响应里带上较大的标签数据或明细对象，扫码后的首帧响应和提示音都会明显变慢。
    修复建议与测试场景
    修复建议

    日志里只保留稳定且小的字段，例如 `errorCode`、`duplicate`、`trackingNumber/customsRefId` 等；对对象日志做白名单提取或长度截断，避免在拦截器中序列化完整 `data.data`。例如：`const summary = data.data ? { trackingNumber: data.data.trackingNumber, duplicate: data.data.duplicate } : undefined;`，然后记录 `JSON.stringify(summary)`。

    测试场景

    构造一个 `playSound: true` 的接口返回大对象（如包含长字符串/标签内容），连续扫码 20 次，观察响应完成到页面更新/提示音触发的耗时是否增加；修复后应保持稳定且日志体积明显下降。
