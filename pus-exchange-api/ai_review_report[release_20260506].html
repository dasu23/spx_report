    pus-domain/src/main/java/io/speedx/pus/domain/utils/BlobRefreshUtil.java1 Critical 1 Major2 个问题
    CRITICALL21安全风险
    在组件初始化时将整个 `blobConfig` 序列化后写入 `info` 日志。该配置对象高概率包含 Azure Blob 访问凭证、账号标识或 SAS 相关敏感参数，一旦进入常规日志链路，会造成敏感信息泄露，符合敏感数据泄露高风险场景。
    修复建议与测试场景
    修复建议

    禁止直接序列化完整配置对象到日志。删除该日志，或改为仅记录脱敏后的非敏感字段（如是否启用、域名、容器名），并明确屏蔽账号密钥、连接串、SAS 策略等敏感属性。

    测试场景

    新增日志安全测试：构造包含 `accountKey`、`connectionString`、`sas` 等敏感字段的 `SpxAzureBlobConfig`，验证初始化过程中产生日志时不包含这些字段值。

    MAJORL37-57业务逻辑
    `refresh` 在 URL 以 `http://` 或 `https://` 开头时直接原样返回，导致已经带有过期 SAS 的完整 Blob URL 无法刷新。该工具类的职责是“重新生成 SAS token”，但对最常见的完整访问链接分支提前返回，会直接造成链接过期后仍返回不可用地址。
    修复建议与测试场景
    修复建议

    不要仅凭 scheme 判断为“可直接访问链接”。应先解析出 host/path，再仅对白名单直链域名放行，或对白名单之外的 Blob URL 重新生成 token。例如：`URI uri = URI.create(url); String path = uri.getPath(); String host = uri.getHost(); boolean directDomain = host != null && host.endsWith("speedx.io"); if (directDomain) { return url; }`，其余带 path 的 Blob URL 继续走 SAS 刷新逻辑。

    测试场景

    新增用例：输入 `https://<blob-host>/container/a.pdf?sv=old...` 时，应返回带新 token 的 `blobDomain + /container/a.pdf?...`，而不是原始 URL。
