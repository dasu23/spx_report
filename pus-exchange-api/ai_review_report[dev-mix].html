    pus-domain/src/main/java/io/speedx/pus/domain/properties/PickupResultCallbackProperties.java1 Major1 个问题
    MAJORL23-36并发安全
    该 `@ConfigurationProperties` + `@RefreshScope` Bean 暴露了多个可变的 `HashMap`/`List` 字段，且类上使用了 Lombok `@Data` 自动生成 getter/setter，调用方拿到的是可直接修改的全局共享配置引用。若后续业务代码在运行期对这些集合执行 `put/add/remove`，会在多线程回调处理场景下引入竞态，导致回调路由配置被意外篡改、请求误投递或拒绝服务。
    修复建议与测试场景
    修复建议

    将配置对象改为不可变结构：使用只读 getter/防御性拷贝，初始化后用 `Collections.unmodifiableMap/List` 包装；避免对外暴露可写集合引用，并优先采用构造器绑定的不可变配置类。

    测试场景

    补充并发测试：两个线程分别读取和修改 `clientOrgIdMaps/clients/extras` 时，验证业务线程无法观察到运行期可变更的共享状态；同时增加单测确认 getter 返回集合不可写。

    pus-domain/src/main/java/io/speedx/pus/domain/service/callback/PickupResultCallbackDispatcher.java1 Major 1 Minor2 个问题
    MAJORL29-37边界条件
    `resolve` 的注释声明“配置缺失时返回 null 跳过”，但这里直接调用 `buildOrgIdToClientNameMap()` 和 `props.getClients().get(...)`。当 `clientOrgIdMaps` 或 `clients` 未配置为非空时，会先触发 `NullPointerException`，导致整个回调链路异常，而不是按预期跳过。
    修复建议与测试场景
    修复建议

    在读取配置前补充空值保护，例如先判空 `props.getClientOrgIdMaps()` / `props.getClients()`，缺失时记录 warn 并返回 `null`。示例：`Map<String, List<Long>> clientOrgIdMaps = Optional.ofNullable(props.getClientOrgIdMaps()).orElse(Collections.emptyMap()); Map<String, PickupResultCallbackProperties.Client> clients = Optional.ofNullable(props.getClients()).orElse(Collections.emptyMap());`。

    测试场景

    增加单测：`clientOrgIdMaps=null`、`clients=null`、两者都为 `null` 时调用 `resolve(orgId)`，断言不会抛异常且返回 `null`。

    MINORL29-30,50-61性能隐患
    `resolve` 每次调用都会重新遍历 `clientOrgIdMaps` 构建一份 `orgId -> clientName` 映射。`PickupResultCallbackProcessor` 会按每个 `orgId` 调一次 `resolve`，同一批回调内会重复做相同构建，配置量增大时会产生不必要的 CPU 和对象分配开销。
    修复建议与测试场景
    修复建议

    将映射预构建并缓存，例如在属性刷新后重建一次，或在 `PickupResultCallbackProperties` 初始化完成后生成不可变映射供 `resolve` 直接查询。若配置支持动态刷新，需要在刷新回调里同步更新缓存。

    测试场景

    增加性能/行为测试：同一次 `process` 中包含多个 `orgId` 时，断言映射构建逻辑只执行一次；配置刷新后再校验新映射生效。

    pus-domain/src/main/java/io/speedx/pus/domain/utils/BlobRefreshUtil.java1 Critical 1 Major2 个问题
    CRITICALL75-103安全风险
    异常分支在 `catch` 中记录 `rawValue`，同样会把调用方传入的完整 Blob URL 写入日志。若 URL 自带 SAS token，异常路径会泄露可直接访问对象存储的临时凭证。
    修复建议与测试场景
    修复建议

    异常日志仅记录脱敏后的 blob 路径、异常类型和必要上下文，禁止记录包含 query/token 的原始 URL；`toSasUrl2` 的对应异常日志也应一并处理。

    测试场景

    传入带 SAS token 的 URL，并构造 `URI.create` 或 `generateBlobSasToken` 抛异常；验证错误日志不包含 `sig`、`token` 等敏感参数。

    MAJORL86-127业务逻辑
    `toSasUrl` 在处理绝对 URL 时直接使用 `URI.getPath()` 作为 `blobName`。如果外部传入的是不带 container 的 CDN/代理地址（例如历史上常见的 `/path/file.xlsx`），这里会把 `/path/file.xlsx` 传给 `generateBlobSasToken`，与注释中要求的“含 container 的签名路径”不一致，最终会生成错误 SAS 或直接导致文件访问失败。这个分支没有任何 container 存在性校验，属于新增的边界条件遗漏。
    修复建议与测试场景
    修复建议
