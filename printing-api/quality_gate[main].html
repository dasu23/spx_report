严重 printing-foundation/src/main/java/io/speedx/printing/util/UrlUtil.java:39-46
当前 SSRF 校验只对 `InetAddress.getByName(host)` 返回的单个地址做检查。对于同时解析出公网和内网地址的主机名，攻击者可以让这里命中公网 IP 通过校验，而实际发起连接时再解析到内网 IP，从而绕过“禁止访问私网/回环地址”的防护。
💡 修复建议：校验时应解析并检查主机名的全部 IP（如 `InetAddress.getAllByName(host)`），并且连接阶段复用已校验过的目标地址，避免再次基于域名解析。可改为先枚举全部地址并全部校验，再使用其中一个已校验的 `InetAddress` 构造连接，或直接拒绝多 A/AAAA 记录且含任一内网地址的域名。
严重 printing-foundation/src/main/java/io/speedx/printing/util/UrlUtil.java:60-61
`openConnection` 在 `validate(url)` 之后又重新执行 `URI.create(url).toURL().openConnection()`，导致校验与实际连接之间存在 TOCTOU 窗口。即使前置校验通过，后续这次重新解析域名仍可能因 DNS rebinding 连接到内网地址，SSRF 防护会失效。
💡 修复建议：不要在校验后再次按域名重新建连。应让 `validate` 返回已确认安全的 `URI`/`InetAddress`，并基于该结果直接建立连接；至少需要在真正发起连接前再次校验最终解析出的远端地址。

主要 printing-domain/src/main/java/io/speedx/printing/service/TemplateService.java:35-40
模板缓存采用 `get` 后再 `downloadTemplate` 再 `put` 的非原子流程。并发请求同一模板 ID 时，多个线程会同时触发下载并覆盖同一本地文件，带来重复远程请求和本地文件竞争，严重时会放大外部依赖压力。
💡 修复建议：改为原子化加载缓存，例如使用 `templateMap.computeIfAbsent(id, key -> downloadTemplate(this.findById(key)))`，并在映射函数内处理受检异常包装，避免同一 ID 的重复下载。
