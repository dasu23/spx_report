major554-571并发安全新增逻辑先查询 `labelPo.getNotes()`，再在内存中拼接字符串，最后执行 `updateNotes`。同一运单并发提交备注时，两个请求可能基于相同旧值分别计算 `newNotes`，后写入的事务会覆盖先写入结果，导致备注丢失。将“读取旧备注并追加”改成单条原子更新，或增加乐观锁/悲观锁保护。例如在 SQL 中直接基于当前值拼接：`update Label set notes = right(case when notes is null or notes = '' then :note else concat(notes, '; ', :note) end, 255) where trackingNumber = :trkNumber`，避免读写分离造成丢数据。并发发起两个针对同一 `trackingNumber` 的 `updateLabelNotesForm` 请求，分别写入不同备注，验证最终结果同时包含两条备注而不是只保留最后一次。
major562边界条件`form.getNotes().trim()` 在判空前直接调用 `trim()`，当 `notes` 为 `null` 时会抛出 `NullPointerException`，接口返回 500。该风险不能仅依赖 `@Valid` 消除，因为当前变更里没有展示 `UpdateLabelNotesForm.notes` 必然带有 `@NotBlank` 之类约束。先对原始值做空判断，再做 `trim()`。例如：`String notes = StringUtils.trimToEmpty(form.getNotes()); if (StringUtils.isBlank(notes)) { ... }`，后续统一使用 `notes` 变量。补充 `notes = null`、`notes = " "` 的请求用例，验证接口返回受控失败结果而不是抛出空指针。
