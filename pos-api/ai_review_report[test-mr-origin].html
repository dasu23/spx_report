    **原因：** 新增的管理端运维接口直接暴露了重置 webhook 重试次数的能力，但在变更代码中看不到任何鉴权/授权注解或数据范围校验（如 `@PreAuthorize`、角色注解、管理员校验、traceId 归属校验）。这会使能够访问该管理端路由的主体具备修改失败 webhook 重试状态的能力。 ```java @ApiOperation(value = "Reset try_times so failed webhook logs re-enter retry") @PostMapping("/retry-reset") public ResponseObject<Integer> resetTryTimes(@RequestBody @Valid ResetTryTimesReq req) { ``` **影响：** 未被明确限制的调用方可批量重置指定 `traceIds` 的 `try_times`，使原本失败且达到重试上限的 webhook 重新进入重试流程，可能导致未授权的业务重放、重复下游调用、状态扰动，属于典型运维能力越权风险。

    修复建议
