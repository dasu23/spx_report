    当 `customerRefId` 为空时，用 `System.currentTimeMillis()` 兜底不能保证唯一性。同一毫秒内的并发请求会生成相同值，而字段注释已明确要求该值“must be unique”，这会把原本可选字段引入偶发重复单号/幂等冲突问题。
    修复建议与测试场景
    修复建议

    改成稳定的唯一 ID 生成方式，例如 UUID、雪花算法，或复用现有业务唯一号生成器；至少应避免单纯使用毫秒时间戳。示例：`req.setCustomerRefId(IdUtil.fastSimpleUUID());`。

    测试场景

    并发构造多个 `customerRefId` 为空的请求，在同一毫秒内调用接口，校验不会产生重复 `customerRefId`。

    pos-service/src/main/java/com/miq/posservice/api/PusApi.java2 Major2 个问题
    MAJORL40-45业务逻辑
    这里在 `payload` 非空时直接反序列化返回，但完全忽略了响应里的 `success/code` 字段。只要对端返回了 `payload`，即使 `success=false`，上层也会继续走成功分支；如果失败场景下 `failList` 为空或字段缺失，`AbstractPartnerService` 会把远端失败误判为成功并继续更新业务状态。
    修复建议与测试场景
    修复建议

    先显式校验 `success` 或 `code`，失败时直接抛出异常；只有确认成功后再解析 `payload`。例如：`boolean success = root.path("success").asBoolean(false); if (!success) { throw new RRException(root.path("message").asText("PUS createPickup failed")); }`，然后再做 `payload` 转换。

    测试场景

    模拟 PUS 返回 `{ "success": false, "message": "downstream error", "payload": {} }` 或 `payload.failList` 缺失，断言 `createPickup` 抛出异常，且调用方不会把 webhook 状态更新为成功。

    MAJORL41-52安全风险
    新增日志在请求前打印完整 `body`，在失败时打印完整下游 `response`。`PickupCreateForm`/响应 payload 很可能包含寄件地址、联系人、追踪号等敏感业务数据；这些内容被直接写入日志后，任何具备日志访问权限的人员都可读取，扩大数据暴露范围。
    修复建议与测试场景
    修复建议

    移除请求/响应全文日志，改为记录调用结果、traceId、bookingCode 等最小必要字段；若必须保留排障信息，需对地址、电话、姓名、追踪号等字段脱敏，并限制为 debug 级且默认关闭。

    测试场景

    模拟 `createPickup` 调用，传入含地址/电话的 `PickupCreateForm` 并制造失败响应，确认日志中仅有脱敏或白名单字段，不包含原始请求体和原始响应体。
