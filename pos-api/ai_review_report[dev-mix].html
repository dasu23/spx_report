    修复建议

    保留自动 A-scan 作为增强逻辑，但不要移除最终兜底校验。建议在自动编码调用后重新校验 `label` 状态，例如：`if (!ContainerSortModeEnum.CROSS.equals(sortMode) && !LabelStatusEnum.ENCODED.name().equals(label.getLabelStatus())) { parcelStatus = ConnectParcelStatusEnum.NOT_ENCODED; return generateFailedResponse("Missing A-scan.     测试场景

    构造非 `CROSS` 的 Audit container，`label.getLabelStatus()` 为非 `ENCODED`，并 mock `parcelContainerService.autoEncoding(...)` 返回 `null`；预期应返回 `Missing A-scan...`，且包裹不能被加入 container。

    原先对非 `CROSS` 且非 `ENCODED` 的 Parcel 一律拒绝；新增逻辑中将该校验替换为 `autoEncoding(...)` 分支，但当 `autoEncodingBo == null` 时既不会拒绝，也不会回退到原有 `NOT_ENCODED` 校验，导致未完成 A-scan 的包裹可能继续进入后续 containerizing 流程，放宽了接口安全校验契约。
    修复建议与测试场景
    修复建议

    为 `autoEncodingBo == null` 明确设置失败分支，或在自动编码未返回有效结果时回退到原有 `LabelStatusEnum.ENCODED` 校验，保证非 `CROSS` 场景下未 A-scan 的 Parcel 不会被放行。

    模拟 `sortMode != CROSS`、`labelStatus != ENCODED` 且 `parcelContainerService.autoEncoding(...)` 返回 `null`，验证接口必须返回 `Missing A-scan...` 或等价拒绝结果，不能继续后续装箱校验。

    新增代码将 `autoEncodingBo.getMessage()` 直接拼接到失败响应中返回给调用方。该消息来自外部/下游编码流程，内容不可控，可能把内部错误细节、下游系统标识或调试信息暴露给前端，形成信息泄露面。

    修复建议

    对外返回固定且通用的错误文案或错误码，例如 `Auto A-scan failed, please retry or contact support.`；将下游返回的详细 message 仅记录在服务端日志中，并做好敏感字段脱敏。

    测试场景

    模拟 `parcelContainerService.autoEncoding(...)` 返回 `encodingResult=false` 且 `message` 含内部异常栈关键词、下游服务名或连接信息，验证接口响应不包含原始 message，只返回通用错误。
