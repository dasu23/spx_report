    backend-common-service/src/main/java/com/miq/backendcommonservice/mq/AbstractKafkaConsumerTemplate.java2 Major2 个问题
    MAJORL267-274异常处理
    发送 DLT 失败时仅记录日志，不中断当前处理流程；而调用方在重试耗尽/反序列化失败后仍会继续 `commit` 当前 offset。这样会导致“原消息处理失败 + DLT 也失败 + offset 已提交”的部分失败场景，消息既不会被业务重新消费，也没有进入死信队列，形成不可恢复的数据丢失。
    修复建议与测试场景
    修复建议

    将 DLT 发送结果纳入提交条件：仅当 DLT 发送成功时才提交 offset；发送失败时抛出异常或返回失败，让消息保持未提交以便后续重试。也可增加本地持久化补偿表，确保失败消息至少落一处可靠存储后再提交。

    测试场景

    模拟 `publisher.send()` 返回失败，触发重试耗尽路径，验证当前消息 offset 不会被提交，或会进入本地补偿存储；重启消费者后消息应可再次处理。

    🔒 安全/权限相关 · 1 项（点击展开）
    MAJORL214-216安全风险
    DLT 分支会把反序列化失败的原始消息体完整写入错误日志。Kafka 消息通常承载业务对象，新增代码在异常场景下直接输出 `record.value()`，一旦消息中包含手机号、地址、证件号、token 等敏感字段，会在日志系统中形成二次泄露，且异常消息往往更容易被集中检索和长期保留。
    修复建议与测试场景
    修复建议

    避免在日志中直接打印完整消息体，改为记录 topic/partition/offset/key 和截断后的摘要或哈希；如确需排障，至少对敏感字段做脱敏后再输出。例如仅记录 `valueLength`、`sha256(value)` 或最多前 128 个字符。

    测试场景

    构造一条包含敏感字段（如 phone/token/address）的非法 JSON 消息，触发反序列化失败，验证应用日志中不出现原始完整消息体，仅出现脱敏/摘要信息。
