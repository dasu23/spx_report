    backend/src/main/java/com/aidpm/common/util/EncryptionUtil.java2 Major2 个问题
    MAJORL14安全风险
    `Cipher.getInstance("AES")` 依赖提供方默认转换，实际通常会落到 `AES/ECB/PKCS5Padding`。ECB 无随机 IV，明文块模式会直接泄露结构特征，且当前实现也没有认证标签，无法检测密文篡改，不适合保护配置口令、Token 等敏感字段。
    修复建议与测试场景
    修复建议

    显式改为带认证的 AEAD 模式，例如 `AES/GCM/NoPadding`；加密时为每条数据生成随机 12 字节 nonce，与密文一起存储/传输；解密时校验 GCM tag。不要使用裸 `AES` 或 ECB 默认模式。

    测试场景

    补充测试：1）同一明文连续加密两次，结果应不同；2）篡改密文任意 1 字节后，解密应失败并返回受控异常；3）历史密文迁移场景下，新旧格式能被正确区分。

    MAJORL47-50异常处理
    `decrypt` 只捕获了 `GeneralSecurityException`，但 `Base64.getDecoder().decode(encryptedText)` 在收到非 Base64 内容时会抛出 `IllegalArgumentException`。这会绕过当前统一的 `RuntimeException("Decryption failed")` 包装，导致调用方对非法输入和解密失败的异常表现不一致，接口在脏数据或历史明文数据场景下容易直接报 500。
    修复建议与测试场景
    修复建议

    把 `IllegalArgumentException` 一并纳入处理，确保非法密文和密码学异常都走统一分支。例如：`try { ... byte[] decoded = Base64.getDecoder().decode(encryptedText); ... } catch (GeneralSecurityException | IllegalArgumentException e) { throw new RuntimeException("Decryption failed", e); }`。如需兼容历史明文，还应在解码失败时增加显式兜底策略。

    测试场景

    补充单测：1. `decrypt("not-base64")` 应返回统一异常类型；2. `decrypt("")` 和截断/损坏的密文应返回统一异常；3. 合法密文仍可正常解密。
