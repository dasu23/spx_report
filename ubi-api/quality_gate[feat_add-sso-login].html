主要 backend/server/src/main/java/com/ubi/server/controller/SsoAuthController.java:29-59
**原因：** 该 SSO 回调入口被 `@Public` 放行为匿名可访问，并直接将请求参数 `authToken` 交给 `ssoLoginService.handleCallback(authToken)` 处理后执行 `response.sendRedirect(redirectUrl)`，但在本分片 diff 中看不到任何对回调来源、一次性 state/nonce、或 token 绑定关系的校验。对于 SSO 登录回调，这类校验缺失会使任意人可直接构造请求命中登录完成流程。 ```java @Public @GetMapping("/callback") public void callback(@RequestParam("authToken") String authToken, HttpServletResponse response) throws IOException { requireEnabled(); String redirectUrl = ssoLoginService.handleCallback(authToken); response.sendRedirect(redirectUrl); } ``` **影响：** 若 `handleCallback` 未在服务层严格校验该 `authToken` 的签发方、时效性、一次性使用及与当前浏览器会话的绑定关系，攻击者可通过伪造/重放回调请求触发登录态建立或登录流程劫持，属于认证边界放宽风险。由于该入口是公开 GET 接口，攻击面直接暴露在外网。
💡 修复建议：在回调入口对应的服务逻辑中强制校验 SSO 防重放与防伪造要素：至少包含 state/nonce、token 时效、一次性消费、签发方校验，以及与发起登录时浏览器会话的绑定；若无法在服务层证明这些校验已存在，建议在控制器参数中显式接收并校验 `state`。同时避免仅凭单个 `authToken` 完成登录。示例：发起登录时生成并保存 state，回调时校验 state 后再兑换 token。
