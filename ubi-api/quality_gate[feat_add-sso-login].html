主要 backend/server/src/main/java/com/ubi/server/controller/SsoAuthController.java:29-40
**原因：** 该登录回调入口被 `@Public` 放行为匿名可访问，并直接将外部请求参数 `authToken` 交给 `ssoLoginService.handleCallback(authToken)` 处理后执行 `response.sendRedirect(redirectUrl)`，但在本分片 diff 中看不到任何与浏览器会话绑定的 `state`/nonce/一次性挑战值校验，也看不到对回调来源上下文的校验。 ```java @Public @GetMapping("/callback") public void callback(@RequestParam("authToken") String authToken, HttpServletResponse response) throws IOException { requireEnabled(); String redirectUrl = ssoLoginService.handleCallback(authToken); response.sendRedirect(redirectUrl); } ``` **影响：** 若服务层未额外做严格校验，攻击者可构造或复用他人的 `authToken` 直接调用该公开回调接口，触发受害者浏览器完成错误账号登录或登录态绑定，形成典型 SSO 登录 CSRF / 会话混淆风险。由于这是认证入口，一旦绑定错误身份，后续所有受保护资源访问都可能在错误主体下进行。

主要 backend/server/src/main/java/com/ubi/server/service/SsoLoginService.java:57
将一次性登录票据通过 URL 查询参数 `?token=` 拼接到前端跳转地址中，票据会暴露在浏览器地址栏、历史记录、代理/网关访问日志以及 Referer 中。虽然这里传递的是 ticket 而非最终 JWT，但 `exchange()` 可直接将其兑换为完整登录态，因此该票据本质上是短期有效的认证凭证，泄露后可被他人抢先兑换登录。
💡 修复建议：不要通过查询参数传递可兑换登录态的票据。改为：1）使用 URL fragment（`#token=`）仅在前端读取，避免进入大多数服务端日志与 Referer；或 2）在后端回调时通过 `Set-Cookie` 下发 `HttpOnly + Secure + SameSite` 的一次性票据，再由前端调用兑换接口；同时确保兑换接口在成功后立即失效并限制极短 TTL。
