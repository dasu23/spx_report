    backend/server/src/main/java/com/ubi/server/controller/SsoAuthController.java1 Major 1 Minor2 个问题
    MAJORL29-59权限与访问控制
    **原因：** 该 SSO 回调入口被 `@Public` 放行为匿名可访问，并直接将请求参数 `authToken` 交给 `ssoLoginService.handleCallback(authToken)` 处理后执行 `response.sendRedirect(redirectUrl)`，但在本分片 diff 中看不到任何对回调来源、一次性 state/nonce、或 token 绑定关系的校验。对于 SSO 登录回调，这类校验缺失会使任意人可直接构造请求命中登录完成流程。 ```java @Public @GetMapping("/callback") public void callback(@RequestParam("authToken") String authToken, HttpServletResponse response) throws IOException { requireEnabled(); String redirectUrl = ssoLoginService.handleCallback(authToken); response.sendRedirect(redirectUrl); } ``` **影响：** 若 `handleCallback` 未在服务层严格校验该 `authToken` 的签发方、时效性、一次性使用及与当前浏览器会话的绑定关系，攻击者可通过伪造/重放回调请求触发登录态建立或登录流程劫持，属于认证边界放宽风险。由于该入口是公开 GET 接口，攻击面直接暴露在外网。
    修复建议与测试场景
    修复建议

    在回调入口对应的服务逻辑中强制校验 SSO 防重放与防伪造要素：至少包含 state/nonce、token 时效、一次性消费、签发方校验，以及与发起登录时浏览器会话的绑定；若无法在服务层证明这些校验已存在，建议在控制器参数中显式接收并校验 `state`。同时避免仅凭单个 `authToken` 完成登录。示例：发起登录时生成并保存 state，回调时校验 state 后再兑换 token。

    测试场景

    1. 未经过正常 SSO 发起流程，直接请求 GET /auth/sso/callback?authToken=任意值，验证不会建立登录态；2. 使用已消费过的 authToken 重放请求，验证被拒绝；3. 使用其他浏览器会话发起、当前会话回调的 token/state 组合，验证被拒绝。

    MINORL44-47业务逻辑
    `/auth/sso/config` 用 `enabled && hasText(loginUrl)` 判断前端是否展示 SSO，但其余实际可用接口（`/callback`、`/exchange`、`/redirect-url`）的开关校验只检查 `ssoProperties.isEnabled()`。当配置为 enabled=true 但 `loginUrl` 为空时，`config()` 会返回不可用，而其他接口仍会继续执行，导致前后端对 SSO 可用性的判断不一致。
    修复建议与测试场景
    修复建议

    统一可用性判定逻辑，抽出一个共享方法，例如 `private boolean isSsoAvailable() { return ssoProperties.isEnabled() && StringUtils.hasText(ssoProperties.getLoginUrl()); }`，并让 `config()` 与 `requireEnabled()` 都基于同一条件判断；如果 `redirect-url` 实际依赖的不是 `loginUrl`，则应相应调整 `config()` 的判定条件，确保对外语义一致。

    测试场景

    增加用例：当 `enabled=true` 且 `loginUrl` 为空时，验证 `GET /auth/sso/config` 与 `GET /auth/sso/redirect-url`、`GET /auth/sso/callback` 的行为一致（要么都不可用，要么都可用并有明确约束）。

    MAJORL58安全风险
    将一次性登录取件码直接拼接到 URL 查询参数 `?token=` 中返回。该设计会使凭证暴露在浏览器地址栏、历史记录、前端埋点、反向代理访问日志以及 Referer 头中；一旦前端后续加载第三方资源或发生跳转，取件码可能被被动泄露。虽然取件码是一次性的，但在被前端兑换前仍可被窃取并完成登录，属于认证凭证泄露风险。
    修复建议与测试场景
    修复建议

    不要通过查询参数传递可兑换登录态的凭证。优先改为：后端回调时设置短时 HttpOnly + Secure + SameSite 的一次性 Cookie，然后前端调用 `/auth/sso/exchange` 时仅凭 Cookie 兑换；或将 ticket 放入 URL fragment（`#token=`，不会进入 Referer/大多数服务端日志）并确保前端首屏立即清除。

    测试场景

    验证 SSO 回调成功后，浏览器地址栏、服务端访问日志、前端埋点日志及第三方资源请求 Referer 中均不包含可兑换的 ticket；并验证 ticket 只能通过受保护通道兑换。
