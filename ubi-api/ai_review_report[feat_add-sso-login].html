    backend/server/src/main/java/com/ubi/server/controller/SsoAuthController.java2 Major2 个问题
    MAJORL29-40权限与访问控制
    **原因：** 该登录回调入口被 `@Public` 放行为匿名可访问，并直接将外部请求参数 `authToken` 交给 `ssoLoginService.handleCallback(authToken)` 处理后执行 `response.sendRedirect(redirectUrl)`，但在本分片 diff 中看不到任何与浏览器会话绑定的 `state`/nonce/一次性挑战值校验，也看不到对回调来源上下文的校验。 ```java @Public @GetMapping("/callback") public void callback(@RequestParam("authToken") String authToken, HttpServletResponse response) throws IOException { requireEnabled(); String redirectUrl = ssoLoginService.handleCallback(authToken); response.sendRedirect(redirectUrl); } ``` **影响：** 若服务层未额外做严格校验，攻击者可构造或复用他人的 `authToken` 直接调用该公开回调接口，触发受害者浏览器完成错误账号登录或登录态绑定，形成典型 SSO 登录 CSRF / 会话混淆风险。由于这是认证入口，一旦绑定错误身份，后续所有受保护资源访问都可能在错误主体下进行。
    修复建议与测试场景
    修复建议

    在回调链路中增加并强制校验与发起登录请求时绑定的 `state`（必要时同时校验 nonce、ticket 一次性使用、过期时间、User-Agent/会话绑定）。回调接口至少应显式接收并校验 `state`，校验失败时拒绝登录并不要重定向到成功页。例如： ```java @Public @GetMapping("/callback") public void callback(@RequestParam("authToken") String authToken, @RequestParam("state") String state, HttpServletResponse response) throws IOException { requireEnabled(); String redirectUrl = ssoLoginService.handleCallback(authToken, state); response.sendRedirect(redirectUrl); } ``` 并在 `SsoLoginService` 中校验 `state` 与服务端保存的会话挑战值完全一致且仅可使用一次。

    测试场景

    1. 浏览器 A 发起正常 SSO 登录，记录其 state；2. 浏览器 B 或未登录浏览器直接请求 `/auth/sso/callback?authToken=<他人token>&state=<缺失或伪造值>`；3. 期望服务端拒绝处理，不创建登录态、不跳转到成功页；4. 验证同一 state/token 重放时也被拒绝。

    MAJORL53-59安全风险
    公开接口 `/auth/sso/redirect-url` 从中心 SSO 拉取 `redirectUrl` 后“原样返回给前端”，但当前控制器层未见任何对返回 URL 的域名白名单、协议限制或目标校验。若上游 SSO 配置错误、被污染，或 `ssoGateway.fetchLoginRedirectUrl()` 可被间接操控，前端将获得并跳转到任意外部地址，形成开放重定向/钓鱼跳转风险。
    修复建议与测试场景
    修复建议

    在返回前校验 `redirectUrl`：仅允许 `https`，并限制 host 必须属于受信任 SSO 域名白名单；校验失败时拒绝返回。建议在控制器或服务层增加统一 URL 校验方法。

    测试场景

    模拟 `ssoGateway.fetchLoginRedirectUrl()` 返回 `http://evil.example/phish`、`javascript:alert(1)`、非白名单域名 URL，接口应拒绝并返回错误；对白名单 `https` 域名应正常通过。

    backend/server/src/main/java/com/ubi/server/service/SsoLoginService.java2 Major2 个问题
    MAJORL47-49业务逻辑
    本地用户查询直接按 `email.trim()` 与 `username` 精确匹配，但未做大小写归一化。SSO 返回的邮箱大小写与本地账号保存格式不一致时（如 `Alice@corp.com` vs `alice@corp.com`），会被误判为账号不存在，导致合法用户无法通过 SSO 登录。邮箱作为身份标识通常应按大小写不敏感处理。
    修复建议与测试场景
    修复建议

    在查询前统一归一化邮箱/用户名，例如转为小写后再匹配，并确保本地账号保存策略一致。示例：`String username = email.trim().toLowerCase(Locale.ROOT);`；若数据库已存在混合大小写数据，可改为大小写不敏感查询或增加规范化字段。

    测试场景

    补充测试：本地用户 `username=alice@example.com`，SSO 返回 `Alice@Example.com`；期望仍能成功签发 ticket，而不是返回 `account_not_found`。

    MAJORL57安全风险
    将一次性登录票据通过 URL 查询参数 `?token=` 拼接到前端跳转地址中，票据会暴露在浏览器地址栏、历史记录、代理/网关访问日志以及 Referer 中。虽然这里传递的是 ticket 而非最终 JWT，但 `exchange()` 可直接将其兑换为完整登录态，因此该票据本质上是短期有效的认证凭证，泄露后可被他人抢先兑换登录。
    修复建议与测试场景
    修复建议

    不要通过查询参数传递可兑换登录态的票据。改为：1）使用 URL fragment（`#token=`）仅在前端读取，避免进入大多数服务端日志与 Referer；或 2）在后端回调时通过 `Set-Cookie` 下发 `HttpOnly + Secure + SameSite` 的一次性票据，再由前端调用兑换接口；同时确保兑换接口在成功后立即失效并限制极短 TTL。

    测试场景

    模拟 SSO 回调成功后，检查浏览器地址栏、反向代理访问日志和后续跳转请求 Referer 是否包含 ticket；验证拿到该 ticket 的第二个客户端能否在 TTL 内调用兑换接口获取登录态。
