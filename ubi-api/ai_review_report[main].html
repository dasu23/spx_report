    CRITICALL30-32安全风险
    `rawQuery` 在 info 日志中直接记录了用户提交的原始 `query` 内容：`log.info("SLS raw query: caller={}, logstore={}, query={}, ...", ..., cmd.getQuery(), ...)`。查询语句通常会包含手机号、设备号、用户标识、过滤条件甚至临时凭证片段；将其完整写入应用日志会造成二次敏感数据泄露，且日志系统往往具备更广的可访问面与更长保留周期。
    修复建议与测试场景
    修复建议

    不要记录完整原始查询内容。改为记录查询摘要（如长度、哈希、模板名）或经过脱敏/截断后的内容；若确需排障，至少降级到受控 debug 日志并默认关闭。

    测试场景

    构造包含手机号、token 片段或邮箱等敏感过滤条件的查询调用 `POST /sls/raw-query`，确认应用日志中不出现原始查询明文，仅出现脱敏/摘要信息。

    MAJORL35-38权限与访问控制
    **原因：** `/sls/schema` 与原始查询接口同处 `@Public` 控制器下，新增代码中未见任何显式权限控制或最小化返回策略，方法直接返回 `slsRawQueryService.schema()`。结合上下文，`schema()` 会遍历 `knownLogstores()` 并返回各 logstore 的字段信息，这会向任意通过 API Key 认证的调用方暴露系统内部日志结构与字段枚举。 ```java @GetMapping("/schema") public SlsSchemaData schema() { return slsRawQueryService.schema(); } ``` **影响：** 即使攻击者暂时无法成功构造高价值查询，也可先通过 `/sls/schema` 枚举所有已知 logstore 及字段名，显著降低后续数据探测与敏感信息挖掘门槛，形成信息泄露与攻击面扩大。若 API Key 仅做身份认证而无范围控制，该接口会把跨 app 的内部观测模型暴露给不应知晓的主体。 --- **原因：** schema 接口会遍历所有已配置的 logstore，并向调用方返回每个 logstore 的索引字段清单，但未基于调用者身份做任何可见性控制。 ```java for (String ls : knownLogstores()) { List<SlsFieldData> fields = new ArrayList<>(); slsClient.getIndexFields(ls).forEach((f, t) -> fields.add(new SlsFieldData(f, t))); logstores.put(ls, fields); } ``` **影响：** 未授权或低权限用户可枚举日志存储名称与字段结构，为后续构造更精准的原始查询、定位敏感字段提供情报，扩大攻击面；结合 `/sls/raw-query` 的透传能力，会显著降低越权读取门槛。

    修复建议

    为 `/sls/schema` 增加与 `/sls/raw-query` 一致的授权校验，并按调用主体裁剪可见 logstore/字段；若该接口仅供内部工具使用，应移除 `@Public` 或限制到受信主体。至少应避免向所有有效 API Key 返回全量 schema。

    测试场景

    使用低权限或普通用户 API Key 调用 `GET /sls/schema`，验证不会返回未授权 logstore/字段；使用无 key 请求应被拒绝；不同用户 key 返回的 schema 应按授权范围差异化。
