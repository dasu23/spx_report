    web/app/api/config/route.ts1 Major1 个问题
    MAJORL9-12API契约
    `GET /api/config` 返回的是脱敏后的 `gitlabToken`，但同文件的 `PUT /api/config` 仍把 `gitlabToken` 当作真实 token 直接保存。任何先读后改的客户端如果未重新输入 token、而是回传 GET 结果中的 `glpat-xxx***`，都会把配置覆盖成无效 token，后续同步/校验全部失败。
    修复建议与测试场景
    修复建议

    不要把脱敏值放在 `gitlabToken` 字段里返回。可改为返回 `hasToken: true`，或返回空字符串并让前端单独处理“已配置”状态；如果必须返回脱敏展示值，也应让 `PUT` 显式忽略形如 `***` 的占位值。

    测试场景

    先调用 `GET /api/config` 获取配置，再仅修改 `gitlabUrl` 并原样回传返回体中的 `gitlabToken` 到 `PUT /api/config`；验证保存后再次调用 GitLab API 会失败。

    测试场景

    向 `POST /api/sync/claude-md` 发送 `{"groups":"a/b","projects":{}}`，预期接口不应抛异常；应返回 400 或按空数组处理。

    测试场景

    未完成配置时，先打开 `/settings`，再手动访问 `/dashboard` 或 `/sync`。验证页面不会先闪现主体内容，也不会提前发起依赖配置的业务请求。

    web/lib/gitlab.ts2 Major2 个问题
    MAJORL103-111业务逻辑
    `getFileFirstCommitDate` 试图获取 `CLAUDE.md` 的首次提交时间，但这里请求 GitLab commits API 时使用了 `per_page=1&order=default`。该接口默认返回的是最新提交，`commits[0]` 通常不是文件首次出现的提交，导致 `initializedAt` 和后续趋势图中的累计初始化曲线被系统性算错。
    修复建议与测试场景
    修复建议

    改为显式获取该文件相关提交的最后一页/升序结果，再取最早一条。例如先请求带分页头的列表并读取最后一页，或使用支持按时间正序返回的参数后取首条；至少要保证返回的是该文件最早一次提交日期。

    测试场景

    准备一个仓库：`CLAUDE.md` 在 2026-01-01 创建，2026-03-01 又被修改一次。执行状态同步后，应返回 `initializedAt=2026-01-01`，而不是 2026-03-01。

    MAJORL20安全风险
    `getGitLabUrl()` 直接信任配置中的 `gitlabUrl`，后续被用于拼接并发起服务端 `fetch`/`HEAD` 请求。若该配置可被低门槛修改，攻击者可将其指向内网或本机地址，触发 SSRF，对内网服务进行探测或利用。
    修复建议与测试场景
    修复建议

    对 `gitlabUrl` 做严格白名单校验，只允许预期的 `https` GitLab 域名；解析后拒绝 `localhost`、私有网段、链路本地地址和非 `http/https` 协议；在写入配置时和实际请求前都执行校验。

    测试场景

    将 `gitlabUrl` 配置为 `http://127.0.0.1:8080`、`http://169.254.169.254` 或内网域名，验证保存配置与后续同步/检查流程都会被明确拒绝。

    web/lib/store.ts2 Major2 个问题
    MAJORL160-163安全风险
    `writeConfig` 将 `gitlabToken` 原样写入 `data/config.json`。这是长期明文落盘的敏感凭证，一旦工作目录被备份、误打包、日志/诊断文件收集或主机上其他进程可读，将直接导致 GitLab API 凭证泄露。
    修复建议与测试场景
    修复建议

    避免将访问令牌明文持久化到仓库工作目录；优先改为环境变量或专用密钥管理服务。若必须落盘，应至少使用系统级密钥存储/加密，并限制文件权限为仅运行账户可读写。

    测试场景

    保存 GitLab 配置后，检查 `data/config.json` 是否仍可直接读到明文 token；同时验证文件权限是否阻止非运行账户读取。

    MAJORL55-61数据一致性
    `mergeCommits` 在按 `group` 覆盖旧数据时，使用 `c.project.startsWith(group + "/")` 判断项目是否属于该分组。对于同步目标正好是顶级项目组同名仓库（例如 group=`foo`，project=`foo`）的场景，这条记录不会被识别为同组旧数据，从而保留旧记录并叠加新记录，造成重复提交和统计翻倍。
    修复建议与测试场景
    修复建议

    分组匹配应同时覆盖“组本身”和“组下子项目/子组”，例如判断 `c.project === group || c.project.startsWith(group + "/")`。如果项目路径可能大小写不一致，还应统一规范后再比较。

    测试场景

    已有一条项目 `foo` 在 2026-04-01 的提交记录，再对 group=`foo`、相同日期范围执行一次同步；合并后该提交应只保留一条，统计结果不应翻倍。
