    web/app/api/config/route.ts1 Critical 1 Major2 个问题
    CRITICALL27-42安全风险
    在校验 GitLab 连接时将 `process.env.NODE_TLS_REJECT_UNAUTHORIZED` 设为 `"0"`，会全局关闭当前 Node 进程的 TLS 证书校验。后续所有 HTTPS 请求都可能接受伪造证书，导致 GitLab Token 在中间人攻击场景下泄露。
    修复建议与测试场景
    修复建议

    删除该全局设置，保持默认 TLS 校验开启；如确需支持自签名证书，应仅对当前请求显式配置受控 CA/agent，且不要影响进程内其他请求。

    测试场景

    构造使用无效/自签名证书的 GitLab HTTPS 端点，验证 `PUT /api/config` 在未显式信任该证书时必须失败；同时回归检查进程内其他 HTTPS 请求仍保持证书校验。

    MAJORL25状态管理
    `process.env.NODE_TLS_REJECT_UNAUTHORIZED = "0"` 在请求处理中直接修改了进程级全局状态，而且没有恢复。该路由一旦被调用，当前 Node 进程后续所有 HTTPS 请求都会关闭证书校验，属于典型的状态泄漏；并发请求下还会把这个副作用扩散到其它 API 调用。
    修复建议与测试场景
    修复建议

    不要在路由里修改全局 `process.env`。如果确实需要支持自签证书，应将是否跳过校验做成显式配置，并只在当前请求使用专用 agent/dispatcher，例如为本次 `fetch` 传入自定义 TLS 配置，或直接拒绝不受信证书。至少要避免全局写入并保证请求结束后恢复原值。

    测试场景

    先调用一次 `PUT /api/config`，再在同一进程内请求另一个使用 HTTPS 的 API，验证其 TLS 校验行为未被前一次请求改变；并补充并发场景测试，确认多个请求之间不会互相污染。
