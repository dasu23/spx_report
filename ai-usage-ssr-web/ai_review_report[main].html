    web/app/api/config/excluded-authors/route.ts1 Minor1 个问题
    MINORL10业务逻辑
    `excludedAuthors` 仅用 `a.trim()` 作为过滤条件，但写回配置时保留了原始字符串。输入如 `" alice@example.com "` 会被保存为带空格的值，后续在导出/统计阶段用精确匹配 `excludedAuthors.has(c.author)` 或 `has(c.email)` 时不会命中，导致排除规则看似保存成功但实际不生效。
    修复建议与测试场景
    修复建议

    在过滤时同步做标准化后再落库，例如：`const excludedAuthors = Array.isArray(body.excludedAuthors) ? body.excludedAuthors.filter((a: unknown): a is string => typeof a === "string").map((a) => a.trim()).filter(Boolean) : []`。

    测试场景

    提交 `excludedAuthors: [" alice@example.com "]`，随后触发依赖排除作者的统计或导出。预期该作者数据被排除；当前实现下仍会保留。

    web/app/settings/page.tsx1 Major1 个问题
    MAJORL75-115异常处理
    `saveExcludedAuthors` 和 `saveExcludedProjects` 把 `fetch` 成功发出请求等同于保存成功，但 `fetch` 在 4xx/5xx 时不会抛错。当前实现会在服务端返回失败时仍然清除 dirty 状态并显示“已保存”，导致用户误以为配置已落库。
    修复建议与测试场景
    修复建议

    检查 `resp.ok` 后再更新保存成功状态，并在失败时保留 dirty 状态及展示错误信息，例如：`const resp = await fetch(...); if (!resp.ok) { const err = await resp.json().catch(() => ({})); throw new Error(err.error || 'Save failed'); }`。同时不要使用空 `catch`，至少要设置错误提示。

    测试场景

    让 `/api/config/excluded-authors` 或 `/api/config/excluded-projects` 返回 500，点击保存后页面应保持未保存状态并展示失败信息，而不是显示成功。

    web/lib/store.ts1 Critical1 个问题
    CRITICALL32-34,161-163安全风险
    `AppConfig` 包含 `gitlabToken`，且 `writeConfig` 直接将整个配置对象明文写入 `data/config.json`。这会把 GitLab Private Token 持久化到工作目录中的普通 JSON 文件，增加被误提交、备份泄露、文件读取缺陷连带暴露等风险。

    修复建议

    不要将 `gitlabToken` 明文落盘到项目目录。优先改为仅保存在服务端环境变量或专用密钥管理中；如果必须持久化，至少将密钥与普通配置分离，使用系统密钥库/加密存储，并把文件权限收紧为仅进程账户可读写。

    保存一组包含测试 Token 的配置后，检查 `web/data` 及相关持久化文件，确认不存在可直接读取的明文 Token；同时验证读取配置接口不会回传完整 Token。
