    web/app/api/config/route.ts1 Major1 个问题
    MAJORL9-10,39API契约
    `GET /api/config` 返回的是脱敏后的 `gitlabToken`，但 `PUT /api/config` 会把收到的 token 原样持久化。只要前端复用 GET 返回值做编辑回填并直接提交，真实 token 就会被覆盖成 `前10位***`，后续所有 GitLab 同步都会失效。
    修复建议与测试场景
    修复建议

    将 `GET /api/config` 的响应拆分为 `maskedGitlabToken` 与可编辑字段，或让 `PUT` 在收到脱敏占位值时保留旧 token。更稳妥的做法是前端不回填 token 明文，后端也拒绝保存形如 `***` 的脱敏值。

    测试场景

    先保存一份有效配置；随后调用 `GET /api/config`，将返回结果原样提交到 `PUT /api/config`；再执行依赖 GitLab token 的接口，验证不会因 token 被覆盖为脱敏值而失败。

    MAJORL90-108异常处理
    流式 SSE 解析在 `reader.read()` 返回 `done` 后直接 `break`，没有继续处理 `buffer` 中残留的最后一条事件。只要服务端最后一个 `event/data` 块未以换行结尾，`done` 事件就会被静默丢弃，页面表现为同步已结束但成功结果不展示。相同问题在 CLAUDE.md 同步分支也重复存在。
    修复建议与测试场景
    修复建议

    在跳出循环前补一次对残留 `buffer` 的解析，或抽成通用的 SSE 事件消费函数统一处理。例如在 `if (done)` 分支中先对 `buffer.split("\n")` 的剩余内容按当前逻辑再跑一遍，再 `break`。

    测试场景

    模拟服务端最后一个 SSE 块为 `event: done\ndata: {...}` 且响应末尾没有额外换行，确认 `setResult`/`setClaudeMdResult` 仍会被调用。

    web/lib/gitlab.ts1 Critical 1 Major2 个问题
    CRITICALL19-26安全风险
    `getGitLabUrl()` 直接信任本地配置中的 `gitlabUrl`，后续所有 `fetch` 都会把 `PRIVATE-TOKEN` 发送到该地址。若该配置被误填或被恶意改成攻击者控制的域名，服务端会把 GitLab Token 主动发往外部主机，形成 SSRF + 凭证泄露。
    修复建议与测试场景
    修复建议

    在保存或使用 `gitlabUrl` 前做严格校验：仅允许 `https`，并限制到受信任的 GitLab 域名白名单；同时在发起请求前校验 `new URL(gitlabUrl).origin` 是否匹配白名单，不匹配则拒绝请求。

    测试场景

    将配置中的 `gitlabUrl` 设置为攻击者控制的测试域名，触发 `collectStats`/`checkClaudeMdForProjects`，验证当前实现会向该域名发送带 `PRIVATE-TOKEN` 的请求；修复后应被明确拦截。
