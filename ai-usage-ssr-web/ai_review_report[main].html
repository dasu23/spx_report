    web/app/api/config/excluded-authors/route.ts1 Minor1 个问题
    MINORL10-17业务逻辑
    `excludedAuthors` 只做了 `filter(a => typeof a === "string" && a.trim())`，但写回配置时保留了原始字符串，没有做 `trim`/大小写标准化。像 `" Alice@example.com "` 这样的值会被保存下来，后续用标准化后的 commit 邮箱比较时无法命中，表现为“已排除但不生效”。
    修复建议与测试场景
    修复建议

    在持久化前先标准化输入，例如 `body.excludedAuthors.map(v => typeof v === "string" ? v.trim().toLowerCase() : "").filter(Boolean)`，避免前后比较规则不一致。

    测试场景

    提交 `excludedAuthors: [" Alice@example.com "]`，然后导出或查询包含 `alice@example.com` 的 commit。当前实现不会过滤；修复后应正常生效。

    web/app/api/config/route.ts1 Major1 个问题
    MAJORL9-22API契约
    `GET /api/config` 返回的是脱敏后的 `gitlabToken`，但 `PUT /api/config` 又强制要求提交完整 token。这样一来，已配置用户如果只想修改 URL 或直接保存现有配置，前端拿到的只有 `glpat-xxx***`，重新提交时会被当成真实 token 去校验并失败，导致设置页的“编辑已有配置”流程不可用。
    修复建议与测试场景
    修复建议

    将读取和更新契约拆开：`GET` 只返回 `hasToken: true/false` 或空字符串，不回填脱敏 token；`PUT` 支持 `gitlabToken` 为空时沿用已保存的 token，或显式拒绝 `***` 形式的占位值并提示用户重新输入真实 token。

    测试场景

    先成功保存一组配置；随后调用 `GET /api/config` 取回脱敏 token，再仅修改 `gitlabUrl` 发起 `PUT`。当前实现会校验脱敏 token 失败；修复后应能保留旧 token 并更新 URL。
