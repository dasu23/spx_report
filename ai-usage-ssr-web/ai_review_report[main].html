    web/app/api/config/excluded-authors/route.ts1 Minor1 个问题
    MINORL10数据一致性
    `excludedAuthors` 只校验了 `a.trim()`，但写回时保存的是原始字符串，没有真正去掉首尾空格。用户若输入或粘贴了带空格的邮箱/作者名，后续按小写精确匹配时会无法命中过滤，表现为“已排除但仍出现在统计中”。
    修复建议与测试场景
    修复建议

    在过滤时直接规范化后再持久化，例如：`body.excludedAuthors.filter(...).map((a: string) => a.trim())`，必要时再统一转小写。

    测试场景

    提交 `{"excludedAuthors": [" alice@example.com "]}`，随后导出或查询统计，确认 `alice@example.com` 的提交会被正确排除。

    web/app/api/project-stats/route.ts1 Major1 个问题
    MAJORL10-13边界条件
    按字符串直接比较 `c.date` 与查询参数 `since/until` 会在 `c.date` 为完整 ISO 时间而查询参数为 `YYYY-MM-DD` 时产生边界错误。典型情况是 `until=2026-04-07` 时，`2026-04-07T10:00:00Z > 2026-04-07`，导致当天提交被全部排除，项目统计结果明显偏少。
    修复建议与测试场景
    修复建议

    将两侧统一转换为时间戳后再比较，或把 `until` 扩展到当天结束时刻再过滤。例如：`const sinceTs = since ? new Date(`${since}T00:00:00.000Z`).getTime() : null; const untilTs = until ? new Date(`${until}T23:59:59.999Z`).getTime() : null;`，过滤时使用 `const ts = new Date(c.date).getTime()` 与时间戳比较。

    测试场景

    构造一条 `c.date = 2026-04-07T12:00:00Z` 的提交，请求 `/api/project-stats?since=2026-04-01&until=2026-04-07`，期望该项目仍被统计到返回结果中。
