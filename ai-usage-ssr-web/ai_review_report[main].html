    测试场景

    将浏览器时区切到 `America/Los_Angeles`，在本地晚间打开同步页并点击 `7d/14d` 快捷按钮，确认输入框展示的都是本地自然日而不是次日。

    web/lib/gitlab.ts2 Major2 个问题
    MAJORL112-123业务逻辑
    `getFileFirstCommitDate` 通过 `repository/commits?path=...&per_page=1&order=default` 获取的实际上是该文件最近一次相关提交，而不是首次引入该文件的提交时间。这样 `initializedAt` 会随着后续编辑 `CLAUDE.md` 不断后移，导致初始化时间、趋势图累计曲线和状态表时间字段都不准确。
    修复建议与测试场景
    修复建议

    改为显式获取该文件的最早一次提交。可先请求该路径的 commits 列表并按时间升序取首条，或分页取最后一页的最后一条；如果 GitLab API 不支持升序参数，至少需要遍历分页直到拿到最早提交后再返回其日期。

    测试场景

    某仓库在 2026-01-01 首次添加 `CLAUDE.md`，并在 2026-03-01 再次修改该文件；执行同步后应始终返回 `2026-01-01`，不能变成 `2026-03-01`。

    MAJORL19-21安全风险
    新增逻辑将 `gitlabUrl` 直接从本地配置读取并作为后续 GitLab API 请求目标使用，而同文件请求会自动附带 `PRIVATE-TOKEN` 头。若该配置可被错误写入为攻击者域名或内网地址，服务端会向任意主机发送带凭证的请求，形成 SSRF 并泄露 GitLab Token。
    修复建议与测试场景
    修复建议

    对 `gitlabUrl` 做强校验后再使用：通过 `new URL()` 限制协议为 `https:`，仅允许预期的 GitLab 域名白名单，拒绝 `localhost`、回环地址、内网网段和带 `username/password` 的 URL；校验失败时直接拒绝保存或拒绝发起请求。

    测试场景

    将配置中的 `gitlabUrl` 改为 `https://attacker.example.com` 或内网探测地址，触发同步流程，验证服务端不会对该地址发起携带 `PRIVATE-TOKEN` 的请求，且接口返回明确的校验失败信息。
