    web/app/api/config/excluded-authors/route.ts1 Minor1 个问题
    MINORL10业务逻辑
    这里用 `a.trim()` 只做了过滤判断，但写回配置时仍保留原始字符串。若前端提交的作者邮箱包含首尾空格，例如 `"foo@bar.com "`，配置中会保存带空格的值，后续按邮箱精确匹配排除作者时将无法命中。
    修复建议与测试场景
    修复建议

    在过滤的同时完成归一化后再写入配置，例如：`const excludedAuthors = Array.isArray(body.excludedAuthors) ? body.excludedAuthors.filter((a: unknown): a is string => typeof a === "string").map((a) => a.trim()).filter(Boolean) : []`。如后续都按小写比较，这里也应统一转小写。

    测试场景

    调用 `PUT /api/config/excluded-authors` 传入带首尾空格的邮箱，再执行依赖 `excludedAuthors` 的统计/导出接口，验证该作者确实被排除。

    调用 `PUT /api/config/excluded-projects` 传入带空格的项目路径，然后访问导出或统计接口，确认该项目会被正确排除。

    web/app/settings/page.tsx1 Major1 个问题
    MAJORL97-104,137-144异常处理
    两个保存排除项的方法都把“请求已发出”当成“保存成功”处理，没有检查 `fetch` 返回的 HTTP 状态。只要服务端返回 4xx/5xx 且连接未中断，页面仍会清空 dirty 状态并展示已保存，导致用户误以为配置已落盘，实际刷新后会丢失。

    修复建议

    在设置成功状态前校验 `resp.ok`，失败时读取错误信息并保留 dirty 状态。例如：`const resp = await fetch(...); if (!resp.ok) throw new Error((await resp.json().catch(() => ({}))).error || HTTP ${resp.status});`，只有成功后再执行 `setExcludedDirty(false)` / `setExcludedProjectsDirty(false)`。

    测试场景

    分别让 `/api/config/excluded-authors` 和 `/api/config/excluded-projects` 返回 500，点击保存后预期不显示“已保存”、dirty 状态保留，并向用户展示错误信息。

    web/app/sync/page.tsx1 Major1 个问题
    MAJORL14-17边界条件
    默认日期通过 `toISOString().slice(0, 10)` 生成，这里使用的是 UTC 日期而不是用户本地日期。在美国时区等非 UTC 环境，凌晨/晚间访问页面时会出现日期偏移 1 天，导致同步范围默认值错误，直接影响查询结果。
    修复建议与测试场景
    修复建议

    改为按本地时区格式化日期，例如：`const formatLocalDate = (d: Date) => new Date(d.getTime() - d.getTimezoneOffset() * 60000).toISOString().slice(0, 10)`，或使用 `Intl.DateTimeFormat('en-CA')` 生成 `YYYY-MM-DD`。同样的修复也需要同步应用到快捷日期按钮逻辑。

    测试场景

    将浏览器时区设为 `America/Los_Angeles`，在本地时间 17:00-23:59 或 00:00-08:00 打开页面，验证默认 `since/until` 与系统本地日期一致，且 7/14/30/90 天快捷按钮不会偏移一天。

    web/lib/gitlab.ts1 Major1 个问题
    MAJORL225-251异常处理
    `collectStats` 的并发处理里直接在多个异步任务中向共享数组 `stats` 执行 `push`，同时还交叉修改 `started/completed` 计数。虽然单线程运行时通常不会崩溃，但结果顺序会受网络返回时序影响而非项目/提交时间，后续 `buildDisplayNames` 等逻辑又依赖“最后看到的数据”决定展示名，导致同一份数据多次同步后统计结果和显示名称可能不稳定。
    修复建议与测试场景
    修复建议

    避免在并发任务内直接写共享可变状态。让 `processProject` 返回该项目的提交数组与进度结果，批量 `await Promise.all(...)` 后再在外层按固定顺序合并；作者展示名也应基于明确的时间排序而不是数组遍历顺序。

    测试场景

    构造同一作者使用两个不同名字、不同项目响应时间不同的场景，重复执行同步多次；验证作者聚合名称和导出顺序保持一致。

    web/lib/store.ts2 Major2 个问题
    MAJORL26,157-160安全风险
    新增的配置存储将 `gitlabToken` 作为明文写入 `data/config.json`。该文件位于应用工作目录下，若目录被误打包、暴露为静态资源、进入构建产物/备份，或被同机其他低权限进程读取，会直接泄露 GitLab Private Token。
    修复建议与测试场景
    修复建议

    不要明文持久化访问令牌。改为仅将 Token 保存在服务端受控密钥存储或环境变量中；若必须落盘，至少使用操作系统级密钥库/加密存储，并限制文件权限为仅服务进程可读。

    测试场景

    配置 GitLab Token 后，检查工作目录下是否生成可直接读取的 `data/config.json`，并验证构建产物、备份目录、静态文件路由中不会包含该 Token。

    MAJORL53-60数据一致性
    `mergeCommits` 仍然只按 `group + date range` 清理旧数据，没有纳入本次新增的 `projectPaths` 维度。当前变更里 `collectStats(...)` 已支持按项目子集同步；如果用户只同步部分项目且 `group` 为空或覆盖较大分组，这里会先删除该时间段内其他未参与本次同步项目的历史提交，再只写回本次子集结果，造成数据被静默丢失。
    修复建议与测试场景
    修复建议

    将合并条件收敛到“本次实际同步的项目集合”。例如给 `mergeCommits` 增加 `projectPaths?: string[]` 参数，并在过滤时优先按项目集合判断：`if (projectPaths?.length) return !(projectSet.has(c.project) && c.date >= since && c.date <= until)`；仅在未传项目集合时再退回现有 group 逻辑。

    测试场景

    准备 A/B 两个项目在同一日期区间内的历史数据，只对 A 执行一次局部同步；验证合并后 B 的既有提交仍然保留。
