    web/app/api/config/route.ts1 Major1 个问题
    MAJORL24并发安全
    在请求处理过程中直接写 `process.env.NODE_TLS_REJECT_UNAUTHORIZED = "0"` 会修改整个 Node 进程的全局 TLS 行为。一次配置校验请求成功执行后，后续所有同进程内的 HTTPS 请求都会继续跳过证书校验，属于跨请求共享状态污染，可能导致后续同步、导出等功能在错误的 TLS 配置下运行。
    修复建议与测试场景
    修复建议

    不要在路由里修改全局环境变量。改为仅对当前 GitLab 校验请求使用受控的 HTTPS agent，或在配置中显式支持“允许自签名证书”并把该行为限制在单次 fetch 调用范围内。

    测试场景

    并发发起两个请求：1）调用 `PUT /api/config` 校验自签名 GitLab；2）随后调用依赖 HTTPS 的其他 API，验证后者不会因前一个请求而永久关闭证书校验。

    MAJORL22-29边界条件

    web/app/sync/page.tsx1 Major1 个问题
    MAJORL89-112边界条件
    SSE 解析把 `currentEvent` 定义在每次 `reader.read()` 的循环内部，导致事件类型不会跨 chunk 保留。只要服务端把 `event: done`/`event: error` 和对应的 `data:` 拆到两个传输块里，下一轮读取时 `currentEvent` 会被重置为空，最终 `done`/`error` 事件被静默丢弃，页面表现为同步结束后没有结果或错误提示。相同问题在 CLAUDE.md 同步逻辑中也存在。
    修复建议与测试场景
    修复建议

    将 `currentEvent` 提升到 `while` 循环外部，按 SSE 状态机持续解析，并在流结束后补处理剩余 `buffer`。示例：`let currentEvent = ""; while (...) { ... }`，不要在每次读块时重新初始化。

    测试场景

    为 `/api/sync` 和 `/api/sync/claude-md` 构造分块响应：第一块只返回 `event: done\n`，第二块再返回 `data: {...}\n\n`，确认页面仍能正确展示成功/失败结果。

    web/components/app/config-guard.tsx1 Major1 个问题
    MAJORL11-30状态管理
    `checked` 在进入 `/settings` 时会被置为 `true`，但后续从设置页跳转到其他页面时，新的配置检查开始前没有先重置为 `false`。这样会沿用旧的 `checked=true` 直接渲染子页面，直到 `/api/config/status` 返回后才可能重定向，产生明显的页面闪现，且在慢网或接口慢时更容易复现。
    修复建议与测试场景
    修复建议

    在非 `/settings` 路径进入检查前先执行 `setChecked(false)`，仅在确认已配置或当前就是设置页时再置回 `true`。例如在 `useEffect` 开头改成：`if (pathname === "/settings") { setChecked(true); return } setChecked(false); ...`。

    测试场景

    先访问 `/settings`，再在未配置状态下跳转到 `/dashboard`，并人为延迟 `/api/config/status` 响应，验证不会先短暂渲染 dashboard 内容再跳回设置页。

    web/lib/gitlab.ts1 Major1 个问题
    MAJORL108-116业务逻辑
    `getFileFirstCommitDate` 通过 `per_page=1&order=default` 取到的是该文件最近一次相关提交，而不是第一次提交时间。这样会把 `initializedAt` 记录成最新修改日期，导致 CLAUDE.md 初始化时间、累计趋势线和同步结果文案都出现系统性偏差。
    修复建议与测试场景
    修复建议

    改为显式获取最早一条提交，再取其日期；如果 GitLab API 不支持直接升序，至少拉取最后一页或使用可确认返回首个提交的排序参数。示例：先请求提交列表并按 `created_at` 升序排序后取最早值，或查询完后使用 `commits.at(-1)` 作为初始化时间。

    测试场景

    准备一个仓库：`CLAUDE.md` 在 2026-01-01 首次提交，2026-03-01 再次修改。执行 CLAUDE.md 状态同步后，断言 `initializedAt` 为 `2026-01-01`，趋势图在 1 月对应时间点开始累计，而不是 3 月。

    web/lib/store.ts1 Major1 个问题
    MAJORL167安全风险
    `writeConfig` 将包含 `gitlabToken` 的完整配置直接以明文 JSON 写入 `data/config.json`。该 token 属于高敏感凭证，当前实现没有做加密、脱敏或最小化权限控制，一旦工作目录被备份、误打包、日志/工件归档或同机其他低权限用户可读，GitLab 凭证会直接泄露。
    修复建议与测试场景
    修复建议

    不要将 GitLab token 以明文持久化到应用数据目录。优先改为从环境变量或专用密钥管理服务读取；若必须落盘，至少单独存放敏感字段并使用系统密钥库/加密方案，同时在写文件时显式设置严格权限（如仅进程账户可读写）。

    测试场景

    配置一个有效 `gitlabToken` 后触发保存，检查生成的 `data/config.json` 是否仍能直接读取到明文 token；再验证文件权限是否限制为仅应用运行账户可读写。
