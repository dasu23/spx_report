    MINORL10数据一致性
    这里用 `trim()` 仅做了过滤判断，但写回的仍是原始字符串。若前端提交 `" foo@example.com "` 这类值，会被保存为带空格的作者标识，后续按邮箱精确匹配时无法命中，导致排除作者配置失效。
    修复建议与测试场景
    修复建议

    在持久化前做标准化处理，例如：`const excludedAuthors = Array.isArray(body.excludedAuthors) ? body.excludedAuthors.filter((a): a is string => typeof a === "string").map((a) => a.trim().toLowerCase()).filter(Boolean) : []`。

    测试场景

    提交 `excludedAuthors: [" foo@example.com "]`，随后查询统计或导出，确认 `foo@example.com` 的提交会被正确排除。

    web/app/api/config/route.ts1 Major 1 Minor2 个问题
    MAJORL24状态管理
    这里直接修改 `process.env.NODE_TLS_REJECT_UNAUTHORIZED`，会把 TLS 校验开关变成进程级全局状态。一次 `PUT /api/config` 成功执行后，后续同进程内其他请求的 `fetch` 也会持续跳过证书校验，产生跨请求状态泄漏，行为不可预期。
    修复建议与测试场景
    修复建议

    不要在请求处理中修改全局环境变量。改为仅对本次校验请求使用专用的 HTTPS agent/dispatcher；如果运行时不支持，应至少在 `try/finally` 中恢复原值。例如：为该次 `fetch` 传入关闭校验的专用 agent，而不是写 `process.env`。

    测试场景

    先调用一次 `PUT /api/config`，再访问一个内部会执行 HTTPS 请求的接口，验证其 TLS 校验行为没有被前一次请求永久改变。若目标站点证书无效，后续请求仍应按预期失败。

    修复建议

    对外返回固定的通用错误文案，详细异常仅写入受控的服务端日志，并对日志做敏感信息脱敏。

    测试场景

    构造 DNS 失败、证书错误和连接超时等异常，确认 API 响应不暴露底层错误细节，只返回统一错误码/文案。

    web/app/api/stats/route.ts1 Major1 个问题
    MAJORL23-29边界条件
    日期过滤直接用字符串比较 `c.date` 与查询参数 `since/until`。当提交时间包含时分秒（如 ISO 时间戳）而查询参数是 `YYYY-MM-DD` 时，`c.date > until` 会把截止当天的提交全部排除，导致统计结果少数据。
    修复建议与测试场景
    修复建议

    先把边界统一成时间值再比较，例如将 `since` 解析为当天 00:00:00，将 `until` 解析为当天 23:59:59.999，或统一把提交日期截成 `YYYY-MM-DD` 后再比较。示例：`const sinceTs = new Date(`${since}T00:00:00.000Z`).getTime(); const untilTs = new Date(`${until}T23:59:59.999Z`).getTime(); if (ts < sinceTs || ts > untilTs) return false`。

    测试场景

    准备一条 `c.date = 2026-04-07T12:30:00Z` 的提交，查询 `since=2026-04-01&until=2026-04-07`，应仍被包含。

    web/lib/gitlab.ts2 Major2 个问题
    MAJORL18-24安全风险
    GitLab 基础地址改为从本地配置动态读取后，后续请求会无校验地把 `PRIVATE-TOKEN` 发送到该地址。若 `gitlabUrl` 被误配或被恶意篡改为攻击者控制的域名，服务端会主动向外部地址发起请求并携带私有 Token，形成 SSRF 与凭证外带风险。
    修复建议与测试场景
    修复建议

    对 `gitlabUrl` 做严格校验：仅允许 `https`、限制到受信任域名/网段白名单，并在保存配置时拒绝非常规协议、内网探测地址和非预期主机；发送 `PRIVATE-TOKEN` 前再次校验目标地址。

    测试场景

    将配置中的 `gitlabUrl` 设置为攻击者控制的测试域名，触发同步；验证服务端是否会向该域名发起请求并在请求头中携带 `PRIVATE-TOKEN`。若修复正确，请求应被保存配置或发起请求阶段拦截。

    MAJORL95-106业务逻辑
    `getFileFirstCommitDate` 通过 `/repository/commits?path=...&per_page=1&order=default` 获取 `CLAUDE.md` 的“首次提交时间”，但这组参数拿到的通常是该文件最近一次相关提交，而不是第一次提交。结果是 `initializedAt` 会在文件后续被修改时不断向后漂移，进而把初始化状态表和趋势图中的 CLAUDE.md 初始化时间统计错。
    修复建议与测试场景
    修复建议

    改为明确获取最早一次提交，再取其日期。可选做法：1. 拉取该文件提交列表后取最后一条；2. 若 GitLab API 支持升序排序，显式指定升序；3. 至少补充注释并用测试锁定“首次提交”语义。

    测试场景

    某项目先在 2026-01-01 新增 `CLAUDE.md`，再在 2026-03-01 修改该文件。同步后 `initializedAt` 应保持为 `2026-01-01`，不能变成 `2026-03-01`。

    web/lib/store.ts2 Major2 个问题
    MAJORL150-165安全风险
    新增配置持久化将 `gitlabToken` 以明文 JSON 形式写入 `data/config.json`。这会把高敏感凭证长期落盘，若运行目录被误共享、打包进制品、被日志/备份收集或被其他本机用户读取，Token 将直接泄露。
    修复建议与测试场景
    修复建议

    避免明文持久化敏感 Token；优先改为从环境变量或专用密钥管理服务读取。若必须落盘，至少使用系统级密钥存储/加密方案，并限制文件权限为仅进程账号可读写。

    测试场景

    保存配置后检查生成的 `data/config.json`，确认当前实现可直接读取明文 `gitlabToken`；修复后应无法从文件内容直接恢复原始 Token，且文件权限应收紧。

    MAJORL52-59数据一致性
    `mergeCommits` 只按 `group` 和日期范围删除旧数据，没有覆盖这次新增的“按 `projectPaths` 同步子集项目”场景。当前如果用户在 `group` 为空时只同步部分项目，同一日期范围内其他未同步项目的历史提交也会被一起删掉，导致本地统计数据丢失。
    修复建议与测试场景
    修复建议

    让 `mergeCommits` 接收本次实际同步的项目范围，并只删除这些项目在 `[since, until]` 内的旧记录。例如新增 `projectPaths?: string[]` 参数：当 `projectPaths` 非空时，仅对 `project` 命中该集合的记录执行覆盖；否则再回退到现有的 `group` 逻辑。

    测试场景

    准备 A、B 两个项目的历史提交数据；在 `group` 为空的情况下仅同步 A 项目某个日期区间。期望结果：A 项目该区间数据被刷新，B 项目同区间历史数据保持不变。
