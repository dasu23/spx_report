    MINORL10数据一致性

    修复建议与测试场景
    修复建议

    在过滤后同步做标准化，例如 `body.excludedAuthors.filter(...).map((a: string) => a.trim())`，必要时再统一转小写。

    测试场景

    调用 `PUT /api/config/excluded-authors` 提交带前后空格的邮箱，再执行依赖该配置的统计/导出接口，验证该作者确实被排除。

    web/app/api/config/route.ts1 Major1 个问题
    MAJORL24-29并发安全
    `PUT /api/config` 在请求处理过程中直接写 `process.env.NODE_TLS_REJECT_UNAUTHORIZED = "0"`。这是进程级全局状态，会影响同一 Next.js 进程后续的所有 HTTPS 请求，而不是仅当前这次 GitLab 连通性校验；一旦被某次请求设置，其他接口的网络行为也会被永久改变，属于典型状态泄漏。
    修复建议与测试场景
    修复建议

    不要在请求内修改 `process.env`。改为仅对当前校验请求使用独立的 TLS/HTTP 配置，例如通过专用 agent/dispatcher 发起请求，或在底层 GitLab 客户端中显式配置是否跳过证书校验，并将作用域限制在单次调用内。

    测试场景

    并发发起一次 `/api/config` 和一次依赖 HTTPS 的其他服务端请求，验证配置校验不会改变后者的 TLS 行为；连续两次调用后也应确认全局环境变量未被污染。

    web/app/settings/page.tsx1 Critical 1 Minor2 个问题
    CRITICALL161前端安全
    页面在初始化时将 `data.gitlabToken` 直接写入前端状态，说明 `/api/config` 会把 GitLab Token 明文返回给浏览器。该密钥随后可被页面显隐按钮直接展示，也可被任意前端脚本、浏览器扩展或 DevTools 读取，属于敏感凭证泄露。
    修复建议与测试场景

    不要在读取配置接口中返回明文 Token。`GET /api/config` 仅返回 `configured`、`gitlabUrl` 及脱敏后的占位信息；前端保持 `gitlabToken` 为空，仅在用户主动输入新 Token 时通过 `PUT /api/config` 提交更新。已有 Token 如需提示，返回固定掩码值而非真实内容。

    测试场景

    访问设置页并抓取 `/api/config` 响应，断言响应体不包含明文 `gitlabToken`；同时验证页面首次加载后输入框默认不持有真实 Token，切换“显示”也不会出现真实凭证。

    MINORL56-74异常处理
    `fetchAuthors` 和 `fetchProjects` 对响应直接执行 `r.json()`，没有检查 `r.ok` 或返回结构是否合法。接口返回 4xx/5xx 或非 JSON 时会静默吞掉异常，页面会表现为空列表，用户无法区分“暂无数据”和“加载失败”。
    修复建议与测试场景
    修复建议

    先校验 `resp.ok`，失败时设置独立错误状态或提示文案，而不是空 catch。示例：`const resp = await fetch(url); if (!resp.ok) throw new Error('Failed to load authors')`，并在 `catch` 中调用 `setError(...)` 或显示局部错误占位。

    测试场景

    分别让 `/api/authors` 和 `/api/project-stats` 返回 500、返回非 JSON，确认设置页能展示明确错误提示而不是静默显示空数据。

    web/app/sync/page.tsx1 Major1 个问题
    MAJORL13-17边界条件
    默认日期使用 `toISOString().slice(0, 10)` 生成 `<input type="date">` 的值，这会按 UTC 截断日期。在非 UTC 时区下，页面打开时间接近本地零点时会出现前一天/后一天的偏移，导致同步范围错误；同类写法在下方快捷日期按钮中也重复出现。
    修复建议与测试场景
    修复建议

    改为按本地时区格式化日期，而不是直接截取 ISO 字符串。例如提取一个 `formatLocalDate(date)`：`const offset = date.getTimezoneOffset(); const local = new Date(date.getTime() - offset * 60000); return local.toISOString().slice(0, 10)`，并在默认值与 7/14/30/90 天快捷按钮里统一复用。

    测试场景

    将浏览器时区设为 `America/Los_Angeles` 或 `Asia/Shanghai`，在本地 00:00-08:00 时间段打开页面并点击快捷日期按钮，确认 `since/until` 与本地自然日一致。

    web/components/dashboard/project-select.tsx1 Major1 个问题
    MAJORL20-35状态管理
    `groups` 变化时会直接发起新的 `/api/projects` 请求，但没有取消旧请求或校验响应归属。用户快速切换分组时，较早请求可能后返回并覆盖较新的 `projects` 列表，同时用过期的 `value` 计算并回写选中项，造成项目列表与当前分组不一致，甚至误清空用户刚选中的项目。
    修复建议与测试场景
    修复建议

    为每次请求绑定版本号或使用 `AbortController`，仅处理最后一次请求的响应；同时避免在异步回调里直接使用可能过期的 `value`。例如在 effect 内创建 `const controller = new AbortController()`，请求时传入 `signal`，cleanup 中 `controller.abort()`，并在响应前比较当前请求 token。

    测试场景

    先选择分组 A，再快速切换到分组 B/C，模拟 A 的请求比 C 更晚返回；确认最终展示的项目列表和已选项目始终与当前分组一致，不会被旧响应覆盖。

    web/components/dashboard/trend-chart.tsx1 Major1 个问题
    MAJORL50-56边界条件
    周模式下通过本地时区 `Date` 计算周日，再用 `toISOString().slice(0, 10)` 作为截止日期。浏览器处于 UTC+ 时区时，本地周日 00:00 转成 UTC 可能变成前一天，导致周日初始化的项目被漏算到下一周，统计结果随客户端时区变化。
    修复建议与测试场景
    修复建议

    不要把本地 `Date` 再转成 UTC 字符串做日期比较。改为按 UTC 计算周截止日，或直接基于 ISO 周字符串生成纯日期文本，避免 `toISOString()` 引入时区偏移。例如使用 `Date.UTC(...)` 和 `getUTC*`/`setUTC*`。

    测试场景

    在 UTC+8 环境下，某项目 `initializedAt` 为周日 `2026-04-12`。切到周视图时，该项目应计入 `2026-W15`，不应被延后到下一周。

    web/lib/gitlab.ts2 Major2 个问题
    MAJORL29安全风险
    将 `process.env.NODE_TLS_REJECT_UNAUTHORIZED` 设为 `"0"` 会全局关闭 Node.js 对 HTTPS 证书的校验，后续所有到 GitLab 的请求都可被中间人伪造证书拦截，`PRIVATE-TOKEN` 也会随请求头一并暴露。该行为属于新增的传输层安全降级。
    修复建议与测试场景
    修复建议

    不要全局关闭 TLS 校验。改为为自签名证书正确配置受信任 CA，或仅在明确的本地开发开关下按请求使用受限的 HTTPS agent，并确保生产环境始终校验证书。

    测试场景

    在启用自签名/错误证书的 GitLab 代理环境中验证：生产配置下请求应因证书校验失败而拒绝；仅开发开关开启时才允许连接，且不会影响其他 HTTPS 请求。

    MAJORL92-126业务逻辑
    `getFileFirstCommitDate` 通过 `per_page=1&order=default` 只取第一条提交，但 GitLab commits 接口默认按最新提交在前返回。这里会拿到最近一次修改 `CLAUDE.md` 的时间，而不是首次引入该文件的时间，导致 `initializedAt` 和后续趋势图累计统计整体偏晚、结果失真。
    修复建议与测试场景
    修复建议

    改为显式获取最早一条提交，而不是依赖默认排序。例如先按时间升序查询，或拉取多条后取最后一条：`.../repository/commits?path=...&order=default&per_page=100` 后取末尾，并在分页场景下继续翻页直到最后一页；如果 GitLab 版本支持，也可使用能返回最早提交的排序参数。至少需要补充单测覆盖“文件被多次修改时 initializedAt 仍应等于首次提交日期”。

    测试场景

    构造一个仓库：`CLAUDE.md` 在 2026-04-01 首次提交、2026-04-10 再次修改。执行状态同步后，断言 `initializedAt` 为 `2026-04-01`，而不是 `2026-04-10`。
