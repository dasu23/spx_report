    web/app/api/authors/route.ts1 Major1 个问题
    MAJORL29边界条件
    首次聚合作者时直接调用 `c.email.toLowerCase()`，当历史 commit 缺少 `email` 字段或该字段为 `undefined/null` 时，这个接口会抛出运行时异常，作者列表整体返回 500。上面 `key` 已经按 `c.email || c.author` 做了兜底，这里没有保持一致。
    修复建议与测试场景
    修复建议

    和 `key` 一样做兜底再写入，例如：`const email = (c.email || "").toLowerCase()`，或在类型层明确 `email` 必填并在读入数据时统一清洗。

    测试场景

    构造一条 `email` 为空或缺失的 commit 记录，请求 `GET /api/authors`，接口应返回该作者数据而不是 500。

    web/app/api/config/route.ts2 Major2 个问题
    MAJORL11API契约
    `GET /api/config` 把脱敏后的 token 仍放在 `gitlabToken` 字段返回。调用方如果按常规把该响应原样回填表单并再次 `PUT`，会把真实 token 覆盖成 `xxxxx***`，导致后续同步/校验全部失效。这个字段名同时承担“真实值”和“展示值”，契约含义冲突。
    修复建议与测试场景
    修复建议

    不要在 `gitlabToken` 字段返回脱敏值。可改为返回 `gitlabToken: ""` 并新增只读字段如 `gitlabTokenMasked` / `hasToken`，前端仅展示脱敏信息而不回传。

    测试场景

    先保存一份有效配置，再调用 `GET /api/config` 并将响应体原样提交到 `PUT /api/config`；修复后不应把存量 token 覆盖为脱敏字符串。

    MAJORL27状态管理
    在请求处理中直接写 `process.env.NODE_TLS_REJECT_UNAUTHORIZED = "0"` 会修改整个 Node 进程的全局 TLS 行为，而且不会恢复。一次配置校验后，后续所有 HTTPS 请求都会继续跳过证书校验，形成跨请求状态泄漏，结果取决于请求先后顺序。
    修复建议与测试场景
    修复建议

    不要改全局环境变量。把“是否允许自签证书”限制在单次请求范围内，例如使用自定义 `https.Agent`/dispatcher 传给当前 `fetch`，或提供显式配置项并在请求结束后不污染进程级状态。

    测试场景

    先调用一次 `PUT /api/config`，再在同一进程内发起一个本应进行证书校验的 HTTPS 请求；修复后第二个请求的 TLS 校验行为不应被前一次接口调用改变。

    web/app/sync/page.tsx1 Major1 个问题
    MAJORL92-116边界条件
    SSE 读取循环只处理了以换行结尾的 `data:` 行，`reader.read()` 返回 `done` 后会直接 `break`，没有再冲刷 `buffer` 中残留的数据。如果服务端最后一个 `done`/`error` 事件没有以 `\n` 结束，前端会丢掉最终事件，表现为同步结束后没有结果卡片、也没有错误提示。相同问题在 `handleClaudeMdSync` 中的 148-172 行重复出现。
    修复建议与测试场景
    修复建议

    在 `done` 为 `true` 时先补一次 `decoder.decode()` 的 flush，并对 `buffer` 中剩余内容再执行一次事件解析；或直接使用成熟的 SSE 解析器，避免手写分包逻辑遗漏尾包。

    测试场景

    构造一个流式响应：最后一个 `event: done` 后不追加空行/换行。点击同步后，页面仍应正确显示成功结果；同样覆盖 `event: error` 的尾包场景。

    web/components/dashboard/trend-chart.tsx1 Major1 个问题
    MAJORL45-52业务逻辑
    周维度截止日期通过本地时间构造 `Date` 后再用 `toISOString().slice(0, 10)` 转回字符串，会受运行环境时区影响。在 UTC+ 时区，这里得到的 `cutoff` 可能比实际周日早一天，导致周累计 `CLAUDE.md` 初始化数漏算周日初始化的项目。
    修复建议与测试场景
    修复建议

    避免用本地时区 `Date` 再转 ISO 字符串；改为按 UTC 计算周结束日，或直接使用纯日期算法输出 `YYYY-MM-DD`。示例：`const jan4 = new Date(Date.UTC(year, 0, 4))`，后续统一用 `getUTCDate/setUTCDate`，最后 `cutoff = sunday.toISOString().slice(0, 10)`。

    测试场景

    在非 UTC 时区（如 `Asia/Shanghai`）下，准备一个 `initializedAt` 恰好落在某周周日的项目，验证该周的 `claudeMdCumulative` 是否正确包含该项目。

    web/lib/store.ts2 Major2 个问题
    MAJORL164安全风险
    `writeConfig` 将 `gitlabToken` 连同其他配置明文写入 `data/config.json`。该 Token 具备 `read_api` 权限，一旦运行目录被误暴露、被备份外泄或被同机其他低权限进程读取，会直接导致 GitLab 凭证泄露。
    修复建议与测试场景
    修复建议

    不要将 GitLab Token 明文持久化到应用数据目录。优先改为从环境变量或专用密钥管理服务读取；若必须落盘，至少对 Token 单独加密存储，并限制文件权限为仅服务账户可读写。对外返回配置时也应始终脱敏该字段。

    测试场景

    配置一个测试 Token 后，检查 `data/config.json` 不应出现明文 `gitlabToken`；同时验证应用重启后仍可通过安全存储恢复配置，且任何配置查询接口返回值均不包含明文 Token。

    MAJORL58-64业务逻辑
    `mergeCommits` 在按分组覆盖旧数据时，只保留 `project.startsWith(group + "/")` 的记录。若同步的是嵌套分组或项目路径与 `group` 精确相等/格式不一致，旧记录不会被正确替换；更严重的是当 `group` 为空字符串时，会删除所有处于日期范围内的提交，导致多次按项目局部同步时覆盖掉同日期其他项目的数据。
    修复建议与测试场景
    修复建议

    覆盖逻辑应基于本次实际同步到的项目集合而不是仅靠 `group` 前缀。建议把本次同步的 `projectPaths` 传入 `mergeCommits`，仅删除这些项目在 `[since, until]` 范围内的旧记录。

    测试场景

    先保存 A、B 两个项目同一天的提交，再执行仅同步 A 项目的相同日期范围，验证 B 项目的历史提交不会被删除。
