    backend/watchtower-api/src/main/java/com/watchtower/api/client/WorkerInternalTriggerErrorDecoder.java1 Minor 1 Info2 个问题
    MINORL45-47异常处理
    读取 Feign 响应体时直接吞掉 IOException 并返回 null，会把“响应体读取失败”和“响应体本来为空”混为一类。后续 404/409 会退化为硬编码文案，其它状态会交给默认解码器，排障时无法区分是 worker 返回异常还是 body 读取链路本身出错。
    修复建议与测试场景
    修复建议

    至少记录并保留读取失败语义，避免静默降级。例如改为抛出包含 methodKey/status 的异常，或返回一个带 cause 的 BusinessException：`catch (IOException e) { throw new IllegalStateException("Failed to read worker error response", e); }`。如果必须兜底，也应补充日志并显式区分“empty body”和“read failed”。

    测试场景

    补充单测：构造 `response.body().asInputStream()` 抛 IOException 的场景，验证不会静默返回 null，且调用侧能拿到明确的失败原因。

    INFOL18-19配置变更
    这里直接 `new ObjectMapper()`，不会继承应用里统一注册的 Jackson 配置与模块。当前只解析 `code/message` 两个简单字段问题不大，但后续如果错误体序列化策略调整（例如命名策略、自定义模块、宽松解析配置），这个解码器会与全局行为产生分叉。
    修复建议与测试场景
    修复建议

    改为通过 Spring 注入复用应用级 `ObjectMapper`，例如在 Feign 配置中声明构造参数并传入现有 Bean，避免客户端侧维护一套独立 JSON 解析配置。
