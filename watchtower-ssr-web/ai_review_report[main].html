    在 CAS 抢占成功后立即返回异步执行，但 `asyncRunner.run(msg, service, record)` 的提交失败没有兜底处理。若 `manualRetryExecutor` 满队列、线程池关闭或 `@Async` 提交时抛出 `TaskRejectedException`，当前请求会在记录已被抢占后直接失败，任务状态可能停留在 RUNNING，后续无法再手动重试。

    修复建议

    MAJORL118-126业务逻辑
    `resend` 重发时没有使用报表记录里已持久化的 `report.getRecipients()`，而是重新读取当前配置收件人。这样一来，历史日报在配置变更后会被发送给不同对象；更严重的是当前配置被清空时，即使该报表原本保存过收件人，也会被标记为失败，和“重发既有日报”的语义不一致。
    修复建议与测试场景
    修复建议

    优先使用 `report` 上已保存的收件人作为重发目标，仅在历史数据为空时再回退到当前配置。例如：`List<String> recipients = report.getRecipients(); if (recipients == null || recipients.isEmpty()) { recipients = configStore.get().getRecipients(); }`。

    测试场景

    补充测试：先生成一条带收件人的日报，再模拟配置收件人变更/清空后调用 `resend`，断言仍按报表自身保存的收件人重发。

    MAJORL72边界条件
    `generate` 直接执行 `new ArrayList<>(recipients)`，当调用方传入 `null` 时会抛出 `NullPointerException`。这会让日报生成流程在边界配置下直接失败，而不是返回可定位的业务错误或按空收件人处理。
    修复建议与测试场景
    修复建议

    在复制前做空值校验并给出明确分支，例如：`if (recipients == null || recipients.isEmpty()) { throw new BusinessException(400, "收件人配置为空"); } report.setRecipients(new ArrayList<>(recipients));`。

    测试场景

    补充单测：`generate(..., null)` 与 `generate(..., Collections.emptyList())`，验证返回明确业务异常而非 NPE。
