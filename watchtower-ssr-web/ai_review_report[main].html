    MINORL26-27API契约
    Feign 接口明知返回结构是固定的 `PipelineResult` 字段集合，却声明为 `Map<String, Object>`。这会把响应契约退化为运行时约定：字段缺失、类型漂移（例如计数字段在不同实现下反序列化为 `Integer`/`Long`）都无法在编译期暴露，后续调用方也更容易出现强转或 key 拼写错误。
    修复建议与测试场景
    修复建议

    为该接口定义显式 DTO，并直接作为 Feign 返回类型，例如：`record SlowQueryTriggerResponse(int fetchedCount, int skippedCount, int newIssuesCount, int reopenedCount) {}`，然后将方法改为 `SlowQueryTriggerResponse trigger(@PathVariable("id") Long id);`。这样可以让响应结构变更在编译期暴露，并减少调用方的动态取值逻辑。

    测试场景

    增加反序列化契约测试：1）worker 正常返回完整字段时可成功绑定 DTO；2）缺少必填字段或字段类型变化时，客户端测试应直接失败，防止接口结构漂移被静默吞掉。

    改派场景把请求体定义为 `{fromStatus:"ASSIGNED", toStatus:"ASSIGNED", ...}`，但同一份 spec 前面的普通流转场景仍使用 `{status:"ASSIGNED", ...}`。这会把同一个 `PUT /api/slow-query-issues/{id}` 接口描述成两套互斥的请求契约，前后端很容易各自按不同字段实现，最终导致改派请求无法通过 DTO 反序列化或服务端状态校验。
    修复建议与测试场景
    修复建议

    统一 update 接口的请求模型，只保留一种状态表达方式，并在所有 scenario、proposal、tasks 中同步修正。若决定沿用 `fromStatus/toStatus`，则把普通流转示例也改为同一结构；若沿用单字段 `status`，则改派场景需明确当前状态从服务端读取而不是额外传入。
