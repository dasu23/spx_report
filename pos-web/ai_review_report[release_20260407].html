    src/pages/dms/route/delivery/components/SplitRoute/FormSubRouteList.tsx1 Major1 个问题
    MAJORL66-94边界条件
    新的校验只检查“是否有 range”和“是否重叠”，没有检查所选 range 是否实际覆盖到任何 package。当前选择器允许在 `minSequence` 到 `maxSequence` 之间任选数字；一旦路线 sequence 存在空洞，用户可以选到一个合法但没有任何包裹的区间，最终 `collectTrackingNos()` 会产出空数组并把空子路线提交给拆分接口。
    修复建议与测试场景
    修复建议

    在表单校验阶段补充“至少命中 1 个 package”的判断。可直接复用当前 `getPackageCount(validRanges)`：当某个 sub route 的总数为 0 时 `Promise.reject('Selected ranges contain no packages')`。同时在提交前再兜底一次，过滤或拒绝 `trackingNos.length === 0` 的子路线。

    测试场景

    构造 sequence 不连续的路线数据（如只有 1、5、9），在弹窗中选择 2-4 作为一个 sub route，确认表单应阻止提交并提示该范围没有包裹。

    src/pages/dms/route/delivery/components/SplitRoute/ModalSubRoute.tsx1 Major1 个问题
    MAJORL50-57状态管理
    这里把 `ModalForm` 的提交结果直接透传为 `onFinish?.(values)`。当父组件没有传 `onFinish`，或者父组件的 `onFinish` 只是做副作用并返回 `void/undefined` 时，本次拆分虽然已经成功调用接口并弹出成功提示，但 `ModalForm` 会拿到 falsy 值而不自动关闭，形成“操作成功但弹窗留在原处”的异常交互。
    修复建议与测试场景
    修复建议

    将关闭弹窗的返回值与父回调解耦，默认在拆分成功后返回 `true`。例如：`const parentResult = await onFinish?.(values); return parentResult ?? true;`；如果希望父组件能阻止关闭，则显式约定仅 `false` 阻止，其余情况都返回 `true`。

    测试场景

    打开 Split Route 弹窗，传入一个不返回值的 `onFinish` 或不传 `onFinish`，完成一次成功拆分后，验证当前弹窗会自动关闭。
