    config/proxy.ts2 Major2 个问题
    MAJORL101-105安全风险
    `test` 环境下同样新增了基于路径动态决定目标主机的 `/cdn-pod/` 代理，问题与开发环境一致：`host` 完全由 URL 路径控制，未做白名单或安全过滤。测试环境通常可被更多人员共享使用，这会扩大被滥用为开放代理的攻击面。
    修复建议与测试场景
    修复建议

    与开发环境保持一致的安全控制：仅允许受信任 CDN 域名白名单，禁止任意主机转发；必要时直接移除此动态代理，改为后端签名下载或固定 CDN 反向代理。

    测试场景

    在测试环境分别请求 `/cdn-pod/metadata.google.internal/x`、`/cdn-pod/10.0.0.1/x`、`/cdn-pod/合法CDN域名/x`，确认非白名单目标全部被阻断。

    MAJORL89-92配置变更
    新增的 `/cdn-pod/` 代理会直接从请求路径首段提取 host 并拼成 target，没有任何白名单或格式校验。这样前端只要构造 `/cdn-pod/<任意host>/...` 就会把开发/测试环境代理到任意外部站点，容易导致调试环境请求行为失控，也会让图片加载问题变成难以排查的跨域/网络问题。
    修复建议与测试场景
    修复建议

    至少对允许代理的 host 做白名单校验，非法 host 直接拒绝；同时校验 host 仅包含合法域名字符。例如：`const allowedHosts = new Set(['a.example.com','b.example.com']); const host = ...; if (!allowedHosts.has(host)) return 'https://placeholder.invalid/';`。如果 host 来源确实动态，建议改为由后端下发受控映射，而不是直接信任 URL 路径。

    测试场景

    访问 `/cdn-pod/example.com/a.jpg` 与 `/cdn-pod/evil.test/a.jpg`，确认前者正常代理、后者被拒绝或回退到无效 target，而不是向任意域名发起请求。
