src/components/PrintContainerLabel/ModalPrintContainerLabel.tsx

SeverityLineCategoryDescriptionSuggestionTest Scenario
major135边界条件`seq` 允许最小值 `0`，而提交时会映射为 `startNumber=0,endNumber=0`。这与列表页“`endNumber=0` 禁止打印”的约束冲突，可能触发无效打印请求。将 `seq` 最小值改为 `1`，并在 `onFinish` 增加校验（例如 `seq < 1` 直接阻断并提示）。输入 `seq=0` 提交，预期前端阻止提交并显示错误提示，不发送打印请求。
major43-45API契约`getSortDetail` 返回结构为 `routeDate`/`seqNo`，但这里直接 `setFieldsValue(payload)`；表单字段实际是 `sortDate`/`seq`。导致查询后日期与序号无法正确回填，单条打印会带着错误或空值提交。在回填前做字段映射：`sortDate: payload.routeDate`、`seq: payload.seqNo`，并仅写入表单实际存在的字段。mock `getSortDetail` 返回 `routeDate=2026-02-23, seqNo=3`，点击查询后断言 `sortDate` 日期控件和 `seq` 输入框都已正确显示对应值。
major69-76API契约`onFinish` 直接提交 `values`，其中 `ProFormDatePicker` 的 `sortDate` 在运行时通常是 `Dayjs` 对象而非字符串，POST JSON 时会产生不符合后端契约的日期格式。提交前显式序列化：`sortDate: values.sortDate ? dayjs(values.sortDate).format('YYYY-MM-DD') : undefined`，再传给 `downloadFile`。提交表单后拦截 `/container-label/batch-print` 请求，断言 `sortDate` 为 `YYYY-MM-DD` 字符串，而不是对象。
