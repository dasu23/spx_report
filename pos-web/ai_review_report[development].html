src/pages/user/sso/index.tsx

SeverityLineCategoryDescriptionSuggestionTest Scenario
critical7-19安全风险SSO 回调直接从 URL 查询串读取 `authToken` 并写入 `localStorage`。这意味着 Bearer Token 会以明文出现在浏览器地址栏、历史记录、代理/CDN/服务端访问日志，以及可能的 `Referer` 链路里；而且当前实现没有校验 `state/nonce`，任意带 `?authToken=...` 的链接都可以把用户会话切换到该 token，对应登录 CSRF / token fixation 风险。不要让前端接收明文 token 查询参数。改为后端只回传一次性 `code`，前端通过 `POST` 到后端换取 token；至少要在发起登录前生成随机 `state` 存到 `sessionStorage`，回调时严格校验后再继续。例如：`const expectedState = sessionStorage.getItem('sso_state'); if (urlParams.get('state') !== expectedState) { setAuthError('Invalid SSO state'); return; }`。更稳妥的做法是把 token 写入 HttpOnly Cookie，前端只做成功跳转。1. 直接访问 `/user/sso?authToken=fake-token`，确认页面不会写入 token。2. 构造 `state` 不匹配的回调 URL，确认被拒绝。3. 检查 SSO 成功链路中地址栏、网络日志、浏览器历史里不再出现 Bearer Token。

src/services/http.ts

SeverityLineCategoryDescriptionSuggestionTest Scenario
major123-132API契约全局响应适配把 `res.data` 从 `res.payload` 改成了 `res.payload || res.data`。当后端合法返回 `0`、`false`、空字符串这类 falsy payload 时，这里会错误地回退成原始响应对象，导致调用方拿到的类型从预期的原始值变成整个响应体，属于全局接口契约回归。这里应只在 `payload` 为 `null/undefined` 时回退，改为 nullish 合并：`res.data = res.payload ?? res.data;`。同时补充对 `0`、`false`、`''` 三类 payload 的单测，避免再次回归。为 `request` 中间件增加用例：后端分别返回 `{ success: true, payload: 0 }`、`false`、`''`，断言调用方读取到的 `res.data` 仍然是 `0/false/''`，而不是整个响应对象。
