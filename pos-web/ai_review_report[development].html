src/pages/user/sso/index.tsx

SeverityLineCategoryDescriptionSuggestionTest Scenario
major10-19安全风险SSO 回调页直接信任 URL 中的 `authToken` 并写入 `localStorage`，没有校验任何 `state`/nonce，也没有校验这次回调是否由当前登录流程发起。攻击者只要构造 `/user/sso?authToken=<attacker-token>` 链接，就可能让受害者浏览器写入攻击者会话，形成登录会话固定/账号串用风险。在发起 Microsoft 登录前生成随机 `state` 并保存到 `sessionStorage`，由后端把该 `state` 带入 OAuth；回调页在写入 token 前必须校验返回的 `state` 与本地一致，不一致时拒绝登录并清理 URL。同时把账号类型等上下文也绑定到该 state，而不是仅依赖可变的 `localStorage`。补充用例：1) 直接访问 `/user/sso?authToken=fake&state=bad` 应显示错误且不得写入 token；2) `state` 缺失或不匹配时不得跳转首页；3) 只有合法 state 才能完成登录。

src/services/http.ts

SeverityLineCategoryDescriptionSuggestionTest Scenario
minor132API契约这里把响应归一化从 `res.payload` 改成了 `res.payload || res.data`。如果后端合法返回的 `payload` 是 `0`、`false` 或空字符串，这些值会被当成 falsy 丢弃，调用方拿到旧的 `res.data`/`undefined`，产生隐蔽的契约回归。改为使用空值合并：`res.data = res.payload ?? res.data;`，只在 `payload` 为 `null/undefined` 时才回退。补充请求层单测，覆盖 `payload` 分别为 `0`、`false`、`''`、`null`、`undefined` 时 `res.data` 的归一化结果。
