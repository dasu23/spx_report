    config/proxy.ts1 Critical1 个问题
    🔒 安全/权限相关 · 1 项（点击展开）
    CRITICALL87-105安全风险
    新增 `/cdn-pod/` 代理通过 `router` 直接从请求路径首段提取 `host` 并拼接为 `https://${host}/`，未做任何白名单、格式校验或目标限制。这会把本地开发/测试前端代理变成一个可由路径参数控制的开放转发器，攻击者只需构造 `/cdn-pod/<任意主机>/<path>` 即可借助受信任前端环境向任意外部或内网地址发起请求，形成 SSRF/内网探测风险；若浏览器会自动携带当前站点凭据到该代理入口，还可能被用于绕过前端同源限制访问非预期资源。
    修复建议与测试场景
    修复建议

    不要直接信任路径中的 host。至少应对可代理的 POD 域名做严格白名单校验，并拒绝 IP、localhost、内网网段及非预期后缀。例如： ```ts const ALLOWED_HOSTS = new Set([ 'pod1.example.com', 'pod2.example.com', ]); router: (req) => { const host = req.path.replace(/^\/cdn-pod\//, '').split('/')[0]; if (!ALLOWED_HOSTS.has(host)) { throw new Error('Invalid proxy host'); } return `https://${host}/`; } ``` 若确需动态域名，至少校验为受控后缀并显式拦截 `localhost`、私网 IP、保留地址。

    测试场景

    访问 `/cdn-pod/127.0.0.1/admin`、`/cdn-pod/169.254.169.254/latest/meta-data`、`/cdn-pod/internal.service.local/api`，应全部被拒绝；访问白名单 POD 域名时才允许转发。

    MAJORL24-27安全风险
    异常图片上传接口同样从 `localStorage` 读取 `SPX_AUTH_TOKEN` 并手工拼接到 `Authorization` 头，重复引入了前端可读 Token 的敏感数据暴露风险。该接口还处理图片上传，一旦凭证被窃取，攻击者可冒充用户提交异常图片/报告。

    修复建议

    不要在业务代码中直接从 `localStorage` 取 Token。复用统一的安全鉴权机制（如 `HttpOnly` Cookie 或内存态 token 注入器），并将上传请求也纳入同一凭证管理链路，避免多处散落读取敏感凭证。

    测试场景

    检查上传异常图片前后，浏览器脚本是否可通过 `localStorage.getItem(SPX_AUTH_TOKEN)` 读取凭证；修复后应不可读，同时上传接口仍能成功鉴权。

    MAJORL29-32安全风险
    PDA 专用请求实例把认证凭证直接从 `localStorage` 读取并注入 `Authorization` 头。`localStorage` 中的 Token 对任意同源脚本可读，一旦页面存在 XSS 或第三方脚本被污染，攻击者即可直接窃取长期有效凭证并调用 PDA 装卸车/封签相关接口。这属于新增的敏感凭证暴露面。
    修复建议与测试场景
    修复建议

    避免将认证 Token 持久化在 `localStorage`。优先改为服务端下发 `HttpOnly + Secure + SameSite` Cookie；若受现有架构限制，至少改为仅保存在内存态，并配合严格 CSP/最小化第三方脚本。示例：移除 `localStorage.getItem(...)` 读取，改由统一鉴权层从受保护的 cookie/session 自动携带。

    测试场景

    在浏览器控制台模拟同源脚本执行 `localStorage.getItem(SPX_AUTH_TOKEN)`，确认当前实现可直接读出凭证；修复后应无法通过 JS 读取认证凭证，同时 PDA 请求仍能正常鉴权。
