major11-18安全风险SSO 回调页会直接读取 URL 中的 `authToken` 并写入本地存储，没有校验任何 `state`/nonce/来源信息。任何拿到自己有效 token 的人都可以构造 `/user/sso?authToken=...` 链接，让受害者浏览器切换到攻击者账号，属于典型登录 CSRF / token 注入风险。不要直接消费 URL 里的成品 token。改为在回调里校验登录发起时保存的随机 `state`，或只接收一次性 `code` 后再向后端换取 token；至少要在前端校验本次发起登录时写入的随机值后再落库。先在浏览器 A 完成 Microsoft 登录拿到一个有效 `authToken`，然后在未登录浏览器 B 直接访问 `/user/sso?authToken=<A的token>`，确认当前实现会直接登录；修复后应拒绝该请求并回到登录页。
