    src/hooks/useDownloadFile.ts1 Major1 个问题
    MAJORL102-108API契约
    这里把所有 `success !== false` 的 JSON 响应都当作成功处理了。该 hook 是通用下载入口，若后端返回的是旧格式错误响应（例如只有 `code/message`，没有 `success` 字段），弹窗会被错误地关闭，真实失败被吞掉。
    修复建议与测试场景
    修复建议

    只对显式成功响应放行，例如改为 `if (data?.success === true) { ...resolve(true) } else { showError(data?.message || 'Download failed'); reject(new Error(...)); }`。如果确实要兼容“双响应模式”接口，建议增加显式开关参数，而不是修改通用默认语义。

    测试场景

    新增用例：1) JSON `{ success: true }` 应 resolve；2) JSON `{ success: false, message: 'x' }` 应 reject；3) JSON `{ code: 500, message: 'x' }` 且无 `success` 时也应 reject。

    src/pages/user/sso/index.tsx1 Critical1 个问题
    CRITICALL11-18安全风险
    SSO 回调从 URL query 中读取 `authToken`，并将其作为长期凭证写入 `localStorage`。这同时引入两类高风险问题：一是 Token 出现在查询串中，容易经浏览器历史、代理日志、Referer 等链路泄露；二是 `localStorage` 中的 Token 可被任意同源脚本读取，XSS 一旦发生将直接导致账户凭证失窃。
    修复建议与测试场景
    修复建议

    不要通过 query string 传递登录凭证，改为后端回调时使用一次性 code，再由前端/后端安全换取会话；前端不要将认证 Token 持久化到 `localStorage`，优先改为服务端 `HttpOnly`、`Secure`、`SameSite` Cookie 承载会话。若短期无法调整协议，至少改为短生命周期内存态存储并避免 URL 载荷携带 Token。

    测试场景

    验证 Microsoft SSO 完成后，地址栏与网络链路中不再出现真实 Token；检查浏览器 `localStorage/sessionStorage` 不存在认证 Token，且页面刷新、跨标签页、发生 XSS 注入测试时无法直接读取会话凭证。
