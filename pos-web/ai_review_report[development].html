    config/routes.ts1 Major1 个问题
    MAJORL283-341权限与访问控制
    **原因：** 新增隐藏路由 `dispatch-mm-edit` 未配置 `access` 权限控制，而其同级入口页 `dispatch-mm` 已显式要求 `dispatchMM:list:query`。这意味着用户即使没有菜单权限，只要知道前端路径，仍可能直接访问编辑页组件。 ```ts { path: 'dispatch-mm-edit', component: './dispatch/dispatch-mm-edit', hideInMenu: true, }, ``` **影响：** 前端路由层面对编辑页面形成直接访问缺口，低权限用户可通过手工输入 URL 进入敏感操作页，扩大功能暴露面；若页面内接口校验不严，还会进一步演变为越权查看/修改 dispatch 数据。 --- 新增的 `dispatch-mm-edit` 路由没有配置 `access`，但同组列表页 `dispatch-mm` 已使用 `dispatchMM:list:query` 做权限控制。当前实现下，用户即使没有该模块权限，只要知道 URL 就可以直接访问编辑页，导致菜单权限与实际页面访问权限不一致。 --- **原因：** 新增扫码页通过 URL 查询参数中的 `dispatchCode` 直接拼接跳转到提交页面，且该按钮未经过任何 `Access/AccessButton` 权限门禁，也未对 `dispatchCode` 与当前 BOL/用户数据范围关系做前端校验。 ```tsx <Button key="submit" type="primary" disabled={!dispatchCode} onClick={() => { history.push(`/dispatch/dispatch-mm-edit?mode=submit&dispatchCode=${dispatchCode}`); }} > ``` **影响：** 只要用户能访问该扫码页并构造带 `dispatchCode` 的 URL，就能进入提交流程入口，形成前端层面的认证/授权绕过面。若目标编辑页或后端接口校验不足，可能进一步导致未授权提交他人 dispatch。即使后端最终拦截，也会暴露不应开放的业务入口。
    修复建议与测试场景
    修复建议

    为 `dispatch-mm-edit` 补充与父功能一致或更严格的 `access` 权限码，并确保页面内关键操作再次校验。例如： ```ts { path: 'dispatch-mm-edit', component: './dispatch/dispatch-mm-edit', hideInMenu: true, access: 'dispatchMM:list:query', } ``` 若编辑权限应独立控制，应新增专用 rightCode（如 `dispatchMM:edit`）并在路由与页面动作上统一使用。
