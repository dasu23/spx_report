    src/pages/dashboard/executive-dashboard/components/ExecutiveKPITable/hooks/useExecutiveKPIFilters.ts1 Major1 个问题
    MAJORL135-168状态管理
    筛选器初始化被拆成两个 `useEffect` 后，空选项分支直接 `return`，不再清空旧状态。切换到一个没有 KPI/Level/SC 数据的日期或数据集时，`filters` 会保留上一次的选择，导致筛选 UI、导出参数和表格过滤继续使用陈旧值，行为与旧实现不一致。
    修复建议与测试场景
    修复建议

    在两个 effect 的空数据分支里显式重置对应筛选项，而不是直接返回。例如当 `legOptions/levelOptions/kpiOptions` 都为空时重置这三项；当 `serviceCenterOptions` 为空时将 `serviceCenters` 置为空数组。

    测试场景

    先进入一个有完整筛选项的日期并选择部分 Hub/SC、KPI，再切换到一个返回空筛选项的日期，验证筛选器 UI 和 `filteredRowData` 不会继续保留上一次的值。

    src/pages/dispatch/dispatch-search/index.tsx2 Major2 个问题
    MAJORL80-82安全风险
    新增逻辑将 `destination`、`carrierName`、`plateNo` 直接拼入 URL 查询串。该类业务数据会进入浏览器历史、地址栏、日志、埋点以及后续页面跳转时的 Referer，形成不必要的数据暴露面；同时未做 `encodeURIComponent`，字段中若包含 `&`、`#` 等字符还会污染查询参数。
    修复建议与测试场景
    修复建议

    不要通过查询串传递这类详情字段。改为仅传 `dispatchCode`，在目标页按编号重新拉取详情，或使用路由 state / 内存态传递临时数据；若必须放入 URL，至少对每个值执行 `encodeURIComponent` 并评估是否允许暴露。

    测试场景

    构造 `carrierName=ACME&foo=bar`、`plateNo=#test` 的记录，点击 Scan，验证目标页 URL 是否被注入额外参数/截断；同时检查浏览器历史和网络请求 Referer 中是否出现 `destination`、`carrierName`、`plateNo`。

    MAJORL82-84边界条件
    这里把 `destination`、`carrierName`、`plateNo` 直接拼进 query string，没有做 URL 编码。只要值里包含 `&`、`#`、`=`、空格等字符，跳转后的 `/dispatch/dispatch-scan` 就会解析出错，展示的详情字段会被截断或错位。
    修复建议与测试场景
    修复建议

    改用 `history.push({ pathname, search: new URLSearchParams({...}).toString() })`，或对每个值使用 `encodeURIComponent`。

    测试场景

    构造 `carrierName='A&B Logistics'`、`destination='JFK #1'` 的记录，点击 Scan 后检查目标页读取到的参数是否完整。

    src/pages/user/sso/index.tsx1 Critical1 个问题
    CRITICALL18安全风险
    SSO 回调页将 `authToken` 持久化到 `localStorage`。这会把认证凭证暴露给任意同源脚本，一旦页面存在 XSS 或第三方脚本被污染，令牌可被直接读取并长期复用，属于敏感数据暴露。
    修复建议与测试场景
    修复建议

    不要把登录 Token 存入 `localStorage`。改为由后端在 SSO 回调完成后下发 `HttpOnly`、`Secure`、`SameSite` Cookie，前端仅依赖浏览器自动携带凭证；若现阶段必须前端接收令牌，也应仅做短暂内存态传递并尽快与后端交换成安全 Cookie。

    测试场景

    增加安全回归用例：SSO 成功后，断言 `localStorage/sessionStorage` 中不存在认证 Token；同时验证回调完成后凭证仅通过安全 Cookie 生效。

    tailwind.config.js1 Minor1 个问题
    MINORL38配置变更
    `content` 仅包含 `src/pages/**/*.tsx`、`src/components/**/*.tsx`、`src/layouts/**/*.tsx`，会忽略其他目录中的 Tailwind 类名扫描。项目中若在 `src/app.tsx`、`src/global.tsx`、`src/hooks` 或其他 TS/TSX 文件里使用 Tailwind utility，生产构建时这些样式会被裁剪，导致页面样式回归。
    修复建议与测试场景
    修复建议

    将扫描范围扩展到整个 `src` 下的相关源码文件，例如：`content: ['./src/**/*.{ts,tsx}']`；如果确实要限制范围，也应显式补上 `src/app.tsx`、`src/global.tsx` 等入口文件。

    测试场景

    在非 `pages/components/layouts` 目录的 TSX 文件中放置一个 Tailwind class，执行生产构建并验证对应样式是否仍然存在。
