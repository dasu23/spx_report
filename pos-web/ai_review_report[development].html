major10-19安全风险SSO 回调直接信任 URL 中的 `authToken` 并写入 `localStorage`，没有校验 OAuth `state`/nonce，也没有校验这次回调是否由当前登录流程发起。攻击者可以构造 `/user/sso?authToken=<attacker-token>` 链接诱导用户访问，造成登录会话固定，用户后续操作会落到错误账号上。发起 Microsoft 登录前生成一次性 `state`（建议 `crypto.randomUUID()`），存入 `sessionStorage`，并随登录请求一起传给后端；回调页读取并严格比对 `state`，不匹配时拒绝写入 token 并提示错误。例如：`sessionStorage.setItem('sso_state', state)`；回调时 `if (urlParams.get('state') !== sessionStorage.getItem('sso_state')) { setAuthError('Invalid SSO state'); return; }`。校验通过后再落 token，并立即清理 `state`。先正常打开登录页但不要发起 SSO，再手工访问 `/user/sso?authToken=fake-token&state=fake`；修复前页面会直接登录，修复后应阻止登录并展示 state 校验失败。
