    src/hooks/useDownloadFile.ts1 Major1 个问题
    MAJORL102-108业务逻辑
    这里把所有 `response.ok` 且 `data.success !== false` 的 JSON 都当成成功处理并直接 `resolve(true)`。该 hook 是通用下载/上传能力，若某些接口沿用 `200 + { code/message }` 的业务失败格式但不返回 `success:false`，调用方会被误判为成功，弹窗关闭且错误信息丢失。
    修复建议与测试场景
    修复建议

    只在明确成功时才走成功分支，例如同时判断 `data.success === true`，或兼容项目统一响应结构后再 fallback：```ts const data = await response.json(); if (data.success === true) { setDownloading(false); message.destroy(); resolve(true); return; } showError(data.message || 'Download failed'); reject(new Error('Download failed')); ```

    测试场景

    补充一个 `response.ok === true` 且返回 `{ code: 500, message: 'Import failed' }`、不含 `success` 字段的用例，验证 hook 不会误判成功。

    src/pages/user/sso/index.tsx1 Critical 1 Major2 个问题
    CRITICALL18安全风险
    新增代码将 `authToken` 持久化到 `localStorage`。`localStorage` 中的数据可被任意同源脚本长期读取，一旦页面存在 XSS 或第三方脚本被污染，登录令牌会直接泄露，属于高风险敏感数据暴露。
    修复建议与测试场景
    修复建议

    不要在 `localStorage` 中保存认证令牌。改为由后端通过 `HttpOnly`、`Secure`、`SameSite` Cookie 下发和维护会话；如果必须在前端暂存，至少改用内存态/短生命周期方案，并配套令牌轮换。

    测试场景

    在 SSO 回调成功后，检查浏览器 `localStorage/sessionStorage`、可读 Cookie 和页面脚本上下文，确认认证令牌不可被前端 JS 直接读取；再做一次 XSS 模拟，验证脚本无法窃取会话凭证。

    MAJORL16-18业务逻辑
    拿到 `authToken` 后直接把当前回调页跳转到 `/`，没有把登录结果回传给发起登录的页面，也没有关闭当前 SSO 窗口。这样一旦 Microsoft 登录是按独立窗口/标签页打开，原登录页无法感知登录成功，用户会被留在多余的回调页里，和本次变更内 `sso.md` 描述的交互流程不一致。
    修复建议与测试场景
    修复建议

    在回调页中把 token 通过 `window.opener.postMessage`、`BroadcastChannel` 或 `localStorage` 事件通知给登录页，然后调用 `window.close()`；登录页收到通知后再更新登录态并跳转首页。至少应避免在回调页内硬编码 `window.location.href = '/'`。

    测试场景

    模拟 `window.open` 打开的 SSO 回调页：回调页拿到 `authToken` 后应通知 opener、关闭自身，并由登录页完成首页跳转；同时校验原登录页能收到正确的 account/admin 类型上下文。
