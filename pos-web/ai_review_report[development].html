    src/hooks/useAbortableProRequest.ts1 Major1 个问题
    MAJORL34-54状态管理
    `then`/`catch` 回调通过共享的 `cancelRef.current` 判断当前请求是否仍然有效，但这个 ref 会被后一次请求覆盖。若前一次请求未真正被底层中断、并在后一请求发起后才返回，前一次回调会误把“后一请求仍在进行中”当成自己的有效标记，进而错误地 `resolve` 旧结果并清空当前请求的 `abortRef/cancelRef`，导致表单显示过期数据或把当前请求状态提前结束。
    修复建议与测试场景
    修复建议

    为每次调用保存独立的 controller / cancel 标识，并在回调中比较“当前 ref 是否仍指向本次请求”后再更新状态，例如用局部 `const currentController = controller`，回调里先判断 `abortRef.current === currentController` 再执行 `resolve/reject` 与清理。也可以引入递增 `requestId`，仅允许最新一次请求提交结果。

    测试场景

    连续快速打开/关闭弹窗并触发两次 `request`：让第一次请求忽略 `AbortSignal` 且延迟返回，第二次请求更晚发起但更早完成，验证页面不会被第一次的旧结果覆盖，第二次 loading 状态也不会被提前清除。

    src/pages/dispatch/dispatch-search/index.tsx1 Major 1 Minor2 个问题
    MAJORL83-85安全风险
    新增跳转将 `destination`、`carrierName`、`plateNo` 直接拼接进 URL 查询串。`plateNo` 等业务标识会进入浏览器历史、日志、书签及潜在的 Referer，形成前端侧敏感信息暴露面；同时未做 `encodeURIComponent`，当字段包含 `&`、`=` 等字符时还会造成查询参数串改。
    修复建议与测试场景
    修复建议

    不要通过查询串传递这类业务字段。优先只传 `dispatchCode`，在目标页重新拉取详情；如必须跨页携带，改用路由 state 或受控的临时缓存，并对所有 URL 参数做 `encodeURIComponent`。

    测试场景

    构造 `carrierName=a&x=1`、`plateNo=ABC 123` 的记录并触发跳转，验证目标 URL 不再直接暴露这些字段，且不会出现参数截断/串改；同时检查浏览器历史与复制链接内容不含车牌等业务信息。

    MINORL82-84边界条件
    新增跳转直接用字符串拼接 query 参数，`destination`、`carrierName`、`plateNo` 中如果包含空格、`&`、`#`、`?` 等字符，会把 URL 截断或拆成错误参数，导致扫码页展示信息错误。
    修复建议与测试场景
    修复建议

    改用 `URLSearchParams` 或 `encodeURIComponent` 构造查询串，例如：`const qs = new URLSearchParams({ dispatchCode: record.dispatchCode ?? '', destination: record.destination ?? '', carrierName: record.carrierName ?? '', plateNo: record.plateNo ?? '' }); history.push(`/dispatch/dispatch-scan?${qs.toString()}`);`。

    测试场景

    构造 `destination='A&B Hub'`、`carrierName='ACME Truck #1'` 的记录，点击跳转后校验扫码页能完整拿到 4 个查询参数且内容未被截断。

    src/pages/user/sso/index.tsx1 Critical1 个问题

    在浏览器控制台或注入脚本环境中执行 `localStorage.getItem('SPX_AUTH_TOKEN')`，验证当前实现可直接读出令牌；修复后应无法通过前端脚本读取认证凭证。

    MAJORL35-39配置变更

    测试场景
