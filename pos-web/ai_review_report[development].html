major10-19安全风险回调页直接信任 URL 中的 `authToken` 并写入本地存储，没有校验任何 `state`/nonce 或请求关联信息。用户只要访问一个伪造的 `/user/sso?authToken=...` 链接，就可能被强制绑定到攻击者提供的会话，属于典型的登录会话固定风险。发起 SSO 前生成随机 `state` 并持久化，回调时先校验 `state` 再接受 token；更稳妥的是只接收一次性 `code`，再由后端换取 token。例如：`const expectedState = sessionStorage.getItem('sso_state'); if (urlParams.get('state') !== expectedState) { localStorage.removeItem(SPX_AUTH_TOKEN); setAuthError('Invalid SSO state'); return; }`。构造一个不经过登录发起流程的 `/user/sso?authToken=fake-token&state=missing` 访问，确认页面拒绝写入 token，并显示认证失败。
major17-21状态管理当回调缺少 `authToken` 或带有 `error` 参数时，当前逻辑只设置错误信息，没有清理已有的 `SPX_AUTH_TOKEN`。如果浏览器中残留旧 token，用户在 SSO 失败后仍可能继续以旧会话访问系统，导致登录状态与页面提示不一致。在处理回调前先清理旧 token，或至少在错误分支显式移除：`localStorage.removeItem(SPX_AUTH_TOKEN);`，再设置错误态。这样失败后的认证状态才是确定的。先在 `localStorage` 预置一个旧 token，再访问 `/user/sso?error=access_denied`，确认页面报错且旧 token 已被清除。
