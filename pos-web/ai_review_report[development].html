    config/routes.ts1 Major1 个问题
    MAJORL283-1029权限与访问控制
    **原因：** 父路由 `/gate-control` 配置了 `access: 'gateControl'`，但其唯一子路由 `driver-entry-scan` 使用的是不同的子权限 `gateControl:driverEntry:scan`。若权限体系允许用户仅拥有子权限而无父权限，则父级菜单会被隐藏，用户仍可直接访问子路由；反之若仅有父权限无子权限，也可能出现菜单组可见但子页访问控制不一致。该变更未展示任何统一的父子权限联动保证。 ```ts { path: '/gate-control', access: 'gateControl', routes: [ { path: 'driver-entry-scan', access: 'gateControl:driverEntry:scan', component: './gate-control/driver-entry-scan', }, ], } ``` **影响：** 会造成权限模型不一致，存在通过直链访问子页面或错误暴露菜单结构的风险；在门禁扫描这类现场操作页面中，错误授权可能导致未授权人员进入扫描流程。
    修复建议与测试场景
    修复建议

    确认 `access.ts`/权限树是否保证父子权限一致；若没有，需统一策略：要么子路由同时要求父权限与子权限，要么确保拥有子权限时父权限必然存在。前端路由可改为显式组合校验，或在权限配置中建立包含关系。

    测试场景

    分别使用仅有 `gateControl`、仅有 `gateControl:driverEntry:scan`、同时拥有两者的账号访问 `/gate-control/driver-entry-scan`，验证菜单可见性与页面访问结果一致且符合预期。

    src/pages/Home/index.tsx1 Critical1 个问题
    CRITICALL20-26安全风险
    新增逻辑从 URL 查询串读取 `authToken` 并持久化到 `localStorage`：`localStorage.setItem(SPX_AUTH_TOKEN, authToken)`。这会把认证凭证暴露给任意同源脚本，一旦站内存在 XSS、第三方脚本污染或浏览器扩展窃取，即可直接读取长期有效 Token，属于敏感凭证落地到 Web Storage 的高风险问题；同时该实现也命中了自定义规则 [SEC-002]（硬编码/设置 token）。
    修复建议与测试场景
