    src/pages/user/sso/index.tsx1 Critical 1 Major2 个问题
    CRITICALL10-17安全风险
    新增 SSO 回调页将 `authToken` 持久化到 `localStorage`。一旦页面存在 XSS、第三方脚本被注入，或浏览器扩展被滥用，长期有效的认证令牌可被直接读取并复用，属于敏感凭证暴露。
    修复建议与测试场景
    修复建议

    不要把认证 Token 存入 `localStorage`。改为由后端在 SSO 回调时下发 `HttpOnly`、`Secure`、`SameSite` Cookie，前端仅依赖受保护 Cookie 建立会话；如必须前端持有，也应使用更短生命周期的内存态令牌并配合刷新机制。

    测试场景

    验证 SSO 登录完成后，浏览器 `localStorage/sessionStorage` 中不存在认证 Token；同时确认刷新页面、跨标签页及 XSS 场景下无法通过 Web Storage 读取会话凭证。

    MAJORL17-19业务逻辑
    SSO 回调拿到 `authToken` 后直接把当前窗口跳到 `/`，没有把结果回传给登录页，也没有关闭回调页。若该流程按设计作为独立 SSO 回调页/弹窗使用，登录页上下文会丢失，失败时用户也只能停留在错误页，无法回到原始登录流程。
    修复建议与测试场景
    修复建议

    按 SSO 回调页模式处理：保存 token 后通过 `window.opener.postMessage(...)` 或 `BroadcastChannel` 通知登录页，再 `window.close()`；若没有 opener，再兜底跳转首页。失败场景也应提供返回登录页或关闭窗口的明确路径。

    测试场景

    补充集成测试：模拟登录页打开 SSO 回调窗口，回调页收到 `authToken` 后应向 opener 发送成功消息并关闭自身，而不是直接覆盖当前窗口地址。

    src/services/http.ts1 Major1 个问题
    MAJORL132API契约
    `res.data = res.payload ?? res.data` 改变了全局响应适配的语义：当后端明确返回 `payload: null` 时，调用方原本应拿到 `null`，现在会退回到原始响应对象/旧 `data`。依赖 `res.data` 获取可空结果的页面会把“空结果”误判成有数据，属于跨模块的接口契约回归。
    修复建议与测试场景
    修复建议

    仅在响应里不存在 `payload` 字段时才回退到原始 `data`，不要把 `null`/`undefined` payload 当成缺失字段处理。例如：`res.data = Object.prototype.hasOwnProperty.call(res, 'payload') ? res.payload : res.data;`。

    测试场景

    补充一个请求适配测试：后端分别返回 `{ payload: null }`、`{ payload: [] }`、`{ data: ... }`，断言 `res.data` 分别为 `null`、`[]` 和原始 `data`。
