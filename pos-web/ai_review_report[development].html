src/pages/user/Login/index.tsx

SeverityLineCategoryDescriptionSuggestionTest Scenario
major82-92安全风险发起 Microsoft 登录时没有生成并持久化随机 `state`/nonce，回调页也没有校验返回的 `state`。当前实现只要访问 `/user/sso?authToken=任意值` 就会接受该令牌，存在 OAuth 登录 CSRF / session fixation 风险，用户可能被强制登录到攻击者准备好的会话。发起登录前生成高熵 `state`（建议 `crypto.randomUUID()` 或随机字节），写入 `sessionStorage`，并把它传给后端生成授权 URL；回调页必须比对 `state` 后才接受结果，不匹配时中断登录并清理本地数据。示例：`const state = crypto.randomUUID(); sessionStorage.setItem('sso_state', state); const loginUrl = await getMicrosoftLoginUrl(type, state);`；回调中校验 `urlParams.get('state') === sessionStorage.getItem('sso_state')`。手工构造两个场景：1. 回调 URL 带错误 `state`；2. 直接访问 `/user/sso?authToken=fake&state=wrong`。修复后页面应拒绝登录并提示认证失败。

src/pages/user/sso/index.tsx

SeverityLineCategoryDescriptionSuggestionTest Scenario
critical10-19安全风险SSO 回调直接从 URL query 读取 `authToken` 并写入 `localStorage`。这会把登录令牌暴露在请求 URL 中，令牌可能进入网关/服务器访问日志、监控链路、崩溃上报或被同源跳转的 Referer 间接带出；即使随后 `replaceState` 清理地址栏，也无法消除首个 HTTP 请求已经携带过该敏感参数的事实。该问题属于认证凭证泄露风险。不要让后端把可用登录令牌放在 query string。改为标准 OAuth/OIDC code flow：回调仅携带一次性 `code`，前端将 `code` 提交给后端换取会话；如果必须前端处理，至少改为 URL fragment 或 popup + `postMessage` 传递短时凭证，避免令牌进入请求行。例如回调只接收 `?code=...&state=...`，随后调用 `/sso-login-api/sso/exchange` 兑换 token。走一遍 Microsoft 登录，检查浏览器 Network 中 `/user/sso` 首次请求 URL、Nginx/网关 access log、前端埋点是否出现 `authToken=`；修复后这些位置都不应再出现真实令牌。
