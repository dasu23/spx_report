    src/pages/Home/shortcuts.config.tsx1 Major1 个问题
    🔒 安全/权限相关 · 1 项（点击展开）
    MAJORL157-191权限与访问控制
    **原因：** 新增的 `tmsLoadingScan` 首页快捷入口未配置 `rightCode`，且注释明确写明“暂不设 rightCode（全员可见）”。这会让所有已登录用户都能在首页直接看到并进入装车扫描功能入口，而不是按既有权限体系进行菜单级访问控制。 ```tsx { key: 'tmsLoadingScan', path: '/tms-pda/loading-scan', titleKey: 'menu.tmsPda.loadingScan', icon: <CarOutlined />, color: '#f5222d', // 权限码待后续开会确定，暂不设 rightCode（全员可见） }, ``` **影响：** 若对应页面或后端接口未做足够严格的二次鉴权，低权限用户可通过该入口接触装车扫描业务，造成功能越权；即使后端最终拦截，也会扩大敏感业务面的可发现性与误操作面。
    修复建议与测试场景
    修复建议

    为该快捷入口补充明确的 `rightCode`，并确保与页面路由及后端接口使用同一权限码。例如： ```tsx { key: 'tmsLoadingScan', path: '/tms-pda/loading-scan', titleKey: 'menu.tmsPda.loadingScan', icon: <CarOutlined />, color: '#f5222d', rightCode: 'tmsPda:loading:scan', } ``` 同时补充页面级 `<Access>`/路由守卫，避免仅靠首页隐藏入口。

    测试场景

    使用不具备装车扫描权限的普通账号登录：1）确认首页不展示该快捷入口；2）直接访问 `/tms-pda/loading-scan` 时应被拒绝或重定向；3）具备权限账号可正常看到并访问。

    src/pages/gate-control/driver-entry-scan/service.ts2 Major2 个问题
    🔒 安全/权限相关 · 2 项（点击展开）
    MAJORL6-10权限与访问控制
    **原因：** 前端将 `facilityCode` 作为可控请求参数直接传给门禁核验接口，而本次 diff 中看不到任何将该参数与当前登录用户数据权限做绑定或校验的证据。页面侧仅通过菜单/路由权限码控制可见性，但真正的 HTTP 请求仍是： ```ts return request<GateVerifyResponse>('/admin/gate-code/verify', { method: 'GET', params, skipErrorHandler: true, }); ``` 这意味着拥有 `gateControl:driverEntry:scan` 的用户如果能篡改请求参数，就可能尝试以任意 `facilityCode` 查询/核验其他仓的司机进场信息。仅靠前端从 `initialState?.currentUser?.facilityCode` 取值不能构成安全边界。 **影响：** 若后端未基于登录态强制校验 `facilityCode` 归属，可能导致跨仓门禁核验越权，泄露司机姓名、DSP、Fleet、扫描时间等运营数据，或对非本仓司机执行错误的进场核验。
    修复建议与测试场景
    修复建议

    后端接口必须忽略前端传入的 `facilityCode`，改为从登录态/服务端数据权限上下文中解析当前用户所属仓；若确需传参，也要校验其必须属于当前用户可操作范围。前端可同步去掉该参数，避免误导为“客户端决定仓库”。示例：服务端以 session/JWT 中的 facilityCode 为准；前端改为仅传 `qrToken`。

    测试场景

    使用具备 `gateControl:driverEntry:scan` 权限但仅限仓 A 的账号，手工将请求 `/admin/gate-code/verify?facilityCode=B&qrToken=...` 中的 `facilityCode` 改为仓 B，验证后端是否拒绝并返回无权限/越权错误。

    MAJORL6-10安全风险
    新增门禁核验请求使用 GET，并将 `facilityCode` 与 `qrToken` 放在 `params` 中拼接到 URL 查询串。二维码 token 属于敏感凭证类数据，出现在 URL 后会更容易被浏览器历史、代理/CDN/access log、监控链路或 Referer 泄露，扩大凭证暴露面。
    修复建议与测试场景
    修复建议

    将该核验接口改为 POST，并把 `facilityCode`、`qrToken` 放入请求体而不是查询参数；同时确认服务端与网关日志对该字段做脱敏处理。例如：`request('/admin/gate-code/verify', { method: 'POST', data: params, skipErrorHandler: true })`。

    测试场景

    抓包或查看前端 request 配置，确认扫描后请求 URL 不再包含 `qrToken`；同时验证代理/网关访问日志中不会记录明文 token。

    异常图片上传接口同样从 `localStorage` 读取 `SPX_AUTH_TOKEN` 并手工拼接到 `Authorization` 头，重复引入了前端可读 Token 的敏感数据暴露风险。该接口还处理图片上传，一旦凭证被窃取，攻击者可冒充用户提交异常图片/报告。

    src/services/tms-pda/request.ts1 Major1 个问题
    🔒 安全/权限相关 · 1 项（点击展开）
    MAJORL29-32安全风险
    PDA 专用请求实例把认证凭证直接从 `localStorage` 读取并注入 `Authorization` 头。`localStorage` 中的 Token 对任意同源脚本可读，一旦页面存在 XSS 或第三方脚本被污染，攻击者即可直接窃取长期有效凭证并调用 PDA 装卸车/封签相关接口。这属于新增的敏感凭证暴露面。
    修复建议与测试场景
    修复建议

    避免将认证 Token 持久化在 `localStorage`。优先改为服务端下发 `HttpOnly + Secure + SameSite` Cookie；若受现有架构限制，至少改为仅保存在内存态，并配合严格 CSP/最小化第三方脚本。示例：移除 `localStorage.getItem(...)` 读取，改由统一鉴权层从受保护的 cookie/session 自动携带。

    测试场景

    在浏览器控制台模拟同源脚本执行 `localStorage.getItem(SPX_AUTH_TOKEN)`，确认当前实现可直接读出凭证；修复后应无法通过 JS 读取认证凭证，同时 PDA 请求仍能正常鉴权。
