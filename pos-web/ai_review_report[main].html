major221-226业务逻辑并发锁分支直接 `return`，会把用户在前一次请求未完成时提交的第二次扫描结果静默丢弃。扫描枪/PDA 场景下用户通常不会感知这次提交被忽略，容易导致包裹漏扫，属于业务行为回退。不要静默忽略后续扫描。至少应给出明确提示，或缓存最后一次输入并在当前请求完成后自动继续处理。例如维护一个 `pendingCodeRef`，在 `finally` 中补执行一次。补充测试：第一次扫描 pending 时再次扫码，第一次完成后应能看到第二次扫描被提示/排队执行，而不是直接丢失。
minor105边界条件`onSearch` 在校验 `code` 之前就把结果区状态设置成了 `loading`。当 `lockSearch` 被空输入触发时，不会发起扫描，但界面会一直显示 `Scan in progress…`，造成错误的处理中状态。先判断 `code` 是否为空，再进入 loading 状态。例如：`if (!code) { setHistory(undefined); return; } setHistory({ message: 'Scan in progress…', status: 'loading' });`。补充一个空输入点击搜索的测试，断言不会显示 `Scan in progress…`，且 `onScan` 不会被调用。
