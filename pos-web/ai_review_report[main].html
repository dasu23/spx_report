SeverityLineCategoryDescriptionSuggestionTest Scenario
major15-17业务逻辑SSO 回调成功后仅把 token 写入 `localStorage` 并强制跳转 `/`，没有按新增需求通知登录页、更新当前登录流程状态并关闭 SSO 页面。如果前面按需求改成弹窗打开，这个页面将无法完成主窗口登录闭环，用户会停留在多窗口/错误页面状态。在成功分支增加跨窗口通知与关闭逻辑，例如：`window.opener?.postMessage({ type: 'sso-success', authToken }, window.location.origin); window.close();`；仅在 `window.opener` 不存在时再兜底跳转首页。失败分支同样应通知主窗口错误状态。用 `window.open` 打开 `/user/sso?authToken=...`，断言主窗口收到成功消息、token 生效且子窗口执行 `close()`；失败场景应回传错误而不是直接留在回调页。

tailwind.config.js

SeverityLineCategoryDescriptionSuggestionTest Scenario
major38配置变更启用 `important: true` 会把所有 Tailwind 工具类提升为 `!important`，在当前项目大量使用 Ant Design 组件的前提下，容易全局覆盖组件内联/主题样式，造成按钮、表单、表格等 UI 表现异常。避免全局开启 `important: true`。如果只是想限制 Tailwind 作用域，可改成 `important: '#root'` 或项目实际挂载节点；若无明确冲突场景，直接移除此配置更稳妥。对包含 Ant Design 组件和 Tailwind 工具类混用的页面做回归，重点检查按钮、表单项、表格单元格的颜色、间距、字号是否被意外覆盖。
major39配置变更`content` 仅扫描 `pages/components/layouts`，会漏掉本次变更上下文里已存在的 `src/app.tsx` 等入口文件中的 Tailwind 类名。生产构建时这些类会被裁剪，导致样式在线上缺失。将扫描范围扩展到整个 `src`，例如：`content: ['./src/**/*.{ts,tsx}']`，或至少补上 `src/app.tsx`、`src/global.tsx` 等实际会写 Tailwind 类名的目录/文件。在 `src/app.tsx` 或其他未被当前 glob 覆盖的文件中添加一个 Tailwind 类名，执行生产构建后验证该样式仍然存在。
