    .claude/commands/opsx/apply.md1 Critical 1 Minor2 个问题
    CRITICALL25安全风险
    文档指示直接将可由用户输入或会话上下文推断得到的 `<name>` 拼接进 shell 命令 `openspec status --change "<name>" --json`。这里使用双引号不能阻止 `$(...)` 或反引号命令替换，后续如果代理按说明执行，会形成命令注入入口。
    修复建议与测试场景
    修复建议

    不要把未校验的变更名直接插入 shell。先用 `openspec list --json` 得到允许值并做白名单匹配；执行命令时改为参数数组传参而非 shell 拼接。如果必须经过 shell，至少对输入做严格字符白名单限制（如 `^[a-z0-9-]+$`）并拒绝任何包含 `$`、反引号、斜杠、空白等特殊字符的名称。

    测试场景

    构造变更名为 `foo$(touch /tmp/pwned)` 或 ``foo`touch /tmp/pwned` ``，验证执行 apply 流程时不会触发命令替换，且系统明确拒绝非法名称。

    MINORL65-70数据一致性
    这里要求对每个 pending task 在完成代码修改后立即把 `- [ ]` 改成 `- [x]`，但没有要求先执行校验、刷新任务状态或确认验收结果。若实现后仍有编译/测试失败或任务部分完成，tasks.md 会被提前标记完成，导致任务进度失真。
    修复建议与测试场景
    修复建议

    在勾选任务前增加明确的完成校验步骤，例如“运行对应验证命令/人工验收并重新获取 `openspec instructions apply --json` 状态，确认当前 task 已满足后再更新 tasks.md`”。

    测试场景

    模拟一个 task 完成代码修改但验证失败的场景；确认流程不会把该 task 直接标记为 `- [x]`，而是保持 pending 并输出失败原因。

    .claude/commands/opsx/bulk-archive.md2 Major2 个问题
    MAJORL39安全风险
    文档要求按 `openspec/changes/<name>/tasks.md` 读取文件，但未要求对 `<name>` 做路径规范化或目录边界校验。若选中的 change name 可被污染，后续实现容易出现 `../` 型路径穿越，导致读取工作区内非预期文件。
    修复建议与测试场景
    修复建议

    读取任务文件前，先将 change name 与 `openspec list --json` 的合法 ID 做白名单校验；拼接路径后执行规范化，并校验结果仍位于 `openspec/changes/` 根目录下，拒绝包含 `/`、`..`、反斜杠等路径控制字符的名称。

    测试场景

    使用伪造名称 `../../.npmrc` 或 `../specs/x` 触发 bulk-archive 流程，验证实现会拒绝该输入，且不会读取 `openspec/changes/` 目录外文件。

    MAJORL68-70业务逻辑
    冲突解决规则把“两个 change 都已实现”的情况简化为按时间顺序直接覆盖应用，默认认为后一个 delta 可以安全覆盖前一个 delta。对于同一 capability 的并行修改，这会静默丢失先前 requirement 的局部变更或场景描述，批量归档后主 spec 可能与实际实现不一致。
    修复建议与测试场景
    修复建议

    冲突场景不要仅靠时间顺序覆盖。应先产出 capability 级差异摘要，再要求显式合并策略：逐 requirement 合并、人工确认保留项，或在无法自动合并时阻止批量归档。至少应在检测到同一 requirement 被多次修改时中止自动 sync。

    测试场景

    选择两个都修改同一 capability 且修改不同 requirement 段落的 change 执行 bulk archive；验证流程不会直接按时间顺序覆盖，而是要求显式合并或阻止继续。
