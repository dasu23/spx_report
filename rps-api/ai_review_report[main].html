    rps-service/src/main/java/com/miq/rpsservice/cmd/oms/module/external/VintedPushModule.java1 Major1 个问题
    MAJORL170-172边界条件
    兜底构建 `VintedExtInfo` 时，只判断了 `recipient != null`，但没有判断 `recipientName/recipientPhone/recipientEmail` 是否为空，直接传给 `CryptoUtil.encrypt(...)`。当请求缺少其中任一字段且又未提供 `vintedExtInfo` 时，这里很容易抛出空指针或加密异常，导致建单失败。
    修复建议与测试场景
    修复建议

    对每个字段分别做空值保护，或在进入默认分支前显式校验必填字段。例如：`recipientName(recipient != null && recipient.getRecipientName() != null ? CryptoUtil.encrypt(recipient.getRecipientName()) : null)`。如果这些字段是必填，建议直接抛出带业务语义的校验异常。

    测试场景

    构造 `features` 为空且 `recipient` 存在、但 `recipientEmail` 或 `recipientPhone` 为 `null` 的表单，验证 `buildCreateRequest(...)` 的行为符合预期且不会出现非受控异常。

    rps-service/src/main/java/com/miq/rpsservice/ext/rate/RateCardDataSource.java1 Major1 个问题
    MAJORL95-101,111-119,129-137边界条件
    新增的 3 个版本缓存加载器都直接对 `rateFacade.*(...).getData()` 解引用。只要下游 RPC 在超时/降级时返回 `null` 响应对象，这里就会在缓存加载阶段触发 `NullPointerException`，进而导致对应查价请求直接失败。
    修复建议与测试场景
    修复建议

    先保存 facade 返回值并做空判断，再读取 `getData()`。例如：`BaseResponse<RateCardVersionResultDto> resp = rateFacade.getRateCardVersionInfoCache(...); return resp == null || resp.getData() == null ? new RateCardVersionResultDto() : resp.getData();`，另外两个 loader 同样处理。

    测试场景

    分别 mock `rateFacade.getRateCardVersionInfoCache/getZoneRateCardVersionCache/getZipCodeRateCardVersionCache` 返回 `null`，验证 `findValid(...)` / `findValidByZipCode(...)` 不抛异常且返回空结果。

    rps-service/src/main/java/com/miq/rpsservice/service/impl/VintedPickupLocationServiceImpl.java1 Minor1 个问题
    MINORL55性能隐患
    缓存刷新周期从 3 小时缩短到 5 分钟，而 loader 注释说明每次刷新会先查数据库。对热点 `pointCode` 来说，这会把刷新频率提升 36 倍，显著增加数据库访问次数，并在访问触发刷新时放大请求延迟。
    修复建议与测试场景
    修复建议

    确认业务是否真的需要 5 分钟级刷新；如果只是希望更快感知变更，优先考虑保留较长 `refreshAfterWrite` 并配合主动失效，或至少评估数据库承载后再下调刷新周期。

    测试场景

    对同一个热点 `pointCode` 做持续访问压测，比较变更前后数据库查询次数和接口 TP99 延迟，确认刷新频率调整不会造成明显性能回退。
