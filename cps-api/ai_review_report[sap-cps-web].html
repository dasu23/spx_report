critical64-80安全风险注册流程虽然校验了邀请码有效性，但未把邀请码与请求中的 `speedxAccountNumber` 绑定校验；`invitationInfo` 结果未被使用。攻击者可持有效邀请码（和匹配邮箱）将请求体中的账号号改为其他已存在账号，从而越权加入非目标账号。注册时以邀请码解析/持久化的账号号为准，禁止信任客户端传入账号号；新增强校验：`req.speedxAccountNumber` 必须等于邀请码对应账号，否则拒绝。更稳妥做法是直接从邀请码反查账号并覆盖请求值。为账号A生成邀请码，注册请求将 `speedxAccountNumber` 改为账号B，预期返回 4xx 且不创建 B 账号下用户。
major118安全风险新注册用户密码使用 `MD5Util.doSign` 存储，属于弱哈希方案（无自适应成本），数据库泄露后易被离线暴力破解。改为 `BCrypt`/`Argon2` 等专用密码哈希（带随机盐和成本因子），并补充平滑迁移策略（登录时识别旧MD5并升级为新哈希）。创建弱密码账户后导出哈希，验证不能在短时间内通过常见字典/彩虹表直接还原。
major64-70,165-168数据一致性邀请码“单次使用”通过流程末尾软删除实现，存在并发竞态：两个并发注册请求可同时通过前置校验，然后都创建用户，最后都尝试删除邀请码，导致同一码被重复消费。将“消费邀请码”改为事务内原子操作并前置执行，例如 `UPDATE ... SET deleted=1 WHERE invite_code=? AND deleted=0`，仅当影响行数=1才继续；或 `SELECT ... FOR UPDATE` 锁行后再执行业务。同一邀请码并发发起两次注册，预期仅一条成功，另一条返回“邀请码已使用”。
