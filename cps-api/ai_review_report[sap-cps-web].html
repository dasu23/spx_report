critical68-84安全风险`registerShipper` 只校验了邀请码存在和邮箱匹配，但实际创建/绑定 CPS 账号时使用的是请求体里的 `req.getSpeedxAccountNumber()`，没有校验它是否等于邀请码中对应的账号。攻击者只要拿到发给自己邮箱的任意有效邀请码，就可以把注册流程指向其他 `speedxAccountNumber`，从而越权创建并接入不属于自己的 shipper account。注册时以 `invitationService.validateShipperInvitation()` 返回的账号号为唯一可信来源，并在进入建档前显式校验 `req.getSpeedxAccountNumber()` 与邀请码中的账号完全一致；更稳妥的做法是去掉请求体中的 `speedxAccountNumber`，后端全程使用邀请码解析出的值。构造一个发给 `userA@example.com` 的有效邀请码，提交注册请求时将 `speedxAccountNumber` 改成另一个已存在但未被邀请的账号号，预期接口应拒绝并且数据库中不产生该账号/用户关联。
