    cps-infrastructure/src/main/java/com/speedx/cps/integration/OmsClient.java3 Major3 个问题
    MAJORL219-227并发安全
    新增批量缓存采用“先查后写”模式：先通过 `getIfPresent(cacheKey)` 判断，再调用远程 OMS，最后 `put` 到共享 `Caffeine` 缓存。该类通常为 Spring 单例 Bean，在并发请求命中同一 `netsuiteCodes` 组合时，多个线程会同时穿透缓存并发起重复下游调用，形成竞态放大；在外部依赖较慢或不稳定时，容易被恶意高并发触发，造成下游接口放大访问和本服务资源消耗。
    修复建议与测试场景
    修复建议

    改为使用 Caffeine 的原子加载接口，避免并发下重复回源。例如：`availablePickupsBatchCache.get(cacheKey, k -> fetchAvailablePickupsByNetsuiteCodes(netsuiteCodes))`；如需区分空结果，也应统一缓存占位值或显式设置短 TTL 的空结果缓存。

    测试场景

    增加并发测试：20~100 个线程同时调用 `listAvailablePickupsByNetsuiteCodes` 且参数完全相同，断言 `fetchAvailablePickupsByNetsuiteCodes`/下游 HTTP 实际只被调用 1 次或受控次数，而不是与并发数近似线性增长。
