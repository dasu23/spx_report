    MAJORL108安全风险
    新增的 info 日志直接记录了 `toEditPickup` 接口完整响应 `res`。该响应反序列化目标为 `CpsPickupCreateForm`，按命名和调用场景推断通常包含取件地址、联系人等业务敏感信息；在生产日志中明文落盘会扩大敏感数据暴露面。
    修复建议与测试场景
    修复建议

    不要记录完整响应体。改为只记录请求链路标识、结果码、是否成功等最小必要信息；如确需排障，先对地址、电话、邮箱等字段做脱敏后再输出，并将日志级别降到受控的 debug。

    测试场景

    构造包含真实取件地址/联系人信息的 `toEditPickup` 响应，执行调用后检查应用日志，确认日志中不再出现完整 JSON 或敏感字段明文。

    MAJORL29-30,37-38,45-46,53-54,61-62,69-70,77-78,85-86,93-94,101-102,110-111边界条件
    多个新方法直接链式访问入参中的嵌套对象，例如 `form.getAccountDto().getCountry()`、`form.getAddressDto().getPickupCountry()`、`form.getPickupAddress().getPickupCountry()`。一旦调用方传入的表单对象缺少子对象或国家字段为空，这里会在发起远程调用前直接触发空指针，导致揽收创建、取消、查询、转编辑等核心流程异常失败。
    修复建议与测试场景
    修复建议

    在读取国家信息前统一做非空校验，并返回明确的业务异常而不是让 NPE 向外冒出。例如先抽出 `private List<String> resolveCountry(...)`，对 form、accountDto/addressDto/pickupAddress 及 country 做判空：`if (form == null || form.getAccountDto() == null || StringUtils.isBlank(form.getAccountDto().getCountry())) { throw new BizException("pickup country is required"); }`。

    测试场景

    补充空值边界测试：分别传入 `form=null`、`accountDto/addressDto/pickupAddress=null`、`country=null/blank`，断言不会抛出 NPE，而是返回预期的业务异常。

    MAJORL31-33,39-41,47-49,55-57,63-65,71-73,79-81,87-89,95-97,103-105,112-114配置变更
    新客户端将 `pickupApiConfig.getTimeout(country)` 的返回值直接参与乘法运算，如 `timeout * 20`。该值是 `Integer`，如果某个国家未配置超时或配置读取失败，会在自动拆箱时抛出空指针；同时 `host` 也未校验，配置缺失时可能拼出非法 URL。新增 `PickupClient` 后，这会把配置遗漏直接放大为运行时故障。
    修复建议与测试场景
    修复建议

    对 `host` 和 `timeout` 做显式校验，并提供默认值或抛出可定位的配置异常。例如：`Integer timeout = Optional.ofNullable(pickupApiConfig.getTimeout(country)).orElse(2000); String host = Objects.requireNonNull(pickupApiConfig.getHost(country), "pickup api host not configured");`。若业务不允许兜底，建议抛出 `BizException` 并带上国家码。

    测试场景

    补充配置缺失测试：模拟 `getTimeout(country)` 返回 `null`、`getHost(country)` 返回 `null`，断言方法返回明确的配置异常信息，而不是发生 NPE 或请求到非法地址。
