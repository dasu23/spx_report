    cps-facade-impl/src/main/java/com/speedx/cps/facade/impl/web/CommonFileController.java3 Major3 个问题
    MAJORL29-35边界条件
    这里直接调用 `req.getFile().getOriginalFilename()` 并据此拼接路径，没有对 `req.getFile()`、`originalFilename` 为空或空串做保护。上传空文件字段或缺失文件时会触发空指针，文件名为空时也会生成非法/异常 blobPath。
    修复建议与测试场景
    修复建议

    在读取文件名前增加显式校验，并在文件名为空时返回可读的业务异常。示例：`if (req.getFile() == null || req.getFile().isEmpty() || StringUtils.isBlank(req.getFile().getOriginalFilename())) { throw new BizException(400, "file is required"); }`。同时使用 `FilenameUtils.getName(...)` 规整文件名。

    测试场景

    分别提交 `file` 缺失、空文件、`originalFilename` 为空字符串的请求，验证接口返回 4xx 业务错误而不是 500/NPE。

    MAJORL30-32业务逻辑
    `_STATIC` 类型上传直接使用 `originalFilename` 作为固定 blob key。相同文件名的再次上传会覆盖已有对象，导致静态资源被静默替换，属于明确的数据丢失风险。
    修复建议与测试场景
    修复建议

    为静态资源引入防覆盖策略，例如在上传前检查 blob 是否已存在并拒绝重复文件名，或追加版本号/哈希后缀生成唯一 key。示例：`String blobPath = "cps/static/" + UUID.randomUUID() + "-" + FilenameUtils.getName(originalFilename);`；如果必须保留原名，至少先做存在性校验并返回业务错误。

    测试场景

    构造两次 `_STATIC` 上传，请求使用相同文件名但不同内容；验证第二次上传不会无提示覆盖第一次内容，而是返回冲突错误或生成新版本路径。

    异常分支将底层 Azure 上传错误的原始 `e.getMessage()` 直接返回给客户端，可能泄露存储桶名称、对象路径、SDK/网络细节等内部实现信息，扩大后续攻击面的信息收集能力。
