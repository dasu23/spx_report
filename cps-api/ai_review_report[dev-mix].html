shippers-portal/src/main/java/com/speedx/shippers/service/ShipperTicketService.java

SeverityLineCategoryDescriptionSuggestionTest Scenario
critical104-129安全风险`getTicketDetail`、`addComment`、`uploadAttachment` 只校验“已登录”，没有校验当前用户是否属于该工单的 requester/CC/所属 shipper account。只要知道或猜到 `ticketNumber`，任意登录用户就可以读取工单详情、追加评论或上传附件，属于典型越权访问与越权修改。在这三个入口统一增加授权校验：先查询工单，再校验当前用户邮箱是否为 requester/CC，或工单是否归属当前用户可访问的 shipper account；校验失败返回 403。不要仅依赖前端隐藏入口。构造用户 A 创建工单、用户 B 登录后调用详情/评论/上传接口并传入 A 的 `ticketNumber`，应稳定返回 403，而不是成功读写。
critical83-85安全风险当 `businessContactEmail` 为空时，这里会生成一个随机 `@gmail.com` 地址并作为 `accountEmail` 发送到 SAP 建单流程。该地址很可能对应真实第三方邮箱，后续若 SAP/人工流程向该邮箱发送通知或工单信息，会把客户数据泄露给外部无关人员。禁止使用随机外部邮箱兜底。缺少业务邮箱时应直接拒绝建单并提示补全资料，或改为平台自有且受控的 `no-reply` 域名地址，同时明确关闭外发通知。准备一个 `businessContactEmail` 为空的 shipper account，调用建单接口；系统应返回可处理的校验错误，且请求体中不应出现任何随机外部邮箱。
