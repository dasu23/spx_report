    MAJORL32-33业务逻辑
    `_STATIC` 类型上传直接使用原始文件名作为 Blob 路径，重复上传同名文件时会覆盖已有对象，存在静态资源被意外替换/数据丢失的风险。该问题出现在新增上传逻辑的主路径上，影响实际业务结果。
    修复建议与测试场景
    修复建议

    避免将用户文件名直接作为唯一存储键。可在静态目录下追加版本号、时间戳或 UUID，例如：`String blobPath = "cps/static/" + UUID.randomUUID() + "-" + originalFilename;`；如果业务要求固定文件名，则应在上传前先检查对象是否已存在，并明确返回“文件已存在”错误而不是直接覆盖。

    测试场景

    上传两次相同 `originalFilename` 且 `type=_STATIC` 的文件，验证第二次请求不会覆盖第一次上传的内容；同时补充“已存在文件”或“自动生成新路径”的断言。

    MINORL48-49边界条件
    下载地址通过 `String.format("%s?%s", fileUrl, sasToken)` 直接拼接查询串；当 `fileUrl` 已包含查询参数时，会生成两个 `?` 的非法 URL，导致返回的下载链接不可用。
    修复建议与测试场景
    修复建议

    按 URL 是否已有查询串选择分隔符，或使用 URI 组件工具安全拼接，例如：`String separator = fileUrl.contains("?") ? "&" : "?"; String redirectUrl = fileUrl + separator + sasToken;`。

    测试场景

    分别传入无查询串的 Blob URL 和已包含查询参数的 URL，断言返回地址分别使用 `?` 和 `&` 拼接，且最终链接可正常访问。
