SeverityLineCategoryDescriptionSuggestionTest Scenario
critical132-157安全风险`getTicketDetail`、`addComment`、`uploadAttachment` 只依赖登录态，随后直接按 `ticketNumber` 调用 SAP 接口，没有校验当前用户是否为工单 requester、CC，或是否属于该工单对应的 shipper account。任意已登录用户只要拿到/猜到工单号，就能查看他人工单详情并追加评论、附件，属于越权访问和敏感数据泄露。在这三个入口先查询工单，再基于 `requesterEmail`、CC 列表或 `speedxAccountNumber` 与当前用户可访问的 `ShipperUserProfile` 做授权校验；校验失败返回 403。示例：先 `TicketDetailDto detail = ticketApiClient.getTicket(ticketNumber);`，再校验 `detail.getRequesterEmail().equalsIgnoreCase(cpsUser.getEmail())` 或 `userProfileRepository.findByCpsUserIdAndShipperAccountId(...)` 命中后才允许继续。准备两个不同 shipper 用户 A/B，由 A 创建工单后，使用 B 调用详情、评论、上传附件接口，预期均被拒绝；A 自己调用应成功。
major110-113数据一致性当 `businessContactEmail` 为空时，这里会伪造一个随机 `gmail.com` 地址传给 SAP。后续 `TicketApiClient.createCase()` 在 SAP 中查不到账号时会用这个假邮箱创建 account/contact，直接把错误联系人写进外部系统，导致工单归属、通知对象和主数据被污染。不要生成虚假邮箱。优先使用真实的 `businessContactEmail` / `billingEmail` / 当前提交人的邮箱；如果都没有，直接阻止建单并提示先完善 shipper account 联系信息。准备一个 `ShipperAccountProfile` 缺少联系邮箱的账号并发起建单；修复前会把随机邮箱写入 SAP，修复后应返回明确的参数/数据校验错误。
