    cps-infrastructure/src/main/java/com/speedx/cps/integration/OmsClient.java2 Major 1 Minor3 个问题
    MAJORL119边界条件
    `rootNode.get("code").asInt()` 在 OMS 返回非预期报文、空响应体或缺少 `code` 字段时会直接触发空指针，导致这里把上游异常格式放大成服务内部错误，无法按预期返回业务异常。
    修复建议与测试场景
    修复建议

    先校验 `rootNode` 和 `code` 节点是否存在，再读取状态码，例如：`JsonNode codeNode = rootNode.path("code"); if (codeNode.isMissingNode() || codeNode.isNull()) { throw new RRException("OMS response missing code"); } int code = codeNode.asInt();`。同时对 `msg` 也改用 `path("msg").asText("")` 避免同类问题。

    测试场景

    补充单测：1) OMS 返回 `{}`；2) 返回 `{"msg":"error"}`；3) 返回空字符串/非法 JSON。断言方法抛出可预期的 `RRException`，而不是 `NullPointerException`。

    MAJORL133安全风险
    新增的异常处理将底层异常消息直接拼接进对外抛出的 `RRException`。当 OMS 返回异常内容、JSON 解析失败或底层 HTTP 客户端报错时，调用方可能直接获得内部实现细节，形成信息泄露面，便于攻击者推断下游接口、报文结构或运行时状态。
    修复建议与测试场景
    修复建议

    对外返回固定的通用错误文案，例如 `request oms listAvailablePickups error`，将详细异常仅记录在服务端日志中；如需保留诊断信息，增加错误码而不是透传 `e.getMessage()`。

    测试场景

    构造 OMS 返回畸形 JSON、连接失败或包含内部报错文本的场景，验证接口响应中不包含底层异常原文、URL、解析细节等内部信息。

    MINORL111-117性能隐患
    `listAvailablePickups` 每次调用都会直接访问 OMS，没有像 `getAccountByPickupId` 一样做任何本地缓存或去重。该接口按 `soldToAccountId` 查询，若在下单页频繁刷新/联动调用，容易形成重复远程请求并放大接口延迟。

    修复建议

    如果该数据短时间内变化不频繁，可按 `soldToAccountId` 增加短 TTL 缓存，或至少使用 `Cache#get(key, mappingFunction)` 做请求合并，避免同一账号在并发场景下重复打 OMS。
