    cps-infrastructure/src/main/java/com/speedx/cps/integration/PickupClient.java2 Major2 个问题
    MAJORL107安全风险
    `toEditPickup` 将外部 Pickup API 的完整响应体直接写入 `info` 日志。该响应会被反序列化为 `ResponseObject<CpsPickupCreateForm>`，按类型推断很可能包含取件地址、联系人等业务敏感信息；一旦日志被集中采集或被低权限运维账号访问，会形成非必要的数据暴露面。
    修复建议与测试场景
    修复建议

    删除该整包响应日志，或改为仅记录必要的非敏感元数据（如请求结果码、traceId、记录数）。如果确需排障日志，应在打印前对地址、电话、姓名等字段做脱敏，并将级别降为受控的 debug 日志。

    测试场景

    构造包含真实地址/联系人字段的 `toEditPickup` 响应，调用该方法后检查应用日志，确认不会输出完整响应体或任何未脱敏的敏感字段。

    MAJORL29-110边界条件
    多个新方法直接链式访问入参的嵌套字段，例如 `form.getPickupAddress().getPickupCountry()`、`form.getAccountDto().getCountry()`、`form.getAddressDto().getPickupCountry()`。一旦调用方传入的表单缺少这些子对象，当前代码会在发起远程调用前抛出 `NullPointerException`，导致创建/取消/查询/编辑提货等核心流程直接失败。
    修复建议与测试场景
    修复建议

    在每个入口方法中先校验必要对象和国家字段，再继续组装请求；对缺失参数返回明确业务异常而不是依赖 NPE。示例：`if (form == null || form.getAccountDto() == null || StringUtils.isBlank(form.getAccountDto().getCountry())) { throw new BizException("country is required"); }`。对 `pickupAddress`、`addressDto` 等同样处理，或抽取统一的 `resolveCountry(...)` 校验方法。

    测试场景

    补充单测覆盖 `accountDto/pickupAddress/addressDto` 为 null、以及 country 为空的场景，验证不会抛出 NPE，而是返回可识别的业务异常。
