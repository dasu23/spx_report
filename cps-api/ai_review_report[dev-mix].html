    MAJORL31-33数据一致性
    `_STATIC` 类型上传直接使用 `originalFilename` 作为 Blob key，两个用户上传同名文件时会写到同一路径 `cps/static/<filename>`，后一次上传会覆盖前一次内容，存在静态资源被误覆盖/数据丢失的风险。
    修复建议与测试场景
    修复建议

    为静态文件路径补充唯一性约束，例如在文件名前追加业务目录、日期分桶或 UUID；如果业务上要求固定文件名，则在上传前显式检查目标 Blob 是否已存在并根据策略拒绝覆盖或生成新版本。示例：`String blobPath = "cps/static/" + UUID.randomUUID() + "-" + originalFilename;`

    测试场景

    连续上传两个同名静态文件（如 `logo.png`），验证第二次上传不会覆盖第一次文件，且两个返回地址都可正确访问。

    MINORL48业务逻辑
    下载地址拼接固定使用 `?%s`，当 `fileUrl` 本身已经带有查询参数时会生成两个 `?` 的非法 URL，导致返回的 SAS 链接不可用。
    修复建议与测试场景
    修复建议

    根据 `fileUrl` 是否已包含 `?` 选择 `?` 或 `&` 进行拼接，例如：`String separator = fileUrl.contains("?") ? "&" : "?"; String redirectUrl = fileUrl + separator + sasToken;`

    测试场景

    分别传入无查询参数的 Blob URL 和已包含查询参数的 URL，验证返回地址都只有一个 `?`，且附加参数位置正确。
