    cps-infrastructure/src/main/java/com/speedx/cps/integration/PickupClient.java2 Major2 个问题
    MAJORL110安全风险
    新增的 `log.info("toEditPickup from pickup api res: {}", res);` 会将下游 Pickup API 的完整响应写入信息级日志。该响应对象类型为 `ResponseObject<CpsPickupCreateForm>`，按上下文推断通常包含提货地址、联系人等业务明细；一旦日志被集中采集或被低权限运维人员访问，会扩大敏感业务数据暴露面。
    修复建议与测试场景
    修复建议

    不要记录完整响应体。改为仅记录请求结果摘要，例如状态码、业务码、traceId，必要时对地址、联系人、电话等字段做脱敏后再打印，并将此类日志降为 `debug` 且默认关闭。

    测试场景

    构造包含完整提货地址和联系人信息的 `toEditPickup` 响应，调用该方法后检查应用日志，确认日志中不再出现完整 JSON/敏感字段，仅保留脱敏后的摘要信息。

    MAJORL29-32,36-39,43-46,50-53,57-60,64-67,71-74,78-81,85-88,92-95,99-103,107-110边界条件
    多个新方法直接链式读取 `form/cpsSo/dto` 的嵌套字段并立即参与路由与超时计算，例如 `form.getPickupAddress().getPickupCountry()`、`cpsSo.getAccountDto().getCountry()`、`timeout * 20`。当入参对象、嵌套 DTO、country，或配置返回的 timeout 为空时，这里会直接触发 `NullPointerException`，导致接口在发起下游请求前崩溃。
    修复建议与测试场景
    修复建议

    在每个入口先校验必填对象和 country，再对 `pickupApiConfig.getHost/getTimeout` 的返回值做非空保护；例如先提取 `String country = Optional.ofNullable(form).map(CpsPickupCreateForm::getPickupAddress).map(PickupAddressDto::getPickupCountry).filter(StringUtils::isNotBlank).orElseThrow(() -> new BizException("pickup country is required"));`，随后校验 `timeout != null` 后再计算超时。

    测试场景

    补充单测/集成测试：1）`form` 或嵌套 DTO 为 null；2）country 为 null/空串；3）`PickupApiConfig` 对该 country 未配置 timeout。期望返回明确业务异常，而不是 NPE。

    `validateTicketAccess` 直接对 `registrationService.getShipperAccountsByUserID(cpsUserId)` 的返回值调用 `stream()`，并在 Lambda 中对 `a.getSpeedxAccountNumber()` 直接调用 `equals(...)`。只要账号列表返回 `null`、列表元素中的账号号为 `null`，这里就会触发 `NullPointerException`，导致查看工单、评论、上传附件等主流程直接失败。

    修复建议

    对返回结果和账号号做空值保护，再比较账号号，例如：`List<ShipperAccountInfoResp> userAccounts = Optional.ofNullable(registrationService.getShipperAccountsByUserID(cpsUserId)).orElse(Collections.emptyList()); boolean hasAccess = userAccounts.stream().map(ShipperAccountInfoResp::getSpeedxAccountNumber).filter(Objects::nonNull).anyMatch(acc -> Objects.equals(acc, speedxAccountNumber));`。同时建议补充 `speedxAccountNumber` 为空时的兜底处理。

    测试场景

    补充单测覆盖以下场景：1. `getShipperAccountsByUserID` 返回 `null`；2. 返回列表中存在 `speedxAccountNumber == null` 的元素；3. 工单 `speedxAccountNumber` 为空。验证接口不会抛出 NPE，而是返回可预期的业务异常或拒绝结果。
