    cps-infrastructure/src/main/java/com/speedx/cps/integration/OmsClient.java2 Major2 个问题
    MAJORL33-45并发安全
    `OmsClient` 是 Spring 单例组件，这里把 `PickupAccountInfoResp` 直接缓存并原样返回。若该 DTO 为可变对象（常见 Lombok/JavaBean 场景），后续调用方一旦修改返回对象，会把修改后的状态留在全局缓存中，被其他线程/请求复用，造成跨请求数据污染，严重时会演变为敏感信息串读。
    修复建议与测试场景
    修复建议

    不要在单例缓存中直接暴露可变 DTO。至少在写入缓存和返回结果时做防御性拷贝，或将 `PickupAccountInfoResp` 设计为不可变对象；如果使用 Caffeine，可改为 `cache.get(key, k -> fetch...)` 配合拷贝逻辑，避免共享同一可变实例。

    测试场景

    新增并发测试：线程 A 调用 `getAccountByPickupId` 后修改返回对象字段，线程 B 随后读取同一 `pickupAccountId`，验证 B 不会看到 A 的修改；再补充多线程同时命中同一 key 的测试，确认缓存内不会暴露共享可变状态。

    MAJORL38-118边界条件
    这里直接对 `rootNode.get("code")` 和 `rootNode.get("msg")` 调用 `asInt()/asText()`，默认假设 OMS 响应一定包含这两个字段。若上游返回非预期 JSON 或缺少字段，`get(...)` 会返回 `null`，这里会触发 `NullPointerException`，导致原本可归类为接口异常的场景变成服务端崩溃。
    修复建议与测试场景
    修复建议

    改为先校验字段存在性，或使用更稳妥的 `path()` 读取并结合缺省值判定。例如：`JsonNode codeNode = rootNode.path("code"); if (codeNode.isMissingNode()) { throw new RRException("OMS response missing code"); }`；`msg` 也同样处理。
