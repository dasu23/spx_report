    cps-infrastructure/src/main/java/com/speedx/cps/config/BiApiConfig.java1 Major1 个问题
    MAJORL11-15安全风险
    `BiApiConfig` 使用 Lombok `@Data`，会为包含 `innerApiKey` 的配置类自动生成 `toString()`。一旦该配置对象被日志、异常或调试输出引用，内部 API Key 可能以明文形式泄露。
    修复建议与测试场景
    修复建议

    不要对含密钥字段的配置类使用 `@Data`。改为显式使用 `@Getter`/`@Setter`，并避免生成 `toString()`；或至少为 `innerApiKey` 增加 `@ToString.Exclude`/自定义脱敏输出。

    测试场景

    补充单元测试，构造 `BiApiConfig` 并断言其 `toString()` 结果不包含 `innerApiKey` 的明文值。

    cps-infrastructure/src/main/java/com/speedx/cps/configuration/TransactionConfiguration.java1 Major1 个问题
    MAJORL21-24配置变更
    默认事务管理器被切换为 `JpaTransactionManager`，会让未显式指定 `transactionManager` 的 `@Transactional` 全部改走 JPA 事务语义。当前工程同时依赖 MyBatis 与 JDBC，若现有业务主要通过 `SqlSessionTemplate`/Mapper 执行，事务边界会被这次配置变更整体影响；一旦底层数据源或连接获取方式与 JPA 未完全对齐，容易出现提交/回滚行为与原先不一致的功能异常。
    修复建议与测试场景
    修复建议

    不要直接替换全局默认事务管理器。若只是给 JPA 场景补充事务管理，建议保留原有 `DataSourceTransactionManager` 作为默认 `transactionManager`，新增一个显式命名的 `JpaTransactionManager` 供 JPA 代码按 `@Transactional(transactionManager = "jpaTransactionManager")` 使用，例如：`@Bean(name = "transactionManager") DataSourceTransactionManager ...`，`@Bean(name = "jpaTransactionManager") JpaTransactionManager ...`。

    测试场景

    补充集成测试：同一服务中分别验证 MyBatis 写入成功后抛异常、JPA 写入成功后抛异常，两类场景都应被完整回滚；同时覆盖未显式指定 `transactionManager` 的 `@Transactional` 方法。

    cps-infrastructure/src/main/java/com/speedx/cps/integration/OmsClient.java2 Major2 个问题
    MAJORL34-48,126-128,156-158,191-193并发安全
    `OmsClient` 作为 Spring 单例在 Caffeine 缓存中保存并直接返回可变的 DTO/List 对象。后续调用方一旦对返回的 `List<PickupAvailableResp>` 或 `List<AvailableFacilityResp>` 做原地修改，会污染共享缓存并影响其他线程读取结果，形成跨请求数据串扰和并发一致性风险。
    修复建议与测试场景
    修复建议

    缓存前或返回前改为不可变对象：例如对列表使用 `List.copyOf(...)`/`Collections.unmodifiableList(...)`，必要时对 DTO 也做 defensive copy，避免外部代码修改缓存中的共享实例。

    测试场景
