    cps-infrastructure/src/main/java/com/speedx/cps/integration/OmsClient.java2 Major2 个问题
    MAJORL132安全风险
    新增的异常处理将底层异常原始信息直接拼接到 RRException 返回值中。若上层将该异常消息透传给前端或调用方，可能泄露内部集成细节，如 OMS 地址、请求参数格式、反序列化失败原因或网络栈信息，形成信息泄露面。
    修复建议与测试场景
    修复建议

    对外抛出固定且不含内部细节的错误消息，例如 `request oms listAvailablePickups error`；具体异常原因仅记录在服务端日志中。如需区分错误类型，可定义受控错误码而不是拼接 `e.getMessage()`。

    测试场景

    模拟 OMS 返回畸形 JSON、连接超时或 DNS 失败，验证接口对外响应中不包含底层异常原文、URL、主机名或解析器报错细节。

    MAJORL34-35边界条件
    `getAccountByPickupId` 新增缓存后直接将 `pickupAccountId` 传给 `accountCache.getIfPresent(...)`。Caffeine 不接受 `null` key，调用方一旦传入 `null`，这里会先抛出 `NullPointerException`，比原先的远程调用路径更早失败，属于新增的空值边界回归。
    修复建议与测试场景
    修复建议

    在访问缓存前显式处理空参数，避免把 `null` 传入 Caffeine。例如：`if (pickupAccountId == null) { return null; }`，或按业务约定抛出更明确的 `IllegalArgumentException/RRException`，然后再执行缓存读取和远程查询。

    测试场景

    补充单测：调用 `getAccountByPickupId(null)`，断言不会出现 `NullPointerException`，并校验返回值/异常类型符合业务约定。
