    MAJORL250-257业务逻辑
    新增的 `queryCurrentVirtualDayParentBatchInfo` 方法名和参数包含 `currentDate`，但 SQL 完全没有使用第 2 个参数做日期过滤，只把它用于 `batch_code like ?2`。这会导致查询结果不再受“当天/虚拟日”约束，而是返回该 facility 下最早创建的任意 ACTIVE PARENT 批次，直接偏离 `queryCurrentDayParentBatchInfo` 的语义，可能复用到历史批次。
    修复建议与测试场景
    修复建议

    如果目标是查询“当前虚拟日”的父批次，应显式增加日期条件，例如按 `created_at` 或批次编码中的日期段过滤；若确实想按批次前缀匹配，请将参数名改为 `batchCodePattern` 并补充真正的虚拟日过滤，避免调用方误用。示例：`and DATE_FORMAT(created_at,'%Y%m%d') = ?2`，或 `and batch_code like concat(?2, '%')` 同时保留日期边界条件。

    pos-service/src/main/java/com/miq/posservice/factory/AbstractReportTaskProcessor.java1 Major1 个问题
    MAJORL203-211异常处理
    新增超时分支在 `TimeoutException` 中直接 `return null`，并在后续通用 `catch (Exception e)` 中同样吞掉异常返回 `null`。这会把“异步任务真实失败/超时”伪装成普通空结果，调用方无法区分是业务上无数据还是执行异常，容易导致报表任务被静默标记为部分成功，掩盖下游 HTTP 卡死、线程池饥饿等安全相关可用性问题。该模式也命中 [EXC-001]：catch 宽泛异常后返回 null/默认值。
    修复建议与测试场景
    修复建议

    不要在异常分支返回 `null` 作为失败哨兵值；应显式封装失败状态并向上汇总，或抛出业务异常让任务整体进入失败态。例如返回包含 `success/timeout/error` 的结果对象，或在 `TimeoutException`/`ExecutionException` 中记录失败原因后统一抛出 `RRException`，避免调用方把异常当成正常空值处理。

    测试场景

    补充测试：构造一个 `CompletableFuture` 永不完成或阻塞超过 60 秒，验证调用方不会把返回值当作正常空数据继续处理，而是能明确感知“超时失败”；同时验证普通业务空结果与异常超时结果可被区分。
