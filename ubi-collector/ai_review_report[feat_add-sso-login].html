    backend/server/src/main/java/com/ubi/server/controller/SsoAuthController.java1 Major 1 Minor2 个问题
    MAJORL31-33边界条件
    `callback` 直接将 `ssoLoginService.handleCallback(authToken)` 的返回值传给 `response.sendRedirect(...)`，但这里没有对返回值做非空/非空白校验。若服务层在异常边界下返回 `null` 或空串，Servlet 容器会在重定向时抛出运行时异常，导致 SSO 回调链路失败且错误表现不稳定。
    修复建议与测试场景
    修复建议

    在控制器层增加兜底校验，确保重定向地址非空白后再调用 `sendRedirect`，例如：`String redirectUrl = ssoLoginService.handleCallback(authToken); if (!StringUtils.hasText(redirectUrl)) { throw new BusinessException(502, "SSO 回调结果异常"); } response.sendRedirect(redirectUrl);`。同时建议在 `SsoLoginService.handleCallback` 契约中明确“永不返回空”。

    测试场景

    补充回调接口测试：mock `ssoLoginService.handleCallback(...)` 分别返回 `null`、`""`、`" "`，断言接口返回明确业务异常而不是 Servlet 500/NPE。

    MINORL62-66业务逻辑
    `requireEnabled()` 仅检查 `ssoProperties.isEnabled()`，但 `config()` 已将“启用”定义为 `enabled && loginUrl 有值`。这会导致 `/callback`、`/exchange`、`/redirect-url` 在 `enabled=true` 但关键配置缺失（如 `loginUrl` 为空）时仍继续执行，形成接口对外宣称“已关闭”但后端实际仍放行的状态不一致。
    修复建议与测试场景
    修复建议

    统一 SSO 可用性的判定条件，抽取一个共享方法，例如 `private boolean isSsoAvailable() { return ssoProperties.isEnabled() && StringUtils.hasText(ssoProperties.getLoginUrl()); }`，并让 `config()` 与 `requireEnabled()` 共用该逻辑。若还有其他必填配置，也应一并纳入校验。

    测试场景

    增加配置边界测试：当 `enabled=true` 且 `loginUrl` 为空时，`/config` 返回 `enabled=false`，同时 `/callback`、`/exchange`、`/redirect-url` 也应返回 404/明确不可用错误，而不是继续调用服务层。

    本地用户查询直接按 `username = email.trim()` 精确匹配，但未对邮箱大小写做归一化。SSO 返回的邮箱大小写与本地账号存储大小写不一致时，会把已存在账号误判为不存在，导致 SSO 登录失败。邮箱本身通常应按大小写不敏感处理，这会直接影响核心登录链路。

    修复建议

    在比对前统一归一化邮箱/用户名，例如 `String username = email.trim().toLowerCase(Locale.ROOT);`，并确保本地存储或查询条件也采用同样规则；若历史数据存在混合大小写，建议改为大小写不敏感查询。

    测试场景

    准备本地用户 `alice@example.com`，模拟 SSO 返回 `Alice@Example.com`。当前实现会返回 `account_not_found`；修复后应能正常签发 ticket。

    MAJORL57-70安全风险
    将一次性登录取件码拼接到 URL 查询参数 `?token=` 中返回。该设计会使凭证暴露在浏览器地址栏、历史记录、前端埋点、反向代理访问日志以及后续跳转的 Referer 中；虽然这里传递的是 ticket 而非最终 JWT，但 `exchange()` 可直接兑换出完整登录态，泄露后仍可被他人接管会话。
    修复建议与测试场景
    修复建议

    避免通过查询参数传递可兑换登录态的凭证。优先改为：1）后端回调时通过 `Set-Cookie` 下发短时、HttpOnly、Secure、SameSite 的一次性票据；或 2）将票据放入 URL fragment（`#token=`，不进入 HTTP 请求）并确保前端立即清除；同时为兑换接口增加客户端绑定校验（如 state/nonce/session 绑定）。

    测试场景

    验证 SSO 回调成功后，浏览器地址栏、服务端访问日志、前端监控上报及跳转 Referer 中均不会出现可兑换登录态的 ticket；尝试复制泄露的 ticket 到另一浏览器，不应成功兑换。
