    backend/server/src/main/java/com/ubi/server/controller/SsoAuthController.java1 Major 1 Minor2 个问题
    MAJORL29-40权限与访问控制
    **原因：** 该 SSO 回调入口被 `@Public` 放行为匿名可访问，并直接将请求参数 `authToken` 交给 `ssoLoginService.handleCallback(authToken)` 处理，随后对返回值执行 `response.sendRedirect(redirectUrl)`。从本分片 diff 无法看到对 `authToken` 的一次性消费、有效期、来源绑定或回调态校验（如 state/nonce）的任何约束，因此新增了一个可被外部直接触发的认证入口。 ```java @Public @GetMapping("/callback") public void callback(@RequestParam("authToken") String authToken, HttpServletResponse response) throws IOException { requireEnabled(); String redirectUrl = ssoLoginService.handleCallback(authToken); response.sendRedirect(redirectUrl); } ``` **影响：** 若服务层未严格校验 token 的来源、时效和单次使用语义，攻击者可直接构造或重放 `authToken` 调用该公开入口，导致登录流程被绕过、会话被劫持，或借由未受限的跳转结果扩大攻击面。由于这是认证边界入口，缺少显式防重放/防伪造约束属于高风险认证失效问题。
    修复建议与测试场景
    修复建议

    在控制器或服务层为 `/auth/sso/callback` 增加明确的回调完整性校验：至少校验 `state/nonce` 与发起登录时保存的服务端上下文一致，并确保 `authToken` 具备短时效、单次消费和来源绑定；消费成功后立即失效。若 `handleCallback` 已实现这些校验，请在该入口附近补充显式注释/断言与测试，避免公开入口被误用。示例：在回调中增加 `state` 参数并校验服务端票据，再调用登录逻辑。

    测试场景

    1. 使用同一个 `authToken` 连续调用 `/auth/sso/callback` 两次，第二次应失败；2. 使用伪造或过期 `authToken` 调用应失败；3. 使用不匹配的 `state/nonce` 调用应失败；4. 验证未登录匿名用户无法仅凭重放历史回调完成登录。

    MINORL44-47业务逻辑
    `/auth/sso/config` 用 `enabled && hasText(loginUrl)` 判断前端是否展示 SSO，但其余实际可用接口（`/callback`、`/exchange`、`/redirect-url`）的开关校验只检查 `ssoProperties.isEnabled()`。当配置为 enabled=true 但 `loginUrl` 为空时，`config` 会返回 false，而其他接口仍会继续执行，导致前后端对“SSO 是否可用”的判断不一致。
    修复建议与测试场景
    修复建议

    统一可用性判定逻辑，抽出一个共享方法同时用于 `config()` 和 `requireEnabled()`，例如：`private boolean isSsoAvailable() { return ssoProperties.isEnabled() && StringUtils.hasText(ssoProperties.getLoginUrl()); }`；然后 `config()` 返回该值，`requireEnabled()` 也基于该方法决定是否拒绝请求。

    测试场景

    配置 `enabled=true` 且 `loginUrl` 为空：验证 `/auth/sso/config` 与 `/auth/sso/redirect-url`、`/auth/sso/callback`、`/auth/sso/exchange` 的行为一致，避免一个返回不可用而其他接口仍可调用。

    MAJORL54-59安全风险
    SSO 登录成功后将一次性 ticket 直接拼接到 URL 查询串 `?token=` 中返回。该票据虽为短期一次性凭证，但仍属于登录凭证，放在 URL query 会进入浏览器历史、代理/网关访问日志、Referer 头及前端埋点，造成凭证泄露后被他人抢先兑换登录。问题代码： ```java return ssoProperties.getFrontendBaseUrl() + ssoProperties.getLoginPagePath() + "?token=" + ticket; ```
    修复建议与测试场景
    修复建议

    避免通过 URL 查询参数传递登录凭证。优先改为：1）后端回调时通过 `HttpOnly + Secure + SameSite` Cookie 下发一次性票据；或 2）将票据放入 URL fragment（`#token=`，不经 HTTP 传输）并由前端立即 POST 兑换；或 3）后端直接完成会话建立而非暴露中转票据。若必须保留票据机制，至少确保前端落地页关闭 Referer 泄露并在兑换后立即清理地址栏。

    测试场景

    验证 SSO 回调成功后，浏览器地址栏、访问日志、前端埋点及跳转到第三方站点时不会泄露 ticket；尝试复制成功登录 URL 到另一浏览器，确认无法因 URL 泄露完成未授权登录。
