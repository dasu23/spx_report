    backend/server/src/main/java/com/ubi/server/controller/SsoAuthController.java2 Major 1 Minor3 个问题
    MAJORL29-40权限与访问控制
    **原因：** 该新增公开入口直接接收外部提供的 `authToken` 并调用 `ssoLoginService.handleCallback(authToken)`，随后对返回的 `redirectUrl` 执行 `response.sendRedirect(redirectUrl)`，但在本分片 diff 中看不到对 `redirectUrl` 的来源域名、协议或白名单校验。若下游服务/配置被污染，或 `handleCallback` 可被构造返回任意地址，则该公开接口会成为开放重定向入口。 ```java @Public @GetMapping("/callback") public void callback(@RequestParam("authToken") String authToken, HttpServletResponse response) throws IOException { requireEnabled(); String redirectUrl = ssoLoginService.handleCallback(authToken); response.sendRedirect(redirectUrl); } ``` **影响：** 未认证用户可利用该接口将浏览器重定向到攻击者控制站点，造成钓鱼、令牌窃取链路放大，且该路径位于受信任业务域名下，容易绕过用户警惕与部分安全策略。
    修复建议与测试场景
    修复建议

    在控制器或 `SsoLoginService.handleCallback(...)` 返回前增加严格的重定向目标校验：仅允许预配置前端域名/相对路径，拒绝非 HTTPS、非白名单 host、包含 CRLF 的值。示例： ```java URI uri = URI.create(redirectUrl); if (!"https".equalsIgnoreCase(uri.getScheme()) || !allowedHosts.contains(uri.getHost())) { throw BusinessException.badRequest("invalid redirect url"); } response.sendRedirect(uri.toString()); ```

    测试场景

    构造 `handleCallback` 返回 `https://evil.example/phish`、`javascript:alert(1)`、`//evil.example`、包含 `%0d%0a` 的地址，验证 `/auth/sso/callback` 均拒绝跳转；对白名单内 HTTPS 地址应允许。

    MAJORL31-33边界条件
    `callback` 直接将 `ssoLoginService.handleCallback(authToken)` 的返回值传给 `response.sendRedirect(...)`，但这里没有对返回值做非空/非空白校验。若服务层在某些异常分支返回 `null` 或空串，Servlet 容器会在重定向时抛出异常，导致 SSO 登录回调走到 500，而不是可控的业务错误响应。
    修复建议与测试场景
    修复建议

    在控制器层增加兜底校验，确保重定向地址有效后再调用 `sendRedirect`，例如：`String redirectUrl = ssoLoginService.handleCallback(authToken); if (!StringUtils.hasText(redirectUrl)) { throw new BusinessException(502, "SSO 登录回调结果无效"); } response.sendRedirect(redirectUrl);`。同时补充服务层契约，明确该方法不得返回空值。

    测试场景

    模拟 `ssoLoginService.handleCallback(...)` 返回 `null` 和 `""` 两种情况，验证接口返回受控错误而不是 500/Servlet 异常。

    MINORL63-67业务逻辑
    `requireEnabled()` 仅检查 `ssoProperties.isEnabled()`，但 `/config` 接口判定可用性时还额外要求 `loginUrl` 非空。这样会出现配置不完整时 `/config` 返回 `enabled=false`，但 `/callback`、`/exchange`、`/redirect-url` 仍继续执行，最终在更深层因缺少必要配置失败，接口行为前后不一致。
    修复建议与测试场景
    修复建议

    统一 SSO 可用性的判定条件。可将 `requireEnabled()` 改为同时校验关键配置，例如 `enabled && hasText(loginUrl)`；如果不同接口依赖的配置项不同，建议拆分为更明确的校验方法并在各接口按需调用。

    测试场景

    配置 `sso.enabled=true` 但 `sso.loginUrl` 为空，分别调用 `/auth/sso/config` 与 `/auth/sso/redirect-url`，验证两者对“SSO 是否可用”的判断一致。

    backend/server/src/main/java/com/ubi/server/service/SsoLoginService.java1 Major1 个问题
    MAJORL46-49安全风险
    SSO 登录仅以中心 SSO 返回的 `email` 直接匹配本地 `username` 完成登录，未对返回的 `state`、`party` 等身份上下文做任何校验，也未校验邮箱域、账号启用状态或是否允许 SSO 登录。这样会把“外部身份提供方中的邮箱字符串”等同于“本地账号唯一身份”，一旦中心 SSO 侧 claim 配置错误、账号复用、邮箱别名/大小写差异或外部身份源被错误映射，可能导致错误账号绑定并签发本地 JWT。

    修复建议

    在本地用户表中建立显式的 SSO 绑定关系（如 `sso_subject` / `issuer` / `email_verified` / `sso_enabled`），登录时按稳定唯一标识匹配，而不是仅按 email/username 字符串匹配；至少增加允许域名校验、账号状态校验和 SSO 开关校验。建议补充类似：先校验 `state/party` 是否符合预期，再按专用 SSO 绑定字段查询用户，最后校验 `user` 是否启用。

    测试场景

    构造两个本地账号与同一邮箱别名/大小写变体场景，或模拟 SSO 返回非预期 `party/state` 但 email 命中本地用户，验证系统不会错误签发 JWT。

    frontend/apps/dashboard/src/pages/login/login-page.tsx1 Major1 个问题
    MAJORL31-42安全风险
    登录页在组件加载时只要 URL 中存在 `token` 参数，就会自动调用 `/auth/sso/exchange` 完成登录，且未校验该 token 是否来自当前站点刚发起的 SSO 流程，也未绑定任何前端侧 state/nonce。这样会形成登录 CSRF / 会话注入风险：攻击者若拿到自己账户对应的有效一次性票据，可诱导受害者访问 `/login?token=...`，受害者浏览器会被自动登录到攻击者账户。
