    drivermgmt-domain/src/main/java/io/speedx/drivermgmt/domain/biz/RouteMgmtBizService.java2 Major2 个问题
    MAJORL223-225边界条件
    新增日志任务的构建条件只判断了 req.getDriverId() != null，但实际传入 buildDispatchOperationLogTasks 的是 driverPo.get()。如果 driverId 非空但 driverPo 为空（例如司机不存在、被过滤或上游校验遗漏），这里会直接抛出 NoSuchElementException，导致派送流程失败。
    修复建议与测试场景
    修复建议

    在构建任务前同时校验 driverPo 是否存在，例如改为 `req.getDriverId() != null && driverPo.isPresent()`；若 driverId 非空但未查到司机，应显式抛出业务异常并给出可读错误信息，而不是调用 Optional.get()。

    测试场景

    提交一个带 driverId 但该司机不存在/不可用的派送请求，验证接口返回明确业务错误，不出现 500 或 NoSuchElementException。

    MAJORL257-261性能隐患
    在遍历 routes 的循环内逐条调用 itemRepository.findByRouteId(route.getId())，会形成典型的 N+1 查询。批量派送/指派司机接口一次处理多条路线时，数据库查询次数会随路线数线性增长，容易导致接口耗时明显上升。
    修复建议与测试场景
    修复建议

    改为一次性按 routeId 集合批量查询 RouteItem，再按 routeId 分组后构建任务。例如新增 repository 方法 findByRouteIdIn(routeIds)，然后使用 Map<Long, List<RouteItem>> grouped = items.stream().collect(Collectors.groupingBy(RouteItem::getRouteId)); 在循环中直接读取 grouped.get(route.getId())。

    测试场景

    构造一次请求包含 100 条 route、每条 20 个包裹，比较变更前后 SQL 次数与接口耗时，确认不会出现 100 次额外的 route_item 查询。

    drivermgmt-domain/src/main/java/io/speedx/drivermgmt/domain/service/PickupService.java2 Major2 个问题
    MAJORL1000-1001权限与访问控制
    **原因：** 新增派单日志任务时直接把 `principal.getOperator()` 写入操作日志，但未对该字段做非空兜底。若当前登录主体没有 operator 标识，后台任务会以空 operator 调用 `operationLogClient.addLog(...)`，导致审计主体缺失。对于 `/pickup/routes/dispatch` 这类会变更路线归属的接口，审计身份缺失会削弱事后追责与越权排查能力。 ```java AddOperationLog opLog = AddOperationLog.builder() .trackingNumber(parcel.getTrackingNo()) .operation(OperationType.PICKUP_DISPATCH) .operator(principal.getOperator()) .remark(remark) .businessData(businessData) .build(); ``` **影响：** 当具备该接口访问能力的主体触发派单时，系统可能产生“无操作者”的关键操作日志。发生误派单、恶意改派或越权操作后，无法可靠定位实际操作者，削弱审计与访问控制闭环。
    修复建议与测试场景
    修复建议

    对操作日志中的 operator 做强制非空校验或兜底，至少回退到用户 ID；若两者都为空则拒绝创建任务并抛出异常。例如： ```java String operator = StringUtils.defaultIfEmpty(principal.getOperator(), String.valueOf(principal.getId())); Assert.hasText(operator, "operator is required"); opLog.setOperator(operator); ``` 同时为 `/pickup/routes/dispatch` 增加测试，覆盖 operator 为空场景。

    测试场景
