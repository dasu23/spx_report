critical123-124安全风险新登录流程保存的 Session 没有写入 `data`，这里在 `data == null` 时却回退成 `new LoginUserBo().setId(session.getId())`。`session.id` 是会话表主键，不是 `driverId`，因此通过 `/login` 拿到的 token 在鉴权后会被映射成错误的用户身份；当会话表自增 id 恰好落到其他司机 id 上时，会直接访问到他人的司机数据。不要用 `session.id` 回填登录身份。最小修复是改为 `new LoginUserBo().setId(session.getDriverId())`；更稳妥的是在创建 Session 时同步写入序列化后的 `LoginUserBo`，并为新旧 token 路径补充统一的会话恢复逻辑。使用一个 `driverId` 与 `session` 自增 id 不相等的账号调用 `/login`，随后携带返回 token 访问 `/account/currentDriverInfo` 或任一依赖 `AuthUtil.getPrincipalId()` 的接口，验证不会读到其他司机的数据。
