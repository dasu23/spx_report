    MAJORL8-10数据一致性
    数据清洗脚本把 `total_distance` 中的英里数直接写入 `route.distance`，但旧脚本会先乘以 `1609.344` 做单位换算。若 `route.distance` 的既有语义仍是米，这次变更会把已清洗数据缩小约 1609 倍，导致距离统计、阈值判断或后续依赖该字段的业务结果异常。
    修复建议与测试场景
    修复建议

    确认 `route.distance` 的统一单位；如果下游仍按米处理，应恢复换算逻辑，例如：`SET distance = CAST(TRIM(REPLACE(total_distance, 'miles', '')) AS DECIMAL(20, 4)) * 1609.344`。如果确实要改为存英里，需要同步修改所有读取方与文档，并补充一次全量兼容性校验。

    测试场景

    构造 `total_distance='10 miles'` 的 route 记录，执行清洗后校验 `distance` 是否等于系统约定单位下的期望值；同时验证依赖 `route.distance` 的统计/展示逻辑在清洗前后结果一致。

    drivermgmt-infrastructure/src/main/java/io/speedx/drivermgmt/integration/AzureBlobUrlSigner.java2 Major 1 Minor3 个问题
    MAJORL138-143边界条件
    `resolveContainerClient()` 对非默认 container 直接执行 `crossContainerClients.computeIfAbsent(container, serviceClient::getBlobContainerClient)`，但 `serviceClient` 只在 `init()` 里且仅 `containerClient != null` 时赋值。若该 Bean 在未完成 `init()` 的时机被调用，或纯逻辑单测里传入了 `null containerClient` 后走到跨 container 分支，这里会直接触发 `NullPointerException`，导致签名功能失败。
    修复建议与测试场景
    修复建议

    在 `resolveContainerClient()` 中显式校验 `serviceClient` 是否已初始化，并给出可控降级或明确异常，例如：`if (serviceClient == null) { throw new IllegalStateException("BlobServiceClient not initialized"); }`；如果需要兼容测试场景，则应在 `sign()` 进入跨 container 分支前直接回退为原值返回，避免对 `null` client 做方法引用。

    测试场景

    补充用例：1) `containerClient == null` 时调用 `sign("/other-container/a.jpg", ...)`；2) 模拟未执行 `init()` 直接调用跨 container 签名，断言不会出现 NPE，而是返回原值或抛出明确的业务异常。

    MAJORL140资源管理
    `crossContainerClients` 是进程级无界缓存，key 直接来自新增解析逻辑中的 `/{container}/{blob}` 输入。只要调用方持续传入不同的 container 名，就会在 `computeIfAbsent` 中永久保留新的 `BlobContainerClient`，造成堆内存持续增长，最终可被利用为应用层 DoS。该风险由本次改动新增，因为旧实现只使用固定注入的 `containerClient`，不会按用户输入扩容缓存。
    修复建议与测试场景
    修复建议

    不要对外部输入驱动的 container 名做无界缓存。可改为：1. 仅允许白名单中的有限 container；或 2. 使用有上限的缓存并在 miss 时拒绝异常 container；至少需要在写入缓存前校验 container 名格式和数量上限。

    测试场景

    构造大量不同 container 名的同 host URL（如 1 万个 `https://<account-host>/c{n}/a.txt`）循环调用 `sign()`，断言进程堆占用和 `crossContainerClients` 大小持续增长；修复后应在达到白名单/容量上限时拒绝新增而不是无限扩容。

    MINORL194-203业务逻辑
    内部 URL 判定从“host+container 匹配”放宽成了“仅 host 匹配”。这样同域名下只要路径形态异常但首段不是合法 container，也会被当作内部地址继续处理，随后在 `sign()` 中走 fallback 按默认 container 重新签名，可能把原本应原样透传的 URL 错误改写成默认容器的 SAS 地址。
    修复建议与测试场景
    修复建议

    在 `toBlobPath()` 或 `sign()` 中增加路径形态校验：只有成功解析出 `/{container}/{blob}` 时才按内部资源处理，否则对绝对 URL 保持 passthrough。示例：`if (isAbsolute && parseContainerAndBlob(relative) == null) { return fileOrUrl; }`。

    测试场景

    补充用例：`https://<self-host>/foo`、`https://<self-host>//bar`、`https://<self-host>/reports` 等不满足 `/{container}/{blob}` 的同 host URL，应保持原样返回，不应被签成默认 container 的 blob URL。
