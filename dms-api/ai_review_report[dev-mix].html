    drivermgmt-infrastructure/src/main/java/io/speedx/drivermgmt/infrastructure/db/po/Parcel.java1 Major 1 Minor2 个问题
    MAJORL14-16API契约
    `Parcel` 实体未显式声明 `@Table`，但 `@SQLDelete` 硬编码了 `update Parcel ...`。在 Spring Boot 2.5 默认 Hibernate 命名策略下，实体表名通常会落到 `parcel`，这会导致逻辑删除 SQL 与真实表名不一致，在区分大小写的 MySQL/部署环境中删除操作直接失败。
    修复建议与测试场景
    修复建议

    为实体补充显式表名，并让 `@SQLDelete` 与其保持一致，例如：`@Table(name = "parcel")` 与 `@SQLDelete(sql = "update parcel set deleted = id where id = ?")`。同时补一条集成测试，覆盖逻辑删除后查询被 `@Where` 过滤的场景。

    测试场景

    在区分大小写表名的 MySQL 环境创建 `parcel` 表，执行 `parcelRepository.deleteById(...)`，验证不会报表不存在/更新 0 行，且后续普通查询无法再查到该记录。

    MINORL19-21向后兼容
    `status` 和 `trackingEnd` 是枚举字段，但未标注 `@Enumerated(EnumType.STRING)`。JPA 默认会按 ordinal 持久化，后续只要调整枚举顺序或中间插入新值，历史数据就会被反序列化成错误状态。
    修复建议与测试场景
    修复建议

    为两个枚举字段显式声明字符串持久化：`@Enumerated(EnumType.STRING) private Status status;`、`@Enumerated(EnumType.STRING) private TrackingEnd trackingEnd;`，避免枚举演进带来的存量数据错读。

    测试场景

    先保存一条 `status=DELIVERED` 的记录，再模拟在枚举中间新增常量后读取，确认使用 `EnumType.STRING` 时历史数据仍能正确映射。
