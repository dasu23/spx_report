major173-176业务逻辑冲突检查用的是 `!req.getRouteIds().contains(dp.getId())`，这里比较的是 dispatch 主键，而 `req.getRouteIds()` 存的是 routeId。结果是重派同一条路线给同一司机时，当前路线的旧 dispatch 也不会被排除，系统会误判成“司机当天已有其他待处理路线”，从而错误返回 202 提示。这里应比较路线 ID：`filter(dp -> !req.getRouteIds().contains(dp.getRouteId()))`。同时建议补充单元测试覆盖“同一路线重派/改派司机”场景，避免 routeId 和 dispatchId 混用。先给司机 A 创建路线 R 的 dispatch，再调用当前新接口重派同一条路线 R 给司机 A；预期应允许覆盖，当前实现会误报冲突提示。
major182-209数据一致性新实现会把目标路线的所有 dispatch 记录直接 `delete` 后再插入新的 `WAITING_ACCEPT` 记录，包含已经 `ACCEPTED` 的历史记录也会被删掉。旧实现是把旧记录改为 `REJECTED` 并保留 remark。现在的做法会丢失派单/接受历史，影响审计、追溯以及基于 dispatch 历史的统计。不要物理删除全部 dispatch。应保留历史记录，至少将旧的有效记录更新为 `REJECTED`（或单独的 redispatch 状态）并记录操作人/原因，再新增一条新的 dispatch。若确实要清理，只能清理当前未接受的临时记录。准备一条已被司机接受的路线，执行重新派单后检查 `route_dispatch` 表。预期应保留旧记录并新增新记录；当前实现会直接删除旧记录。
