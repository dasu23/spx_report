    MAJORL13安全风险
    将认证上下文对象改为 `@Data` 会新增自动生成的 `toString()`/`equals()`/`hashCode()`。`LoginUserBo` 包含 `operator`、`roleCodes`、`permissions` 等安全敏感身份信息，一旦被日志、异常包装或调试输出隐式调用 `toString()`，会扩大认证信息泄露面。此前仅生成 getter/setter，不会默认暴露整对象字符串表示。
    修复建议与测试场景
    修复建议

    不要在认证/会话类上使用 `@Data`。改回显式的 `@Getter`/`@Setter`，或至少使用 `@ToString(onlyExplicitlyIncluded = true)` 并显式排除 `roleCodes`、`permissions`、`operator` 等字段，同时按需手写 `equals`/`hashCode`。

    测试场景

    增加测试覆盖日志/异常序列化场景：对 `LoginUserBo` 执行 `toString()` 或作为日志参数输出时，断言返回内容不包含 `operator`、`roleCodes`、`permissions` 等字段。

    MAJORL14API契约
    新增 `@Accessors(chain = true)` 会把 Lombok 生成的 setter 从标准 JavaBean 签名 `void setXxx(...)` 改成链式返回 `LoginUserBo setXxx(...)`。`LoginUserBo` 是跨层传递的 BO，这类改动容易让依赖标准 Bean 规范的绑定/拷贝组件（如部分 Spring BeanUtils、MapStruct 配置、老式反射工具）无法识别可写属性，出现字段未赋值或对象转换失败。
    修复建议与测试场景
    修复建议

    不要在对外 DTO/BO 上直接启用链式 setter。保留标准 setter，例如改回 `@Setter @Getter`，或显式为需要兼容的字段声明 `public void setFacilityCode(String facilityCode)` 这类 void setter，仅在内部构造对象时使用 builder/fluent API。

    测试场景

    补充一个兼容性测试：对 `LoginUserBo` 执行 Jackson 反序列化、`BeanUtils.copyProperties` 或项目内实际使用的对象映射逻辑，断言 `facilityCode/operator/id/roleCodes/permissions` 都能正确写入。
