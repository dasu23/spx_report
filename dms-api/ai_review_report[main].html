    MAJORL388-399并发安全
    `rejectRoute` 先在事务外读取并校验 `Route` / `RouteDispatch` 状态，再在 `transactionTemplate.execute(...)` 中分别保存两条记录。这里是典型的 check-then-act 竞态窗口：如果司机重复点击，或并发发生“接受/拒绝”操作，两个请求都可能基于同一份 `WAITING_ACCEPT` 旧状态通过校验，最终把 `route.status` 和 `routeDispatch.status` 覆盖为相互冲突的结果，造成派单状态不一致。该问题会影响业务状态完整性，属于安全维度中的并发一致性风险。
    修复建议与测试场景
    修复建议

    将状态校验和更新放入同一个数据库事务内，并结合行级锁或乐观锁版本字段保证原子性。例如对 `RouteDispatch` / `Route` 使用 `select ... for update`，或在更新时增加 `where status = WAITING_ACCEPT` 条件并检查受影响行数，确保只有一个请求能成功完成拒绝。
