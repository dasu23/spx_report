SeverityLineCategoryDescriptionSuggestionTest Scenario
major173-176业务逻辑司机冲突校验里用 `req.getRouteIds().contains(dp.getId())` 排除当前路线，但 `dp.getId()` 是 dispatch 主键，不是 routeId。结果是同一条路线再次派给同一司机时也不会被正确排除，接口会误判为“司机当天已有其他待处理路线”，必须依赖 `suppressWarnings` 才能继续。这里应比较 `dp.getRouteId()` 而不是 `dp.getId()`，例如改成 `filter(dp -> !req.getRouteIds().contains(dp.getRouteId()))`。同时补一个回归测试覆盖“同一路线重复提交派单请求不应触发冲突提示”。准备一条已存在 `WAITING_ACCEPT` 指派的路线，使用同一 `routeId`、同一司机、同一天再次调用派单接口；期望返回成功或幂等，而不是 202 warning。
major182-210数据一致性`routeDispatch` 现在会把目标路线的全部 `route_dispatch` 记录直接删除后再重建，但没有再拦截已被司机接受的指派。这样运营侧重新派单时，会把 `ACCEPTED` 记录也一起删掉，导致已接受路线的指派历史和当前归属被覆盖，司机端与后台状态可能发生错配。在删除旧指派前先校验是否存在 `ACCEPTED` 记录，存在时直接拒绝重派，或仅允许显式的撤销流程处理。例如先筛出 `acceptedDispatches = allDsipatches.stream().filter(dp -> dp.getStatus() == DispatchStatus.ACCEPTED)`，非空时抛出业务异常；若确需重派，也应保留历史记录并将旧记录逻辑失效，而不是物理删除。构造一条 `RouteStatus.ASSIGNED` 且存在 `RouteDispatch.status=ACCEPTED` 的路线，调用 `/route/dispatch` 重派后，断言接口应失败且原 `ACCEPTED` 记录仍存在。
