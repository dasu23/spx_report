MAJORL10-14配置变更

修复建议

在建索引前先全量检查并清理 `(route_id, deleted)` 重复数据，不要只处理 `REJECTED`；必要时先执行去重 SQL，再创建唯一索引。

迁移脚本只把 `REJECTED` 记录的 `deleted` 改成 `id`，随后立即给 `route_dispatch(route_id, deleted)` 建唯一索引。历史上同一路线若存在多条未拒绝的旧 dispatch 且 `deleted` 仍为默认值，会在建索引时直接失败，阻塞上线。
修复建议与测试场景
修复建议

在建唯一索引前先清理/回填同一路线的历史重复有效记录，只保留最新一条 `deleted=0`，其余记录更新为唯一的逻辑删除值，再执行 `ADD UNIQUE INDEX`。

测试场景

准备同一 `route_id` 下两条 `deleted=0` 的历史 dispatch 数据，执行迁移脚本，验证清理脚本后索引可成功创建。

MAJORL13-15数据一致性
脚本直接为 `route_dispatch(route_id, deleted)` 增加唯一索引，但前面只把 `REJECTED` 数据做了逻辑删除，没有清理同一 `route_id` 下现存的多条有效记录。历史数据一旦存在重复有效 dispatch，这条 DDL 会在发布时直接失败，导致迁移中断。
修复建议与测试场景
修复建议

在建索引前补充去重脚本，先按 `route_id` 找出 `deleted=0/null` 的重复记录并保留一条，再执行唯一索引创建；或分两步发布，先治理存量数据再加约束。

MAJORL14-15数据一致性
新增的唯一索引是 `(route_id, deleted)`，但前面的脚本只把 `REJECTED` 记录更新为 `deleted = id`，其余有效记录大概率仍为 `NULL`。在 MySQL 中唯一索引允许多条 `NULL`，因此无法阻止同一 `route_id` 出现多条“未删除”有效 dispatch，约束目标落空。
