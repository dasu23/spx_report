    backend/src/main/java/com/htsre/controller/AuthController.java1 Critical1 个问题
    CRITICALL56-63权限与访问控制
    **原因：** 刷新接口新增逻辑直接对客户端提交的 JWT 做 `Base64` 解码并读取 `email`/`role`，随后调用 `authService.refresh(null, email, role)`，显式绕过了原本基于旧 token 的服务端校验。这里既没有签名校验，也没有校验 token 是否由本系统签发，攻击者只需伪造任意 payload 即可把自己声明为任意邮箱和角色。 ```java String[] parts = token.split("\\."); ... Map<String, Object> claims = new ObjectMapper().readValue(payloadJson, Map.class); String email = String.valueOf(claims.getOrDefault("email", claims.getOrDefault("sub", "unknown"))); String role = String.valueOf(claims.getOrDefault("role", "USER")); LoginResponse response = authService.refresh(null, email, role); ``` **影响：** 若该接口用于签发新访问令牌，则未持有有效签名 token 的攻击者也可能通过构造伪造 JWT 获取任意身份/角色的新 token，形成认证绕过与权限提升，风险为高危。 --- 刷新接口现在仅通过 Base64 解码旧 token 的 payload 提取 email/role，然后调用 `authService.refresh(null, email, role)`，不再把原 token 传入服务层做校验。这样即使请求方提交的是伪造的 JWT 字符串，只要 payload 中带有任意 email/role，也可能被当作合法身份刷新出新 token，导致认证边界失效。
    修复建议与测试场景
    修复建议

    刷新接口必须仅接受并验证由服务端签发的 refresh token/expired access token，禁止从未验签的 payload 直接信任 `email` 和 `role`。应恢复/增加服务端验签与过期、签发者、用途校验，例如先调用 `jwtService.parseClaims(token)` 或在 `authService.refresh(token, ...)` 内完成完整校验，再从已验证 claims 中提取身份信息；若校验失败统一返回 401。

    测试场景

    构造一个签名为空或伪造签名的 JWT，payload 中填入 `{"email":"admin@example.com","role":"SUPER_ADMIN"}`，调用刷新接口；期望返回 401，且绝不能签发新的管理员 token。
