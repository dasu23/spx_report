    backend/src/main/java/com/htsre/controller/AuthController.java1 Critical 1 Major2 个问题
    CRITICALL59-61权限与访问控制
    **原因：** 刷新接口新增逻辑直接对客户端提交的 JWT 进行 `Base64` 解码并从未校验签名、签发者、过期状态或服务端绑定关系，就把其中的 `email` 与 `role` 当作可信身份使用，并调用 `authService.refresh(null, email, role)` 生成新登录态。这样攻击者可自行构造任意 payload（例如伪造 `SUPER_ADMIN` 角色），即使原 token 从未由系统签发，也可能换取新的有效凭证。 ```java Map<String, Object> claims = new ObjectMapper().readValue(payloadJson, Map.class); String email = String.valueOf(claims.getOrDefault("email", claims.getOrDefault("sub", "unknown"))); String role = String.valueOf(claims.getOrDefault("role", "USER")); log.info("[AUTH] Refresh: email={}", email); LoginResponse response = authService.refresh(null, email, role); ``` **影响：** 这是典型的认证绕过/权限提升风险。未持有合法签名 token 的攻击者可伪造任意用户身份和角色刷新出新 token，进一步访问受保护接口，导致后台管理能力被接管或敏感数据被越权访问。 --- **原因：** 刷新接口不再调用上游 Nimbus 刷新能力，也未在该方法内校验 `currentToken` 的有效性、过期状态或签发来源，而是直接基于传入的 `email`、`role` 重新签发本地 JWT： ```java String token = jwtService.generateToken(email, email, role); return new LoginResponse(token, email, email, role, null, null); ``` 如果控制器层把 `email`/`role` 来自请求参数、请求体或未经严格校验的上下文传入，这里会形成“凭声明刷新”的能力，导致令牌续签绕过真实会话状态。 **影响：** 认证过期控制被削弱：已失效、被吊销或来源不可信的会话仍可能获得新 token；在最坏情况下，攻击者可借构造的身份字段刷新出更高权限令牌，造成长期会话维持或权限提升。
    修复建议与测试场景
    修复建议

    刷新接口必须只接受并验证由服务端签发的 refresh/access token，严禁从未验签的 payload 直接信任 `email/role`。应调用统一的 JWT 校验逻辑（验签、过期、issuer、token type），并从校验后的 claims 中取身份；同时不要把 `null` 传给刷新服务。 ```java @PostMapping("/refresh") public ResponseEntity<?> refresh(@RequestHeader("Authorization") String authHeader) { String token = extractBearer(authHeader); Claims claims = jwtService.parseAndValidate(token); // 验签+校验过期/类型 String email = claims.getSubject(); String role = claims.get("role", String.class); LoginResponse response = authService.refresh(token, email, role); return ResponseEntity.ok(response); } ``` 如系统采用独立 refresh token，应额外校验 token type=refresh，并结合服务端存储/黑名单机制防重放。

    测试场景

    1. 构造一个未签名或使用错误密钥签名的 JWT，payload 中填入 `email=admin@example.com, role=SUPER_ADMIN`，调用 `POST /refresh`，预期应返回 401/403，绝不能签发新 token。2. 使用已过期 access token 调用刷新接口，若不是合法 refresh token，预期拒绝。3. 篡改合法 token 的 role 后再次调用，预期拒绝。
