critical181-203安全风险`validateCreate` 删除了“container 必须属于当前登录仓”的校验。现在只要知道 `containerNum`，任意仓库用户都可以基于其他仓库的 container 创建 dispatch，属于跨仓越权操作，直接突破了原有的仓库隔离边界。恢复仓库归属校验，并在进入创建流程前同时校验当前用户仓库与 `posContainer.facilityCode` 一致，例如：`if (!Objects.equals(LoginUserUtil.getCurFacilityCode(), posContainer.getFacilityCode())) { throw new RRException("Container must belong current login facility"); }`。同时补一条回归测试，验证 A 仓用户不能用 B 仓 container 创建 dispatch。构造一个 `facilityCode=B` 的 `PosContainer`，在 `LoginUserUtil.getCurFacilityCode()` 返回 `A` 时调用 `create`，期望抛出 `RRException`，且 `dispatchV2Dao.save` 不被调用。
major340-347数据一致性扫描校验现在只检查 `findByDispatchCodeAndContainerNum(dispatchCode, containerNum)`，意味着同一个 container 只要不在“当前 dispatch”里，就可以再次绑定到别的 dispatch。这里同时还删掉了原来的跨 dispatch 占用校验和 `DISPATCHING` 状态拦截，导致一个 container 可以并发存在于多个未完成 dispatch 中，后续 remove/close 会出现状态和关系不一致。在扫描前恢复“全局占用”检查，而不是只查当前 dispatch。可以新增 `findFirstByContainerNum`/`existsByContainerNum`，若已绑定到其他 dispatch 则返回 EXCEPTION；同时恢复 `DISPATCHING`/`DISPATCHED` 状态校验，确保 container 在任一时刻只归属一个进行中的 dispatch。先将同一个 `containerNum` 绑定到 `dispatch-A`，再扫描到 `dispatch-B`。期望返回 EXCEPTION（或明确的 already bound 错误），且数据库中该 `containerNum` 仍只存在一条 dispatch 绑定记录。
