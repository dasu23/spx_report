critical181-203,294-320安全风险`validateCreate` 和 `validateScan` 已移除 `PosContainer.facilityCode` 与当前登录 facility 的归属校验。这样任意仓库用户只要知道容器号，就可以基于其他仓库的 container 创建/扫描 dispatch，并在后续流程里修改该 container 的 dispatch 关系和状态，属于跨仓越权操作。在创建和扫描两条路径都恢复归属校验，例如在取到 `posContainer` 后立即执行 `if (!Objects.equals(LoginUserUtil.getCurFacilityCode(), posContainer.getFacilityCode())) { throw new RRException("Container must belong current login facility"); }`。同时补一条接口测试，验证 A 仓用户不能操作 B 仓 container。使用 `facilityCode=LAX` 的登录态，传入 `facilityCode=JFK` 的 containerNum 调用 `create` 和 `scan`；期望都返回越权/业务异常，且 `pos_dispatch_container_v2`、`pos_container.status` 无任何变更。
major340-347数据一致性扫描校验现在只查 `findByDispatchCodeAndContainerNum`，并且原本针对 `DISPATCHING` 状态的拦截已被删除。结果是同一个 container 只要不在“当前 dispatch”里，就能再次扫入其他 pending dispatch；第一次扫描后状态即使已经变成 `DISPATCHING`，第二次扫描仍会通过，导致一个 container 同时绑定多个 dispatch。恢复全局唯一性校验，而不是仅按当前 dispatch 去重。至少在扫描前增加 `if (PosContainerStatusEnum.DISPATCHING.name().equals(posContainer.getStatus()) || PosContainerStatusEnum.DISPATCHED.name().equals(posContainer.getStatus()))` 的拦截，并增加一个按 `container_num` 全局查询/唯一索引，确保同一时刻只能属于一个有效 dispatch。先将同一个 container 扫入 dispatch A，再在未 remove/submit 的情况下扫入 dispatch B；期望第二次扫描失败，并且数据库中该 `container_num` 只存在一条有效 dispatch 绑定记录。
