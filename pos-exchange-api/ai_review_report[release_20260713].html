    MAJORL179-183业务逻辑
    子查询 `SELECT COUNT(DISTINCT receptacle_code) AS total_cnt FROM pos_shipment_parcel WHERE shipment_id = :shipmentId` 在 shipment 没有关联 parcel/receptacle 明细时会返回 `total_cnt = 0`，而后续条件使用 `COALESCE(ps.received_receptacle_qty, 0) + 1 >= receptacle_totals.total_cnt`。这意味着首次调用时 `1 >= 0` 恒成立，会把 shipment 直接置为 `RECEIVED` 并写入 `receiving_at/receiving_by`。攻击者或异常调用路径可借此对缺少明细的数据直接推进收货完成状态，属于状态机边界缺陷。
    修复建议与测试场景
    修复建议

    在 SQL 中显式限制 `total_cnt > 0` 后才允许状态推进，例如在 CASE 条件中增加 `receptacle_totals.total_cnt > 0`；或在服务层调用前校验 shipment 必须存在至少一个有效 receptacle/parcel 明细，否则拒绝处理并记录异常。

    测试场景

    准备一个 `shipmentId` 存在但 `pos_shipment_parcel` 无任何记录的数据，调用该更新方法后应保持原状态且不写入 `receiving_at/receiving_by`。

    MAJORL184-188并发安全
    新增的原子更新语句仅按 `shipmentId` 做自增：`ps.received_receptacle_qty = COALESCE(ps.received_receptacle_qty, 0) + 1`，但同一 diff 中没有任何新增条件把本次扫描的 `receptacleCode` 与 shipment 进行幂等约束或去重校验。若同一个 receptacle 被重复扫描、重放请求，或并发重复提交到同一 shipment，这条 SQL 会无条件重复累加，导致 `received_receptacle_qty` 被放大，并可能提前把 `shipment_status` 置为 `RECEIVED`。这属于典型的并发/重放安全问题，最终会造成状态被错误推进。
    修复建议与测试场景
    修复建议

    在更新条件中加入幂等保护，确保“同一 shipment + 同一 receptacle”只记一次。例如先在明细表/批次表建立唯一约束并以插入成功作为是否允许累加的前置条件；或在本 SQL 前增加基于 `receptacleCode` 的存在性校验并配合唯一索引、事务回滚。若业务允许重复请求，应返回已处理结果而不是再次自增。
