    MAJORL39安全风险
    `zipCode` 被直接拼接进 URL 路径，未做路径段编码或格式校验。若上游可传入包含 `/`、`?`、`#`、`%2F` 等特殊字符的值，实际请求路径可能被改写，导致调用到错误的 PUS 资源，属于新增的下游调用面安全风险。
    修复建议与测试场景
    修复建议

    对 `zipCode` 先做白名单校验（例如仅允许预期的邮编字符集和长度），再按 URL path segment 进行编码后拼接；优先使用 `UriComponentsBuilder` 等安全 URL 构造方式，避免手工字符串拼接。

    测试场景

    补充单测覆盖异常输入：`"12345/extra"`、`"12345?x=1"`、`".."`、`"%2Ftmp"`。断言生成的 URL 不会改变既定路径层级，非法邮编会被拒绝。

    MINORL38-39配置变更
    `getPickupCoverageUrl` 直接拼接 `pickupCoverage` 和 `zipCode`，没有对新增配置项是否为空做校验。若配置中心未下发 `pickupCoverage`，这里会生成类似 `hostnull/12345` 的错误地址，问题会在调用外部接口时才暴露，定位成本较高。
    修复建议与测试场景
    修复建议

    在构造 URL 前增加非空校验并尽早失败，例如：`Assert.hasText(pickupCoverage, "pus api pickupCoverage must not be blank");`。如果该配置为必填，也建议在配置类启动时统一校验。

    测试场景

    增加配置缺失场景测试：`pickupCoverage` 为空或未配置时，调用 `getPickupCoverageUrl("12345")` 应抛出明确异常，而不是返回 malformed URL。
