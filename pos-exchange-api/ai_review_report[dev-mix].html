    pos-service/src/main/java/com/miq/posservice/factory/pipeline/ProducerConsumerPipeline.java2 Major2 个问题
    MAJORL31-38边界条件
    `run(...)` 未校验 `pageSize` 和 `queueCapacity` 的边界。`queueCapacity <= 0` 时 `new LinkedBlockingQueue<>(queueCapacity)` 会直接抛 `IllegalArgumentException`；`pageSize <= 0` 时会把非法分页参数透传给生产者，且 `AbstractDataProducer` 中 `page.size() < pageSize` 永远不成立，容易导致分页逻辑失效甚至死循环。
    修复建议与测试场景
    修复建议

    在入口处显式校验参数并快速失败，例如：`if (pageSize <= 0) throw new IllegalArgumentException("pageSize must be > 0"); if (queueCapacity <= 0) throw new IllegalArgumentException("queueCapacity must be > 0");`。

    测试场景

    分别传入 `pageSize=0`、`pageSize=-1`、`queueCapacity=0` 调用 `run()`；验证方法立即抛出清晰的参数异常，而不是在线程执行中随机失败或卡住。

    MAJORL60-79并发安全
    这里串行等待 `producerFuture.get()` 再等待 `consumerFuture.get()`，会在消费者提前失败时造成整体卡死：消费者退出后不再消费队列，生产者可能阻塞在 `queue.put(...)`，主线程却一直卡在第一个 `producerFuture.get()`，无法感知消费者异常并取消生产者。该场景在有界队列下会直接导致任务无法结束。
    修复建议与测试场景
    修复建议

    改为并行感知两个 Future 的完成状态，并在任一侧失败时立即取消另一侧，例如使用 `CompletionService`/`CompletableFuture.anyOf(...)`，或先轮询 `isDone()` 检查两个任务；一旦消费者失败，立刻 `producerFuture.cancel(true)`。至少要避免固定先 `get()` 生产者再 `get()` 消费者的等待顺序。

    测试场景

    构造 `queueCapacity=1`，让消费者处理首个 batch 时抛异常，生产者持续产生至少 2 个 batch；验证 `run()` 不会永久阻塞，而是能及时返回消费者失败异常。
