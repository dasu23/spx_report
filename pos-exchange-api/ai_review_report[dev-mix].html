major190-191边界条件`loginSSO` 直接对 `sysUser.getRoleIds()` 做 `split(",")` 和 `Long.valueOf` 转换，没有判空或过滤空串。若 SSO 用户未配置角色、角色串为空，或存在尾随逗号/脏数据，这里会触发 `NullPointerException` 或 `NumberFormatException`，导致登录流程直接失败。在拆分前校验 `roleIds` 非空，并过滤空白项后再转换，例如：`if (StringUtils.isBlank(sysUser.getRoleIds())) { throw new RRException("Role is not set!"); }`，随后使用 `Arrays.stream(...).map(String::trim).filter(StringUtils::isNotBlank)`。覆盖 `roleIds == null`、`roleIds == ""`、`roleIds == "1,2,"` 三种场景，验证 `loginSSO` 不会抛出运行时异常，并返回明确业务错误。
major287-306安全风险`performLogin` 在 `currentUser.isAuthenticated()` 为 `true` 时直接返回当前会话信息，而不会校验该会话是否属于传入的 `sysUser`。这会导致浏览器已有 A 用户会话时，调用 B 用户的 SSO 登录接口仍返回 A 的登录态，出现账号串用/会话混淆。进入 SSO 登录前先校验当前主体与目标用户是否一致；若不一致，显式 `logout()` 后再按目标用户建立新会话。至少增加类似 `if (currentUser.isAuthenticated() && !Objects.equals(currentPrincipal, sysUser.getLoginName())) { currentUser.logout(); }` 的保护。先登录用户 A，再在同一会话中调用用户 B 的 SSO 登录接口；期望结果应切换为 B 或拒绝请求，不能继续返回 A 的 `LoginRes`。
major288-290业务逻辑`performLogin` 用 `sysUser.getPasswd()` 构造 `UsernamePasswordToken`。这里的 `sysUser` 来自数据库，`passwd` 通常是已加密后的持久化密码；Shiro 登录流程期望的是用户提交的明文凭证，再由 `CredentialsMatcher` 进行比对。结果是 SSO 用户在未登录状态下大概率会因为“二次加密比对”而登录失败。不要复用用户名密码登录链路去提交库里的密码哈希。应为 SSO 定义独立的认证 Token/Realm，或在 IdP 校验通过后直接建立本地登录态。例如新增 `SsoAuthenticationToken(loginName)` 并在专用 Realm 中仅校验用户状态、角色、facility 等本地信息。新增 SSO 登录测试：构造数据库中已加密密码的 `SysUser`，调用 `loginSSO` 时应能成功建立会话；当前实现下应复现认证失败。
