    pos-exchange-api/src/main/java/com/miq/posexchangeapi/controller/sorter/DamonController.java3 Major3 个问题
    MAJORL438-499并发安全
    `handleFirstStop` 先通过 `existsBySortIdAndStatusInAndFacilityCode(...)` 判空，再调用 `createContainerWithParcels(...)` 创建 ACTIVE 容器，这两个操作之间没有任何锁或唯一性约束保护。并发的两次 Stop 请求可同时通过存在性检查，最终为同一 chute/facility 重复创建多个 ACTIVE/CLOSED 容器，导致容器状态错乱，后续分拣与关箱流程可被并发请求破坏。
    修复建议与测试场景
    修复建议

    将“检查是否已有容器”和“创建容器”合并为单个原子操作：在数据库层为 `(facility_code, sort_id, status in ACTIVE/CLOSED)` 增加唯一约束或使用事务内行锁/悲观锁；若无法直接建部分唯一索引，则引入单独的幂等表或分布式锁，确保同一 chute 同时只能成功创建一个容器。

    测试场景

    增加并发测试：两个线程同时对同一 `facilityCode + sortId` 调用第一次 Stop，验证最终只创建 1 个 ACTIVE 容器，另一个请求返回明确的幂等/冲突结果。

    MAJORL455-456安全风险
    异常分支将 `e.getMessage()` 直接作为接口响应返回给外部调用方。底层异常消息常包含内部表名、容器编号规则、SQL/JPA 错误或第三方调用细节，这会向未受信任调用方泄露内部实现信息，放大后续攻击面。相同问题在 `handleFullStop` 的异常返回中再次出现。
    修复建议与测试场景
    修复建议

    对外统一返回通用错误文案和稳定错误码，例如 `Operation failed`；将详细异常仅保留在服务端日志中，并确保日志不包含敏感数据。

    测试场景

    构造 DAO/Service 抛出包含内部细节的异常，验证接口响应不再回显原始异常消息，仅返回通用错误码/文案。

    MAJORL487-488边界条件
    这里直接按固定下标读取 `parts[4]`，并假设最后一段一定是数字序号。只要历史 `containerNum` 格式不符合注释中的约定，或数据中存在脏值，就会触发 `ArrayIndexOutOfBoundsException` / `NumberFormatException`，导致满载 Stop 直接失败。
    修复建议与测试场景
    修复建议

    在解析前增加格式校验，至少校验 `parts.length` 和最后一段是否为数字；不满足格式时返回明确业务错误而不是依赖运行时异常。例如先判断 `parts.length >= 6 && NumberUtils.isDigits(parts[parts.length - 1])`，否则记录错误并中止创建新容器。

    测试场景

    准备一个 `containerNum` 为非标准格式（如缺少日期段、尾段非数字）的已存在容器，调用满载 Stop，预期返回可读错误信息且不抛 500。
