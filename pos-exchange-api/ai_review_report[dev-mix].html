    MAJORL439-451并发安全
    `handleFirstStop` 先执行 `existsBySortIdAndStatusInAndFacilityCode(...)`，随后再创建 ACTIVE 容器，这两个步骤之间没有看到事务锁或唯一性约束保护。并发的首次 Stop 请求可同时通过存在性检查，导致同一 chute 被重复创建多个 ACTIVE 容器/标签，造成状态竞争和数据不一致。
    修复建议与测试场景
    修复建议

    将“检查是否已存在容器”和“创建新容器”收敛到同一原子操作中，例如在服务层开启事务并对 chute 记录加行级锁，或在数据库层增加能约束同一 `facilityCode + sortId` 活跃容器唯一性的索引/幂等约束；创建冲突时返回明确的业务错误。

    测试场景

    增加并发测试：同一 `facilityCode + sortId` 同时发送两次首次 Stop，请求结束后应最多只存在一个新建 ACTIVE 容器，且只有一个请求拿到新 label。

    MAJORL446-452业务逻辑
    查找目标容器时按 `createTime desc` 取第一个 `ACTIVE/CLOSED` 且 `scanType=A` 的容器，没有优先选择 `ACTIVE`。如果同一 chute 同时存在“较新的 CLOSED 容器”和“较早的 ACTIVE 容器”，当前逻辑会把本次包裹挂到 CLOSED 容器上，旧的 ACTIVE 容器则一直不关闭，导致容器归属和状态流转错误。
    修复建议与测试场景
    修复建议

    选择目标容器时应优先命中 `ACTIVE`，仅在不存在 `ACTIVE` 时再回退到最新 `CLOSED`。可拆成两步查询，或在流中过滤后先找 `ACTIVE`：`List<PosContainer> containers = ...; Optional<PosContainer> targetOpt = containers.stream().filter(c -> isScanTypeA(c.getContainerNum()) && PosContainerStatusEnum.ACTIVE.getName().equals(c.getStatus())).findFirst().or(() -> containers.stream().filter(c -> isScanTypeA(c.getContainerNum()) && PosContainerStatusEnum.CLOSED.getName().equals(c.getStatus())).findFirst());`
