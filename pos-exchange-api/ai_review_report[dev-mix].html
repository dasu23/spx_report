    MAJORL443-517并发安全
    `handleFirstStop` 先查询当前 chute 是否存在 ACTIVE/CLOSED 的 A 类容器，再调用 `createContainerWithParcels` 创建新容器；这两个步骤之间没有任何锁或事务隔离。并发的首次 Stop 请求可同时通过 `hasExisting` 检查，最终重复创建多个 ACTIVE 容器并打印多张标签，形成竞态条件。
    修复建议与测试场景
    修复建议

    将“查重 + 创建”下沉到单个事务中，并对 chute 维度加行锁/唯一约束；至少为 `(facilityCode, sortId, scanType, active-like status)` 建立唯一性保护，或在服务层引入幂等键避免重复创建。

    测试场景

    针对同一 `facilityCode + sortId` 并发发送两次空 `details` 的 Stop 请求，断言最终只存在一个新建的 ACTIVE A 类容器，且只返回一张新标签。

    MAJORL491-492边界条件
    这里直接假设容器号一定满足 `CTN-...-{date}-...-{seq}` 格式，并且最后一段一定可转为整数。当前方法只通过 `isScanTypeA()` 校验了 `parts.length > 3`，如果库里存在历史容器号、脏数据或其他格式变体，`parts[4]` 会触发数组越界，`Integer.parseInt(parts[parts.length - 1])` 会抛 `NumberFormatException`，导致第二次 Stop 直接失败。
    修复建议与测试场景
    修复建议

    在解析前补充格式校验，并对异常格式返回明确业务错误而不是依赖异常兜底。例如：`String[] parts = targetContainer.getContainerNum().split("-"); if (parts.length < 6 || !NumberUtils.isDigits(parts[parts.length - 1])) { return BaseResponse.<String>builder().code(HttpStatus.HTTP_INTERNAL_ERROR).msg("Invalid container number format: " + targetContainer.getContainerNum()).build(); }`；同时再读取 `parts[4]`。
