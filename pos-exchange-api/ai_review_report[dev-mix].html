    pos-service/src/main/java/com/miq/posservice/utils/RedisUtils.java2 Major2 个问题
    MAJORL212-215边界条件
    新增方法 `hasKey` 直接调用 `redisDataEnum.getIndex()` / `redisDataEnum.getRedisKey()`，当调用方传入 `redisDataEnum == null` 时会触发空指针异常。该异常虽被 catch 后返回 `false`，但会把入参错误与“key 不存在”混淆，导致业务判断失真。
    修复建议与测试场景
    修复建议

    在方法入口做显式非空校验并区分参数错误与 Redis 查询结果，例如：`if (redisDataEnum == null) { throw new IllegalArgumentException("redisDataEnum must not be null"); }`；`setIfAbsent` 同样增加该校验，避免隐式 NPE。

    测试场景

    新增单测：1) `hasKey(key, null)` 断言抛出 `IllegalArgumentException`；2) `setIfAbsent(key, value, null)` 断言抛出 `IllegalArgumentException`；3) 传入合法 `redisDataEnum` 时保持原有行为。

    MAJORL216异常处理
    `hasKey` 在捕获任意 Redis 异常后直接返回 `false`，属于失败默认放行（fail-open）语义。若上层将 key 存在性用于会话有效性、幂等防重或风控状态判断，Redis 短暂故障会被误判为“key 不存在”，可能导致安全校验被绕过。
    修复建议与测试场景
    修复建议

    将异常路径改为失败默认拒绝：1) 抛出业务异常由上层中断流程；或 2) 返回三态结果（存在/不存在/未知）并要求调用方在“未知”时拒绝或重试。避免在底层工具类吞掉异常并返回与正常语义相同的布尔值。

    测试场景

    模拟 Redis 不可用（连接超时/断连）时调用 `hasKey`：断言不会返回与“key 不存在”相同语义；上层依赖该判断的安全流程（如 token 黑名单检查、幂等锁检查）应进入拒绝或降级阻断路径。
