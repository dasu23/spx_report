SeverityLineCategoryDescriptionSuggestionTest Scenario
major1031-1050业务逻辑新逻辑先过滤掉 `businessNo == null` 的记录，再仅按 `validIds` 判断有效任务。这样 `businessNo` 为空的任务不会进入 `latestByFacility`，随后会被直接标记为 `RESOLVED`，等价于被静默丢弃，而不是继续处理或显式报错。对 `businessNo` 为空的任务单独分支处理，不要直接按“过期任务”结论落成 `RESOLVED`。可改为：空 `businessNo` 的任务保留在 `validPushLogs` 中单独执行，或者先置为 `PENDING` 并写入明确错误信息。准备一条 `PushLog.businessNo = null` 的待处理记录，执行该流程后校验它不会被错误标记为 `RESOLVED`，而是按预期处理或进入可重试失败状态。
major1057-1066并发安全这里把 `PushLog` 实体处理和 `pushLogDao.save(...)` 放进 `parallel()` 并行流中执行。JPA Repository/持久化上下文默认不是线程安全的，并行线程同时修改同一批受管实体并落库，容易出现状态覆盖、脏写或 `EntityManager` 相关并发异常。改为串行处理，或先在并行阶段只计算纯内存结果，汇总后在单线程中统一更新数据库。例如去掉 `parallel()`：`for (PushLog pushLog : validPushLogs) { try { labelRouteService.refreshRouteProfile(pushLog, finalRefreshFetchSize); } catch (Exception e) { ... pushLogDao.save(pushLog); } }`。构造同一批次多个 `validPushLogs`（20+）并发刷新，压测观察是否出现状态错乱、重复更新、`Session/EntityManager` 并发异常或最终 `tryTimes`/`status` 不一致。

SeverityLineCategoryDescriptionSuggestionTest Scenario
major281-290并发安全`skipOptimizeRoute` 仅按 `routeId` 读取并直接更新，没有任何状态校验或并发保护。若优化任务线程正在处理同一条 route，人工跳过与自动优化完成可能同时发生，导致 `optimizeSkipFlag/hasOptimize/optimizeTryTimes` 等字段被相互覆盖，出现重复推送或最终状态不可预期。在更新前增加状态条件校验并使用乐观锁/条件更新，例如只允许在 `hasOptimize=false` 且未被其他流程处理时执行跳过；必要时改为 DAO 条件更新：`update ... where id=? and has_optimize=0`，并依据更新行数决定是否继续推送。并发模拟“优化线程完成”与“人工点击跳过”同时操作同一 `routeId`，验证最终只会有一个流程生效，且不会重复推送 DMS。
major289-290数据一致性`skipOptimizeRoute` 先调用 `fixRouteScanService.pushRouteToDms(inboundRoute)` 执行后续推送/业务更新，再持久化 `hasOptimize/optimizeSkipFlag/optimizeSkipOperator`。如果推送成功后 `inboundRouteDao.save(inboundRoute)` 失败，外部/DMS侧已经按“跳过优化”继续流转，但本地路由仍保持未跳过状态，产生状态不一致。先落库再触发后续推送，或把“更新路由跳过状态”和“创建/触发推送任务”放入同一事务边界，并通过事务提交后事件异步推送。例如先 `inboundRouteDao.save(inboundRoute);`，再在事务提交后调用 `pushRouteToDms`。构造 `pushRouteToDms` 成功、`inboundRouteDao.save` 抛异常的场景，验证不会出现 DMS 已推送但数据库仍未标记 `optimizeSkipFlag=true` 的不一致状态。
