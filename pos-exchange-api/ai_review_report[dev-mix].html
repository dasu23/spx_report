major51-53安全风险SSO token 被放在 URL path 中，并且在 53、69、75 行被直接写入日志；其中 68-69 行还会把最终 `authToken` 拼进重定向 URL 后完整打印。这样无论是应用日志、网关访问日志还是浏览器历史，都可能暴露可用于登录的敏感令牌，形成会话泄露。相同问题也存在于 `pos-admin-api` 的同名控制器。不要记录原始 SSO token 或登录后的 `authToken`；日志里最多保留脱敏后的 requestId / userId。若前端协议允许，避免把敏感 token 放在 URL path/query 中，改为后端交换后写入安全 Cookie，或至少使用一次性短期 code。增加日志安全测试：执行一次 SSO 回调后，断言应用日志中不包含完整 `ssoLoginToken`、`authToken` 或带 token 的重定向 URL。
