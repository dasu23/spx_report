    pos-exchange-api/src/main/java/com/miq/posexchangeapi/controller/sorter/DamonController.java2 Major2 个问题
    MAJORL438-499并发安全
    `handleFirstStop` 采用“先查是否存在，再创建容器”的非原子流程。并发的两次 Stop 请求可同时通过 `existsBySortIdAndStatusInAndFacilityCode(...)` 检查，然后各自执行 `createContainerWithParcels(...)`，导致同一 chute 被重复创建多个 ACTIVE 容器，造成标签重复生成和后续分拣状态混乱。
    修复建议与测试场景
    修复建议

    将“存在性校验 + 创建容器”放入单一事务并增加并发控制：优先使用数据库唯一约束（例如 facilityCode+sortId+status in ACTIVE 的唯一性）兜底，并在服务层对 chute 维度加悲观锁/分布式锁；创建失败时返回明确的幂等结果。

    测试场景

    构造同一 `facilityCode/sortId` 的两个并发 first-stop 请求，验证最终仅能成功创建 1 个 ACTIVE 容器，另一请求返回幂等/冲突结果。

    MAJORL501-503边界条件
    这里直接按固定下标解析 `containerNum`，并假设最后一段一定是数字序号。当前仅通过 `isScanTypeA()` 校验了 `parts.length > 3`，无法保证 `parts[4]` 存在或末段可 `parseInt`。一旦库里存在历史格式不符的容器号，会在满载 Stop 时触发数组越界或数字解析异常，导致接口直接返回 500。
    修复建议与测试场景
    修复建议

    在解析前补充完整格式校验，例如先校验 `parts.length >= 6` 且最后一段为纯数字；校验失败时返回明确业务错误而不是继续创建新容器。示例：`if (parts.length < 6 || !NumberUtils.isDigits(parts[parts.length - 1])) { return BaseResponse.<String>builder().code(HttpStatus.HTTP_INTERNAL_ERROR).msg("Invalid container number format: " + targetContainer.getContainerNum()).build(); }`。

    测试场景

    补充用例：`targetContainer.getContainerNum()` 分别为 `CTN-262-A`、`CTN-262-FR-A-260331-ORD-ORD-c1-X`、`null`/空串时，`handleFullStop()` 应返回可读错误而不是抛异常。

    pos-service/src/main/java/com/miq/posservice/service/api/TrkApiService.java1 Major1 个问题
    MAJORL55-57API契约
    `findStateHolidayDates()` 的查询结果第二列是实体字段 `h.holidayDate`，运行时通常返回 `Date`/`LocalDate` 一类对象，而这里强制转换为 `String`。当 tracking-service 调用该接口加载假期数据时，容易在流处理中触发 `ClassCastException`。
    修复建议与测试场景
    修复建议

    不要直接把 `Object[]` 强转为 `String`。改为使用 DTO/Projection 承接查询结果，或对日期字段做显式格式化转换，例如 `Collectors.mapping(row -> ((Date) row[1]).toString(), Collectors.toList())`，具体类型按实体字段实际类型处理。

    测试场景

    补充单测或集成测试：模拟 `findStateHolidayDates()` 返回 `Object[]{"CA", java.sql.Date.valueOf("2025-01-01")}`，验证 `getAllHolidaysByState()` 不会抛 `ClassCastException`，并能输出预期字符串日期。

    pos-service/src/main/java/com/miq/posservice/service/xhc/DamonService.java1 Major1 个问题
    MAJORL258-261性能隐患
    这里对 `targetNoList` 中每个 chute 都单独调用一次 `existsBySortIdAndStatusInAndFacilityCode(...)`，形成典型 N+1 查询。自动线一次落件候选格口较多时，请求耗时会随格口数线性放大，容易拖慢分拣主流程。
    修复建议与测试场景
    修复建议

    改为一次性批量查询已有容器的 chute 集合，再在内存中过滤。例如新增 DAO：`findSortIdBySortIdInAndStatusInAndFacilityCode(...)`，返回命中的 `sortId` 列表或集合，然后 `targetNoList.stream().filter(existingSet::contains)`。

    测试场景

    补充性能/行为测试：当 `targetNoList` 含 50 个 chute 时，验证只发生 1 次数据库查询，且过滤结果与逐个 `exists` 的逻辑一致。
