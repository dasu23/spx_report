    pos-service/src/main/java/com/miq/posservice/dao/DispatchV2Dao.java1 Minor1 个问题
    MINORL81-84性能隐患
    新增的子查询在 `pos_dispatch_container_v2` 聚合前增加了 `where dispatch_code in (select dispatch_code from pos_dispatch_v2 where id in (:ids))`。这会让数据库先执行一个 `IN (subquery)` 过滤，再做 `group by dispatch_code`，在 `ids` 较多或 `dispatch_code` 索引不理想时，容易退化为额外的子查询扫描；而外层本身已经通过 `t3.dispatch_code = pdv.dispatch_code` 与 `where pdv.id in (:ids)` 约束结果集，存在重复过滤。
    修复建议与测试场景
    修复建议

    优先改为直接基于外层 dispatch 关联后聚合，避免 `IN (subquery)` 的二次筛选，例如将聚合子查询改为与 `pos_dispatch_v2` 显式 join，或保留原聚合并依赖外层 `pdv.id in (:ids)` 过滤。如果必须保留当前写法，建议确认 `pos_dispatch_container_v2.dispatch_code` 与 `pos_dispatch_v2.dispatch_code/id` 上有可用索引，并补充执行计划验证。

    测试场景

    构造 500~2000 个 `ids` 的批量 BOL 导出场景，对比变更前后 SQL 执行时间与 explain plan，重点观察 `pos_dispatch_container_v2` 是否出现全表扫描或临时表/文件排序。
