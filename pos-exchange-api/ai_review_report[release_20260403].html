    pos-service/src/main/java/com/miq/posservice/api/PusApi.java2 Major2 个问题
    MAJORL39-48安全风险
    `log.info("[PusApi] createPickup url={}, body={}", url, body)` 会记录发往下游 PUS 的完整创建 pickup 报文。该报文通常包含收寄件地址、联系人、电话或运单信息，属于明显的敏感数据落日志问题。
    修复建议与测试场景
    修复建议

    禁止记录完整请求报文；仅记录请求唯一标识、目标接口和必要的非敏感摘要字段。若必须排障，增加统一脱敏器后再输出。

    测试场景

    构造包含姓名、电话、地址的 `PickupCreateForm` 调用 `createPickup`，检查日志中不应出现完整 JSON、明文电话或详细地址。

    MAJORL45-48业务逻辑
    这里在 `payload` 非空时直接把响应当作成功结果返回，没有校验顶层 `success`/`code`。如果 PUS 返回 `success=false` 但仍带空对象或不完整的 `payload`，下游 `createPickupStaging` 只检查 `failList`，会把实际失败请求当成成功处理。
    修复建议与测试场景
    修复建议

    先校验顶层成功标记，再解析 `payload`。例如：`boolean success = root.path("success").asBoolean(false); int code = root.path("code").asInt(-1); if (!success || code != 200) { String message = root.path("message").asText("PUS createPickup failed"); log.error("[PusApi] createPickup failed, url={}, response={}", url, resp); throw new RRException(message); }`，之后再做 `payload` 转换。

    测试场景

    模拟 PUS 返回 `{"success":false,"code":500,"message":"remote error","payload":{}}`，断言 `createPickup`/`createPickupStaging` 抛出异常或标记 FAILED，而不是继续按成功流程执行。

    修复建议与测试场景
    修复建议

    对配置增加启动期校验，并规范化 URL 拼接。可在属性上增加 `@Validated` + `@NotBlank`，同时使用安全拼接逻辑，例如 `StringUtils.appendIfMissing(host, "/") + StringUtils.removeStart(pickupCreate, "/")`。

    测试场景

    分别覆盖 `host` 为空、`pickupCreate` 为空、`host` 末尾无 `/`、`pickupCreate` 开头有 `/` 的场景，断言启动期失败或生成的 URL 始终合法。
