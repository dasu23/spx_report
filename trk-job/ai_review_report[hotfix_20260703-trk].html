    MAJORL69-107并发安全
    在 `buildAndWireBlobUtilMap` 中通过多次静态 setter 依次更新 `TrkBlobSasUtil` 的路由状态（utilMap、orgContainerMap、webDomainMap、开关）。这些更新不是原子操作；若刷新事件与业务线程并发执行，静态工具类可能在短时间内观察到“新旧配置混合”的中间态，例如 utilMap 已切换但 orgId→container 映射仍是旧值，导致上传或 SAS 生成路由错误。
    修复建议与测试场景
    修复建议

    将 `TrkBlobSasUtil` 的多项配置封装为一个不可变快照对象，通过单次原子替换（如 `volatile`/`AtomicReference<ConfigSnapshot>`）完成刷新；或提供一个 `refreshAll(...)` 方法一次性接收全部配置并在内部原子切换，避免多个 setter 分步更新。

    测试场景

    构造并发测试：一个线程持续调用 `TrkBlobSasUtil.toSasUrl/upload`，另一个线程触发 `onRefresh()` 并切换 base container、org 映射和 webDomain；验证任意时刻不会出现找不到 container、路由到错误 container 或 URL host 与 token 来源不一致的情况。

    MINORL87-88业务逻辑
    当 `azure.*` 下存在多个配置项指向同一个 `containerName` 时，这里直接 `put` 到 `utilMap`，后一个配置会静默覆盖前一个配置，启动和刷新都不会报错。由于 map 的 key 实际使用的是 `containerName` 而不是 Nacos 子键，这种重复配置会导致某个账号/域名配置被悄悄替换，排查困难。
    修复建议与测试场景
    修复建议

    在 `put` 前增加重复检测，例如 `if (utilMap.containsKey(containerName)) { throw new IllegalStateException(...) }`，并在异常信息中同时打印 YAML key 与 containerName，阻止歧义配置进入运行态。

    测试场景

    配置两个不同的 `azure.*` 子项但 `container-name` 相同，验证应用启动/刷新时明确失败并输出重复 container 的诊断信息。
