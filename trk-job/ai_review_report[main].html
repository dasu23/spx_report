    不要记录原始响应体；改为记录状态码、请求标识和截断后的错误摘要，必要时对响应内容做脱敏后再打印。

    `patchCase` 仅依赖 `requestPatch` 是否抛异常来判断成功，未校验 HTTP 状态码。如果底层工具在 4xx/5xx 时返回响应但不抛异常，这里会直接记录 `success` 并返回 `true`，造成调用方误以为 SAP 更新成功，属于明显的业务结果误判。
    修复建议与测试场景

    测试场景

    模拟 SAP PATCH 返回 412 或 500 且底层不抛异常，验证 `patchCase` 返回 `false` 且不会打印成功日志。

    MAJORL41-42业务逻辑
    线程池拒绝策略使用 `DiscardOldestPolicy`，当队列 `LinkedBlockingDeque<>(1000)` 被打满时会静默丢弃最早进入队列的 webhook 任务。对 webhook 场景，这会直接导致部分回调未被处理，属于可见的业务数据丢失风险。
    修复建议与测试场景
    修复建议

    测试场景

    构造超过 1000 个 webhook 任务并持续提交，验证队列满载时不会出现任务静默丢失；同时校验拒绝场景有明确日志或失败反馈。

    MAJORL42资源管理
    `ThreadPoolExecutor.DiscardOldestPolicy()` 会在队列打满时静默丢弃最早排队的 webhook 任务。该线程池用于 SAP webhook 异步处理时，突发流量或恶意请求可通过填满队列触发任务丢失，导致合法 webhook 事件未被处理，带来可用性和数据完整性风险。
    修复建议与测试场景
    修复建议

    不要对 webhook 这类关键任务使用静默丢弃策略。改为 `AbortPolicy` 或 `CallerRunsPolicy` 并结合明确告警/监控；更稳妥的方案是将 webhook 事件先持久化或投递到可靠队列，再异步消费，并确保失败可重试。

    测试场景

    构造超过 1000 个并发 webhook 任务压测该线程池，验证队列饱和时是否出现任务静默丢失；补充测试断言拒绝策略会产生可观测错误或触发重试，而不是直接丢弃历史任务。
