    MAJORL90-95数据一致性
    这里把下游同步从 Kafka 切换为 POS API 后，失败语义发生了变化：前面的 `batchSyncPoints` 和 `refreshCache` 已经提交成功，但 `posApi.syncPudoLocations(internalDtos)` 一旦抛错，任务会整体失败且没有任何补偿/降级路径，导致数据库与缓存已更新、下游却未收到同步，形成部分成功的数据不一致。原先 Kafka 路径至少与新路径是不同的交付机制，现在按开关二选一后，这个部分失败窗口被直接暴露出来。
    修复建议与测试场景
    修复建议

    为第 4 步增加明确的失败处理策略，例如：1）捕获 POS API 异常后落回 Kafka 重试通道；或 2）记录待补偿任务并由独立重试任务补发；或 3）将“数据库更新/缓存刷新/下游通知”纳入可观测的状态机。至少应在 `posApi.syncPudoLocations` 失败时持久化失败批次，避免静默丢失下游同步。

    测试场景

    构造 `batchSyncPoints` 和 `refreshCache` 成功、`posApi.syncPudoLocations` 抛异常的场景，验证任务失败后是否会产生可重试记录或自动降级到 Kafka，且不会出现数据库已更新但下游永久未同步的状态。

    MINORL79-88性能隐患
    构造 `internalDtos` 时，对 `saved` 中的每个元素都再次遍历整个 `pointDtos` 做 `filter(...).findFirst()`，时间复杂度为 O(n*m)。当同步点位数量较大时，这一步会明显放大任务耗时。
    修复建议与测试场景
    修复建议

    先把 `pointDtos` 预构建为以 `pointCode` 为 key 的 Map，再在遍历 `saved` 时 O(1) 查找对应 DTO，例如 `Map<String, VintedPointDto> pointMap = pointDtos.stream().collect(Collectors.toMap(VintedPointDto::getPointCode, Function.identity(), (a,b)->a));`，随后用 `pointMap.get(i.getPointCode())` 填充扩展字段。

    测试场景

    准备数千条 `saved` 和 `pointDtos` 数据，比较当前实现与基于 Map 的实现耗时，确认大批量同步下不会因嵌套流遍历导致任务明显变慢。
