    pos-service/src/main/java/com/miq/posservice/service/api/TrkApiService.java1 Major1 个问题
    MAJORL49-52API契约
    `findStateHolidayDates()` 的查询结果第二列来自 `h.holidayDate`，这里被直接强转为 `String`。如果实体字段类型是 `Date`/`LocalDate`/`Timestamp`（JPA 中更常见），运行时会在聚合时触发 `ClassCastException`，导致 tracking-service 拉取假期数据失败。
    修复建议与测试场景
    修复建议

    不要对 `Object[]` 做字符串硬转型。改为显式投影或在这里做类型转换，例如返回 DTO/接口投影；或使用 `row[1].toString()` 仅作为兜底，并按实际日期类型统一格式化。示例：`Collectors.mapping(row -> ((LocalDate) row[1]).toString(), Collectors.toList())`。

    测试场景

    为 `StateHolidayMappingDao.findStateHolidayDates()` mock 一条 `{ "CA", LocalDate.of(2025,1,1) }` 或 `java.sql.Date` 结果，调用 `getAllHolidaysByState()`，验证不会抛出类型转换异常且返回格式正确。

    pos-service/src/main/java/com/miq/posservice/service/container/impl/PosContainerV2ServiceImpl.java1 Major1 个问题
    MAJORL257-274数据一致性
    `addParcelsToContainer`/`addParcelsAndCloseContainer` 直接用 `findByTrackingNumberIn(trackingNumbers)` 的返回结果落库并关闭容器，但没有校验查询结果是否覆盖了本次上报的全部追踪号。只要有追踪号不存在、重复或被 DAO 去重，方法就会静默少关联部分包裹，随后 `addParcelsAndCloseContainer` 仍把容器置为 `CLOSED`，造成包裹漏关联且难以补救。
    修复建议与测试场景
    修复建议

    在保存前校验 `labels` 与请求参数的一致性：对 `trackingNumbers` 去重后比对数量，并找出缺失单号；一旦存在未查到的追踪号，直接抛异常回滚，不要关闭容器。示例：`Set<String> requested = new HashSet<>(trackingNumbers); Set<String> found = labels.stream().map(Label::getTrackingNumber).collect(toSet()); if (!found.containsAll(requested)) throw new RRException(...)`。

    测试场景

    传入 3 个 tracking number，其中 1 个数据库不存在；验证方法抛错并回滚，`posContainerParcelDao.saveAll`、`posContainerDao.save` 都不会提交，容器状态保持原值。
