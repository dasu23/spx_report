    config/routes.ts1 Major1 个问题
    MAJORL78权限与访问控制
    父级 `/general-setting` 路由的 `access` 从单一 `customer-mapping:view` 扩大为 `customer-mapping:view|user:list`。这会让仅具备 `user:list` 的用户也能进入整个 General Setting 分组，而该分组下默认子路由仍指向 customer-mapping 页面。若路由匹配先落到默认子页，用户可能先进入自己无权访问的页面，出现 403/空白页/错误跳转，属于菜单可见性与实际子页权限不一致。
    修复建议与测试场景
    修复建议

    避免用父级 OR 权限直接放大分组入口。可选修复：1）给 `/general-setting` 增加一个无组件的分组路由，仅用于菜单展示，并把首个可访问子路由作为重定向目标；或 2）保留父级最小权限，并单独为 `user-management` 提供可直达菜单入口；或 3）在进入父级时根据权限动态重定向到 `customer-mapping` 或 `user-management`。

    测试场景

    使用仅有 `user:list`、没有 `customer-mapping:view` 的账号访问 `/general-setting`，确认不会先落到 customer-mapping 再报无权限，而是直接进入 `/general-setting/user-management` 或展示合法默认页。

    src/access.ts1 Major1 个问题
    MAJORL6-33权限与访问控制
    **原因：** 在统一 `hasPermitted` 逻辑中新增如下分支： ```ts if (MOCK_GRANTS_ALL && typeof p === 'string' && !excludes.includes(p as any)) { return true; } ``` 该实现对所有字符串权限码一刀切放行，且 `excludes` 默认是空数组，意味着新增的用户管理、缓存清理等敏感能力都会被自动授权。由于这是权限表达式求值后的兜底逻辑，影响范围覆盖整个前端路由、组件按钮和操作入口，而不是单个页面。 **影响：** 任意在该环境访问前端的用户都可能获得管理类功能入口，例如用户列表、删除用户、清缓存等。若后端存在任何遗漏鉴权的接口，前端这一改动会直接放大为实际越权操作；即便后端安全，也会造成敏感功能暴露和错误操作风险。
    修复建议与测试场景
    修复建议

    删除该全局兜底放行分支，改为在 mock 初始态中按页面/角色精确注入权限集合；若必须保留，也应改成仅对白名单权限生效，并要求显式开关与开发环境双重限制。 示例： ```ts const MOCK_PERMISSION_WHITELIST: (keyof Permissions)[] = ['demo:read']; if ( process.env.NODE_ENV === 'development' && process.env.MOCK === 'all-access' && typeof p === 'string' && MOCK_PERMISSION_WHITELIST.includes(p as keyof Permissions) ) { return true; } ```

    测试场景

    1. 构造一个仅有基础查看权限的 mock/真实账号；2. 在开启 MOCK_GRANTS_ALL 的环境访问用户管理、清缓存、费率卡等页面；3. 验证菜单、路由和操作按钮是否全部可见；4. 修复后确认未授予的权限码不会因 mock 环境被统一放行。

    src/components/AzureFile/DownloadTemplate.tsx1 Major1 个问题
    MAJORL29-34前端安全
    新增支持 `templatePath` 传入任意异步函数并直接将其返回值交给 `window.open(url, '_blank')`。这里不再经过原先 `getDownloadUrl('oms/static/...')` 的固定路径约束，导致调用方只要可控返回值，就能打开任意外部 URL，甚至 `javascript:`/钓鱼站点链接。该组件是通用下载组件，风险会扩散到所有后续调用点。
    修复建议与测试场景
    修复建议

    对动态返回的 URL 做严格校验，只允许 `https:` 且域名属于受信任下载域，或仅允许后端返回相对路径后再统一拼接下载域；同时显式拒绝 `javascript:`、`data:` 等危险协议。 示例： ```ts const raw = typeof templatePath === 'function' ? await templatePath() : await getDownloadUrl(...); if (!raw) return; const parsed = new URL(raw, window.location.origin); if (parsed.protocol !== 'https:' || !parsed.hostname.endsWith('trusted.example.com')) { throw new Error('Invalid download url'); } window.open(parsed.toString(), '_blank', 'noopener,noreferrer'); ```

    测试场景

    传入一个返回 `javascript:alert(1)`、`data:text/html,...`、以及第三方钓鱼域名 URL 的 `templatePath` 回调，验证当前实现会直接打开；修复后应全部被拦截，仅允许受信任下载地址。
