    src/components/AzureFile/index.tsx1 Critical 1 Major2 个问题
    CRITICALL171前端安全
    新增的 `onPreview` 直接将 `file.url`/`response.data.fileUrl` 传给 `window.open`，且失败回退分支会直接打开原始 `url`。同一文件新增的 `parseFileEntry()` 允许 `withFileName` 模式下把任意 `filename:url` 字符串解析为 URL；如果该值来自后端存量数据或被篡改为 `javascript:`/`data:` 等危险 scheme，点击预览即可触发 DOM XSS。
    修复建议与测试场景
    修复建议

    在预览前使用 `new URL()` 严格校验协议和域名白名单，只允许 `https:`（必要时允许受控的 `blob:`）资源；`parseFileEntry()` 解析出的 URL 也应复用同一校验。校验失败时中止 `window.open` 并提示用户，不要在异常分支直接回退打开原始 URL。

    测试场景

    向组件注入 `value=['evil:javascript:alert(1)']`（`withFileName=true`）或 `value='javascript:alert(1)'`，点击预览后应被拦截并显示错误提示；同时模拟 `getPrivateBlob()` 抛错，确认回退路径也不会打开危险 scheme。

    MAJORL145-149边界条件
    `value` 不再默认初始化为 `[]`，但多文件上传成功时仍直接执行 `[...]value`。当父组件首次传入 `undefined`/`null` 且 `multiple=true` 时，这里会抛出 `TypeError: value is not iterable`，导致上传流程中断。
    修复建议与测试场景
    修复建议

    在构造新值前兜底数组，例如 `const currentValue = Array.isArray(value) ? value : []; const newValue = multiple ? [...currentValue, entryValue] : entryValue;`，删除分支也应同样用数组兜底。

    测试场景

    多文件模式下，父组件不传 `value` 或传 `undefined`，上传首个文件后应正常回填，不应出现运行时异常。
