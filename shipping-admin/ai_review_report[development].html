    src/models/customerManagement.ts2 Major 1 Minor3 个问题
    MAJORL50安全风险
    `writePersisted` 将整个 `PersistedState` 序列化到 `sessionStorage`，其中包含 `selectedCustomer` 对象。客户实体通常会携带联系人、账号或其他业务敏感字段；把整对象落到浏览器存储会扩大 XSS 场景下的数据暴露面，并且这些数据会在标签页生命周期内持续可读。
    修复建议与测试场景
    修复建议

    不要持久化完整客户对象到 `sessionStorage`。仅保存最小必要的非敏感状态，例如 `selectedCustomerId`、分页参数和筛选条件；页面恢复时再基于 ID 重新拉取详情。如果确需缓存，至少在写入前显式挑选允许落盘的字段。

    测试场景

    进入客户管理页并选中一个客户后，检查 `sessionStorage['customer-management-state']`，确认其中不再包含完整 `selectedCustomer` 明细，仅包含非敏感最小化字段；同时验证页面刷新后仍能按 ID 正常恢复所需状态。

    MAJORL75异常处理
    离开页面时直接调用 `sessionStorage.removeItem(STORAGE_KEY)`，这里没有像 `readPersisted` / `writePersisted` 一样做异常兜底。在浏览器禁用存储、隐私模式或 `sessionStorage` 不可用时，这里可能抛出 `SecurityError`，导致路由切换过程报错，页面状态也无法按预期重置。
    修复建议与测试场景
    修复建议

    将 `removeItem` 也放入 `try/catch` 中，保证存储不可用时仍能继续执行本地状态重置。例如：`try { sessionStorage.removeItem(STORAGE_KEY); } catch { /* ignore disabled storage */ }`。

    测试场景

    在 `sessionStorage` 被 mock 为抛错的场景下，从 `/product/customer-management` 跳转到非相关页面，验证不会抛异常，且 `filters/page/pageSize/selectedCustomer` 仍会被重置。

    MINORL28边界条件
    `pathname.startsWith('/product/customer-management/pickup-config')` 会把同前缀但并非该模块的路由也判定为相关页面，例如 `/product/customer-management/pickup-config-history`。一旦后续新增这类路由，当前页面缓存会被错误保留或复用，造成筛选条件和选中客户串页。
    修复建议与测试场景
    修复建议

    按路径段做精确匹配，只放行目标页面及其子路由，例如：`pathname === '/product/customer-management/pickup-config' || pathname.startsWith('/product/customer-management/pickup-config/')`。
