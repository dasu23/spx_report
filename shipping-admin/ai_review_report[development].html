    src/components/AzureFile/DownloadTemplate.tsx2 Major2 个问题
    MAJORL23-28异常处理
    这里把下载改成先异步获取 URL 再执行 `window.open`。由于 `window.open` 不再发生在用户点击的同步调用栈内，常见浏览器会把新窗口判定为弹窗并直接拦截，导致“点击下载无反应”的功能回归。
    修复建议与测试场景
    修复建议

    在点击时先同步打开一个占位窗口，再在异步请求完成后设置其 `location`；或改回 `<a href>` / `link.click()` 这类仍保留用户手势上下文的实现。例如：`const win = window.open('', '_blank'); const url = await getDownloadUrl(...); if (win && url) win.location.href = url; else win?.close();`。

    在 Chrome/Safari 默认拦截弹窗配置下，点击下载按钮，验证模板页签能够稳定打开而不是被浏览器拦截。

    MAJORL27前端安全
    新增的 `window.open(url, '_blank')` 未显式切断 `window.opener`。如果下载链接最终落到非同源页面，目标页可通过 `window.opener` 反向操控当前管理后台页面，形成 tabnabbing 风险。
    修复建议与测试场景
    修复建议

    改为使用 `window.open(url, '_blank', 'noopener,noreferrer')`，或先拿到返回窗口对象后将 `newWindow.opener = null`；同时优先确保下载链接限定为可信域名。

    测试场景

    构造一个会返回到外部页面的下载 URL，验证新页面无法访问 `window.opener`，且不会把当前后台页面重定向到钓鱼地址。

    MAJORL20-24业务逻辑
    模板下载改为先异步请求下载地址再调用 `window.open`。多数浏览器只把“同步发生在用户点击栈内”的 `window.open` 视为可信用户手势；在 `await getDownloadUrl(...)` 之后再打开新窗口，容易被拦截为弹窗，导致用户点击“Download Template”后没有任何下载结果。

    新增的模板下载逻辑直接对接口返回的 `url` 执行 `window.open(url, '_blank')`，未校验协议/目标域名，也未隔离 `opener`。一旦后端返回被污染的外链或 `javascript:` URL，前端会在新窗口执行不受信任内容，且新页面可通过 `window.opener` 反向操控当前管理后台页面。
    修复建议与测试场景
    修复建议

    测试场景

    Mock `getDownloadUrl` 分别返回 `javascript:alert(1)`、`https://attacker.example/poc` 和合法 Blob 下载地址；验证前两者不会被打开，合法地址打开后新窗口无法访问 `window.opener`。
