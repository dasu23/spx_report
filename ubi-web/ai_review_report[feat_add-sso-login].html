    backend/server/src/main/java/com/ubi/server/controller/SsoAuthController.java2 Major 1 Minor3 个问题
    MAJORL29-33异常处理
    `/auth/sso/callback` 直接将 `ssoLoginService.handleCallback(authToken)` 的返回值传给 `response.sendRedirect(...)`，这里没有对返回值做非空/非空白校验。一旦服务层在边界场景下返回 `null` 或空字符串，控制器会在运行时抛出异常或产生无效跳转，导致 SSO 登录主链路失败。
    修复建议与测试场景
    修复建议

    在调用 `sendRedirect` 前增加显式校验，并对非法返回值转换为可预期的业务异常，例如：`String redirectUrl = ssoLoginService.handleCallback(authToken); if (!StringUtils.hasText(redirectUrl)) { throw new BusinessException(502, "SSO 回调结果无效"); } response.sendRedirect(redirectUrl);`。同时建议在 `SsoLoginService` 层约束该方法永不返回空值。

    测试场景

    补充控制器测试：mock `ssoLoginService.handleCallback(...)` 分别返回 `null`、`""`、`" "`，断言接口返回明确错误而不是容器 500/NPE。

    MAJORL29-40权限与访问控制
    **原因：** 该 SSO 回调入口被 `@Public` 直接放行为匿名可访问，但当前控制器层未见任何对 `authToken` 来源、一次性使用约束、回调发起上下文（如 state/nonce）或调用频率的校验逻辑；控制器收到任意请求后即调用 `ssoLoginService.handleCallback(authToken)` 并执行重定向。 ```java @Public @GetMapping("/callback") public void callback(@RequestParam("authToken") String authToken, HttpServletResponse response) throws IOException { requireEnabled(); String redirectUrl = ssoLoginService.handleCallback(authToken); response.sendRedirect(redirectUrl); } ``` **影响：** 若服务层未严格校验 token 绑定关系或缺少 state/nonce 防重放，攻击者可直接构造对该公开入口的请求，触发未授权登录流程、登录 CSRF、票据重放，或借由服务端重定向放大为认证绕过链路。由于该入口是认证边界，控制器层完全公开会扩大攻击面。
    修复建议与测试场景
    修复建议

    为回调入口补充与登录发起阶段绑定的 state/nonce 校验，并确保 `authToken` 仅可一次性消费、具备短时效；同时限制仅接受来自预期 SSO 流程的回调。示例： ```java @GetMapping("/callback") public void callback(@RequestParam("authToken") String authToken, @RequestParam("state") String state, HttpServletResponse response) throws IOException { requireEnabled(); ssoLoginService.validateState(state); // 校验会话绑定/一次性/过期 String redirectUrl = ssoLoginService.handleCallback(authToken, state); response.sendRedirect(redirectUrl); } ``` 并在服务层对票据做幂等消费与重放拦截。

    测试场景

    1. 未经过正常登录发起流程，直接请求 `/auth/sso/callback?authToken=attackerToken`，应被拒绝；2. 使用同一 `authToken`/`state` 重放两次，第二次应失败；3. 使用其他用户会话生成的 `state` 发起回调，应失败。

    MINORL61-65配置变更
    `requireEnabled()` 仅检查 `ssoProperties.isEnabled()`，但 `/callback`、`/exchange`、`/redirect-url` 实际运行还依赖其他 SSO 配置。当前 `/config` 已将 `loginUrl` 为空视为未启用，而这里仍可能在 `enabled=true` 但关键配置缺失时继续放行请求，造成接口对外表现与配置探测结果不一致。
    修复建议与测试场景
    修复建议

    统一启用判定逻辑，抽取一个共享方法，例如 `isSsoAvailable()`，同时用于 `/config` 和 `requireEnabled()`；若至少要求 `enabled && hasText(loginUrl)`，则两处都应复用同一条件。

    测试场景

    增加配置边界测试：当 `enabled=true` 且 `loginUrl` 为空时，断言 `/config` 与 `/redirect-url`、`/callback` 的行为一致（都视为不可用）。

    MAJORL65-67安全风险
    命中了 [LOG-001] 敏感日志检测：异常分支将 `authToken`（即使做了部分 mask 仍属于认证凭证片段）写入 warn 日志。认证 token 片段进入日志后，若日志系统被更广泛访问、聚合或外发，可能造成凭证泄露面扩大，且为后续撞库/关联分析提供线索。
    修复建议与测试场景
    修复建议

    不要在任何级别日志中记录认证凭证或其可关联片段；改为记录请求追踪 ID、远端响应码等非敏感上下文。例如：`log.warn("SSO userinfo request failed: {}", e.getMessage());`，如需排障可引入 requestId/correlationId。

    测试场景

    构造 SSO userinfo 请求异常，断言日志内容不包含 `authToken` 原文、前缀、后缀或任何可逆/可关联片段。

    MINORL102-105API契约
    `buildUrl` 使用 `encodePathSegment` 对 `authToken` 编码后再替换 `{authToken}`。如果 `userinfo` 地址模板实际把 token 放在 query 参数中而不是 path segment，中间的 `+`、`/`、`=` 等字符编码规则会不匹配，可能导致上游无法识别 token。这里把 URL 模板语义固定成 path segment，和配置契约耦合过紧。
    修复建议与测试场景
    修复建议

    若该占位符可能出现在 query 中，应改为基于 URI builder 按路径/查询参数分别编码；至少在配置层明确约束 `userInfoUrl` 中 `{authToken}` 只能位于 path segment。若要兼容两者，可拆分为 baseUrl + query/path 参数配置，避免字符串替换。

    测试场景

    增加包含 `+`、`/`、`=` 的 token 用例，分别验证 `{authToken}` 位于 path 和 query 两种 URL 模板时生成的请求地址符合上游预期。

    backend/server/src/main/java/com/ubi/server/service/SsoLoginService.java2 Major2 个问题
    MAJORL38-39安全风险
    命中了 [LOG-001] 敏感日志检测：登录失败日志记录了 `authToken` 的掩码值。这里的 `authToken` 是外部 SSO 回调携带的认证凭证，写入日志会增加凭证暴露风险；部分掩码并不能消除其敏感属性，尤其在短 token、重复请求或与其他日志字段关联时仍可被利用。
    修复建议与测试场景
    修复建议

    删除日志中的 `authToken` 字段，仅保留失败原因和无敏感性的关联标识；若必须定位问题，使用服务端生成的 traceId 替代。

    测试场景

    模拟 `fetchUserInfo` 返回 empty，检查日志输出中不应出现 `authToken` 或其任何截断形式。

    MAJORL47-50边界条件
    本地用户查询直接使用 `email.trim()` 作为 `username` 精确匹配，但未做大小写归一化。SSO 返回的邮箱大小写与本地账号存储大小写不一致时，会被误判为账号不存在，导致合法用户无法通过 SSO 登录。邮箱本身通常应按大小写不敏感处理，这会造成明显的登录失败。
    修复建议与测试场景
    修复建议

    在查询前统一做大小写归一化，并与本地账号存储规则保持一致，例如：`String username = email.trim().toLowerCase(Locale.ROOT);`；若数据库中历史数据大小写不统一，则改为大小写不敏感查询（如使用 `lower(username) = ?` 或对应 ORM 能力）。同时补充迁移/校验，确保本地用户名规范一致。

    测试场景

    新增用例：SSO 返回 `User@Example.com`，本地用户为 `user@example.com` 时应能成功登录；反向大小写组合也应通过。

    CRITICALL52-55前端安全
    新增 `loginWithToken` 将服务端返回的 JWT 直接写入 `localStorage`。`localStorage` 中的令牌可被任意同源脚本读取，一旦页面存在 XSS、第三方脚本污染或浏览器扩展窃取，即可直接接管用户会话。这属于敏感凭证持久化暴露。

    在浏览器控制台或注入脚本环境中执行 `localStorage.getItem('ubi_token')`，确认当前实现可直接读取 JWT；改造后应无法通过 JS 读取会话凭证。

    frontend/apps/dashboard/src/pages/login/login-page.tsx1 Major1 个问题
    MAJORL29-46状态管理
    SSO 票据兑换的 useEffect 只在组件首次挂载时执行一次，并且通过 `eslint-disable` 忽略了 `searchParams`、`navigate`、`loginWithToken`、`setSearchParams` 依赖。这会导致同一个 LoginPage 实例下如果路由仅更新 query（例如用户从 `/login` 再次被重定向到 `/login?token=...`，或从一个 `ssoError` 切换到另一个），副作用不会重新执行，新的 token/ssoError 将被忽略，页面停留在旧状态。
    修复建议与测试场景
    修复建议

    让该 effect 显式依赖 query 参数变化，而不是只在 mount 时运行。例如先提取 `const token = searchParams.get('token')`、`const ssoError = searchParams.get('ssoError')`，再将它们加入依赖数组：`useEffect(() => { ... }, [token, ssoError, loginWithToken, navigate, setSearchParams])`。如果担心重复兑换，可在成功/失败后仅移除已消费的参数，或用 `useRef` 记录最近处理过的 token。

    测试场景

    在同一个 MemoryRouter/BrowserRouter 实例中，先渲染 `/login`，随后仅通过路由更新到 `/login?token=t1`，断言会触发 `ssoExchange` 并跳转；再更新到 `/login?ssoError=account_not_found`，断言错误文案会刷新。
