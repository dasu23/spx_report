    backend/server/src/main/java/com/ubi/server/service/AnalyticsService.java1 Critical 1 Major 1 Minor3 个问题
    CRITICALL143-145安全风险
    新增的 `messageWhere` 通过字符串拼接构造 SLS SQL 片段：`" WHERE message = '" + SlsFieldUtils.escapeSqlString(message) + "'"`。即使做了单引号转义，这里仍然是将外部输入直接拼入查询语句，属于 [SEC-003] 明确禁止的 SQL 拼接模式；一旦 `escapeSqlString` 对目标方言转义不完整，或后续复用到其他字段/方言，容易演变为查询注入或语义绕过。
    修复建议与测试场景
    修复建议

    避免拼接 SQL 字面量，优先改为 SLS 官方支持的参数化/结构化查询方式；若底层 SDK 不支持参数化，应将 `message` 作为检索过滤条件传入专门的安全构造器，并对允许字符做白名单限制（如长度、可见字符集），禁止任意原始文本直接进入 SQL 片段。

    测试场景

    补充针对 `message` 的恶意输入测试，例如包含单引号、反斜杠、换行、SQL 关键字、注释符（如 `a' OR 1=1 --`）时，断言最终查询不会扩大结果集、不会报语法错误、不会改变原始过滤语义。

    MAJORL251-253业务逻辑

    让分母与当前筛选口径保持一致。若产品期望“筛选后错误率”，则应至少对 `userId` 应用同样过滤；若 `type` 也参与错误率口径，则需同步到分母定义并在接口文档中明确。示例：`String totalFilter = "app_key: " + appKey + (hasUserId ? " AND user_id: " + userId : "");` 然后使用 `totalFilter + " | SELECT count(*) as totalEvents"`。如果确实要展示“全站基线占比”，建议新增独立字段名避免与筛选结果混淆。
