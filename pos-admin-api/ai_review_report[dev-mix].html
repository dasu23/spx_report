critical63安全风险这里把 `req.getSession().getId()` 直接当作 `authToken` 拼进 302 Location。Session ID 属于服务端会话凭据，放入 URL 后会暴露在浏览器历史、代理日志、Referer 和前端埋点中，攻击者拿到该值后可直接复用会话，造成登录态劫持。不要把 Session ID 透传到 URL。改为：1）登录成功后仅通过 `Set-Cookie` 下发会话；或 2）生成一次性短时授权码，再由前端调用后端接口换取登录态。例如将 `location` 改为不携带会话凭据的固定前端地址，登录态由 Cookie 维持。完成一次 SSO 登录，验证 302 跳转地址、浏览器地址栏、访问日志和埋点日志中都不包含 `JSESSIONID`/sessionId。
major49,73安全风险新增日志直接打印 `ssoLoginToken`。该 token 来自第三方 SSO 回调，属于敏感认证凭据；一旦进入应用日志、链路日志或错误日志，容易被运维平台、日志聚合系统或调试人员获取，形成凭据泄露风险。不要记录原始 token，改为只打印脱敏后的前后缀或请求唯一标识。例如：`log.debug("[SSOLogin] SSO login callback received, tokenHash={}", DigestUtils.sha256Hex(ssoLoginToken));`；异常日志同样去掉 token 原文。构造一次成功和一次失败的 SSO 回调，检查应用 debug/error 日志中不出现完整 `ssoLoginToken`。
