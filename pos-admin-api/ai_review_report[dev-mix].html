    MAJORL52-70异常处理
    `counterService.applyEvent` 失败后仅记录日志且不重试/不入 DLQ，消息会被直接确认并丢失，导致计数长期不一致（依赖人工补数）。
    修复建议与测试场景
    修复建议

    为消费失败建立可恢复机制：至少接入重试或 DLQ（如 `DefaultErrorHandler + DeadLetterPublishingRecoverer`），并对不可恢复异常落地补偿队列，避免静默丢数。

    测试场景

    模拟 `counterService.applyEvent` 抛异常，验证消息会进入重试或 DLQ；恢复后可重放并使计数最终一致。

    `JSON.parseObject` 在输入为 `"null"` 等场景可能返回 `null` 而不抛异常，随后 `msg.getType()` 触发 NPE。该异常虽然进入 `catch`，但当前 `catch` 内又调用 `msg.getEventId()`，会再次 NPE，导致监听方法异常退出。
    修复建议与测试场景
    修复建议

    在第二个 `try` 前增加 `msg == null` 判空并直接返回；同时日志打印 `eventId` 前做空值保护，例如：`String eventId = (msg == null ? null : msg.getEventId());`。
