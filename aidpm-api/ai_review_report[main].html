    backend/src/main/java/com/aidpm/common/util/EncryptionUtil.java2 Major 1 Info3 个问题
    MAJORL31安全风险
    `Cipher.getInstance(ALGORITHM)` 在 `ALGORITHM = "AES"` 的情况下会落到提供方默认模式，通常是 `AES/ECB/PKCS5Padding`。ECB 不使用随机 IV，导致相同明文块产生相同密文块，容易泄露敏感字段模式；同时该实现也未提供完整性保护，密文被篡改后无法可靠检测。
    修复建议与测试场景
    修复建议

    显式改为带随机 IV 的认证加密模式，例如 `AES/GCM/NoPadding`：加密时为每次操作生成新的 12 字节随机 IV，将 IV 与密文一起编码存储；解密时先解析 IV 再初始化 `GCMParameterSpec`。不要依赖 JCA 默认算法模式。

    测试场景

    新增加解密安全测试：1）同一明文连续加密两次，结果必须不同；2）篡改密文任意字节后，解密应失败并抛出认证异常；3）验证历史密文迁移或兼容策略。

    MAJORL47-50异常处理
    `Base64.getDecoder().decode(encryptedText)` 在收到非法 Base64 字符串时会抛出 `IllegalArgumentException`，当前 `catch` 只处理 `GeneralSecurityException`。这会让解密失败走到未预期的运行时异常路径，接口侧通常表现为 500，而不是统一的“解密失败”错误。
    修复建议与测试场景
    修复建议

    把 `IllegalArgumentException` 一并纳入解密失败分支，例如：`catch (GeneralSecurityException | IllegalArgumentException e) { throw new RuntimeException("Decryption failed", e); }`。如果上层有统一业务异常体系，建议改为抛出明确的业务异常。

    测试场景

    补充非法密文测试：传入 `"not-base64"`、截断后的 Base64 字符串，断言不会泄漏原始 `IllegalArgumentException`，而是返回统一的解密失败结果。

    INFOL14,31,45向后兼容
    `Cipher.getInstance("AES")` 依赖 JCE provider 的默认转换实现，实际模式/填充未在代码中固定。不同运行环境或后续 provider 调整后，可能出现新老实例加解密结果不兼容，导致历史数据无法解密。
    修复建议与测试场景
    修复建议

    显式声明完整转换并保持全链路一致，例如 `private static final String TRANSFORMATION = "AES/ECB/PKCS5Padding";`，然后统一使用 `Cipher.getInstance(TRANSFORMATION)`。如果后续要切换实现，建议通过版本化或迁移策略处理已有密文。

    测试场景

    frontend/proxy.ts1 Major1 个问题
    MAJORL7-18业务逻辑
    登录态判断仅依赖 `auth_logged_in` Cookie 是否存在，而未校验真实会话/Token 的有效性或过期状态。新增逻辑会把任意携带该 Cookie 的请求视为“已登录”，以及把缺少该 Cookie 但可能仍有其他有效认证状态的请求视为“未登录”，属于认证失效处理不完整。典型后果是过期/伪造的前端标记导致错误放行或错误重定向，形成失效会话残留。
    修复建议与测试场景
    修复建议

    不要用布尔型前端 Cookie 作为登录态真值来源。应改为校验服务端可验证的会话凭证，例如签名/加密且带过期时间的 Session Cookie，或在 middleware/proxy 中解析并校验认证 Token；至少在判断登录态时同时校验过期时间，并在失效时清理相关 Cookie。

    测试场景

    构造仅含 `auth_logged_in=true`、但无有效会话凭证或凭证已过期的请求访问受保护页面，预期应被重定向到 `/login` 且清理失效 Cookie；再构造会话刚过期的场景，确认不会继续被识别为已登录。
