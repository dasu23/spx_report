    pus-domain/src/main/java/io/speedx/pus/domain/service/impl/PickupRouteServiceImpl.java1 Major1 个问题
    MAJORL78-83数据一致性
    `updateRouteStatus` 和 `recordPickupLog` 被移到 `retryTemplate` 外部后，重试只覆盖 `dmsAdapter.syncPickupRoute(req)`。如果 DMS 已成功接收，但后续状态更新或日志记录抛异常，方法会直接失败并保留路由为 `ROUTED`，后续补偿任务会再次推送同一路线，造成重复下发和状态不一致。
    修复建议与测试场景
    修复建议

    将 `updateRouteStatus(req.getBizId())` 与 `recordPickupLog(bookCodes)` 保持在重试闭包内，或至少对“DMS 已成功但本地更新失败”做幂等保护。例如先在闭包内完成状态更新，再记录日志；若必须拆开，需为 DMS 推送增加去重键/幂等校验。

    测试场景

    构造 `dmsAdapter.syncPickupRoute(req)` 成功、`updateRouteStatus(req.getBizId())` 抛异常的场景，验证不会留下 `ROUTED` 状态并在再次执行补偿时重复推送同一 `bizId`。

    pus-facade-impl/pus-facade-impl-job/src/main/java/io/speedx/pus/facade/impl/job/PushRouteToDmsJob.java1 Major 1 Minor2 个问题
    MAJORL63-94性能隐患
    新增的补偿任务按 `routedRoutes` 逐条循环，并在每次迭代中调用 `buildRouteDtoList(route)`；该辅助方法在新增代码里又会执行 `findByRouteId`、`findByBookingCodeIn`、`findByBookingCodeIn` 三次仓储查询。批次内 route 数量较大时会形成明显的 N+1 数据库访问，补偿任务本身容易放大数据库与下游 DMS 压力，导致可用性风险。
    修复建议与测试场景
    修复建议

    将任务改为批量预取所需数据，至少按 `routeIds` 和全部 `bookingCodes` 一次性查询 `PickupRouteItem`、`Pickup`、`PickupParcel`，在内存中按 `routeId/bookingCode` 分组组装 `PickupRouteDto`，避免在 `for` 循环内触发仓储查询。

    测试场景

    构造一个包含 500+ `ROUTED` 路线、每条路线含多条 `PickupRouteItem` 的批次，执行该 Job 并统计 SQL 次数与执行时长；修复后应显著降低查询次数，且不因批次规模扩大出现线性放大的数据库访问。

    MINORL70-78业务逻辑
    当某条路由查不到 `PickupRouteItem` 时，代码仅 `warn` 后继续，最后仍统一 `handleSuccess()`。这会把“部分路由未补推”的情况误报为任务成功，导致仍处于 `ROUTED` 的脏数据被静默遗留。
    修复建议与测试场景
    修复建议

    累计失败/跳过的 `routeId`，只要存在未处理路由就调用 `XxlJobHelper.handleFail(...)` 并输出明细；或者至少在成功结果中返回跳过数量，避免人工误判补偿已完成。

    测试场景

    制造一条 `ROUTED` 路由但无 `PickupRouteItem` 的数据，执行任务后应返回失败或部分成功信息，而不是直接标记成功。
