    MAJORL512,540,646,697,767-782数据一致性
    本次变更将批量删除、确认、加件、路由后的 `createOperationLog(...)` 全部注释掉，并连同私有实现一起停用。这样会导致这些操作不再落库/发出操作日志，形成永久性的业务留痕缺失；如果下游还有依赖操作日志做追踪、对账或异步处理，也会直接出现数据缺口。
    修复建议与测试场景
    修复建议

    不要直接注释掉整条日志链路。若当前热点问题是日志异步线程或消息量过高，建议改为可开关的降级方案，例如通过配置开关关闭非关键场景，或增加采样/批量提交/限流，而保留关键操作日志：`if (operationLogEnabled) { createOperationLog(context, batchCode); }`。同时补充回放或补偿方案，避免这段时间的操作日志永久丢失。

    测试场景

    执行批量删除、确认、加件、路由四类操作后，校验操作日志表/消息队列中仍能看到对应记录；关闭开关时应有明确降级行为且不影响后续补偿。

    MAJORL512-696安全风险
    将 `createOperationLog(...)` 调用直接注释掉，导致批次删除等关键操作不再产生审计日志。该类日志通常用于安全追踪、事后取证和异常操作定位；热修后若发生误操作或恶意批量处理，将缺少可追溯证据。
    修复建议与测试场景
    修复建议

    保留操作日志链路；如果当前实现导致 CPU 过高，应改为限流、批量异步、降采样或修复具体热点，而不是整体移除审计。至少为删除、确认、路由等状态变更保留最小必要日志。

    测试场景

    执行批次删除/确认/分配路线后，校验仍能生成对应 operation log，并验证高并发下日志链路不会导致 CPU 异常。

    pus-facade-impl/pus-facade-impl-job/src/main/java/io/speedx/pus/facade/impl/job/PushRouteToDmsJob.java1 Major 1 Minor2 个问题
    MAJORL66-73性能隐患
    这里按 `routedRoutes` 逐个循环调用 `buildRouteDtoList(route)`，而后者会再次查询 `PickupRouteItem`、`Pickup`、`PickupParcel`。当一个 batch 下 route 数量较大时，会形成明显的 N+1 查询模式，容易在补偿任务执行时放大数据库压力，触发可用性风险。
    修复建议与测试场景
    修复建议

    改为按 batch 一次性预取所需 routeItem、pickup、pickupParcel 数据，再在内存中按 routeId/bookingCode 分组组装 DTO；至少应避免在循环体内发起多次 repository 查询。

    测试场景

    构造包含大量 routed route 的 batch（如数百条），监控该 job 的 SQL 次数、执行时长和数据库负载，确认优化后查询次数与 route 数量不再线性增长。

    MINORL68-76业务逻辑
    补偿任务在发现某条路线没有 route item 时仅 `warn` 后 `continue`，但循环结束后仍统一 `handleSuccess()`。这会把“部分路线未补推”的结果标记成成功，运维侧难以及时发现仍处于 `ROUTED` 的遗留数据。
    修复建议与测试场景
    修复建议

    累计被跳过/失败的 routeId，并在存在未处理路线时返回失败或至少返回带明细的失败信息，例如：`List<Long> skipped = new ArrayList<>(); ... skipped.add(route.getId()); ... if (!skipped.isEmpty()) { XxlJobHelper.handleFail("skipped routes=" + skipped); return; }`。

    测试场景

    构造一个 batchCode，其中一条 ROUTED 路线没有 route item，另一条正常；执行任务后应返回失败或部分失败信息，而不是直接 success。
